# Comments to Contracts to HELP v0

Status: proof_linked
Audience: advanced student, developer, maintainer
Registry ID: lesson.student.selfdoc_comments_to_help

## Purpose

Teach that a software system can explain and check part of itself when source
comments, usage contracts, HELP output, and validation reports are kept in one
evidence chain.

## Campus Path

```text
source comment
-> @dottalk.usage contract
-> comments evidence
-> CMDHELP
-> CMDHELPCHK
-> proof.lab.selfdoc.comments_to_contracts.first_run
-> lesson.student.selfdoc_comments_to_help
```

## Student Path

1. Read the lab document for Comments to Contracts to HELP.
2. Identify the command being documented.
3. Compare source intent, HELP output, and validation output.
4. Classify the claim as source-defined, help-documented, validated, or review-needed.

## Expected Observations

- Comments are useful only when they can be collected and checked.
- HELP text should stay tied to command behavior and source contracts.
- A validation report is not decoration; it is part of the lesson evidence.
