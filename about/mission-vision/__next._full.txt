12:T1b19,<h2 id="mission"><a class="no-underline" href="#mission">Mission</a></h2>
<p>Build a configurable Laboratory Campus where people learn data systems by
using, inspecting, and helping to build the same engine, language,
documentation, and proof tools that operate the campus.</p>
<p>x64base exists to make database literacy practical and visible. It should run
well enough to support real work while remaining transparent enough for a
learner to trace a table, command, schema, index, document, decision, and proof
back to the system that produced it.</p>
<h2 id="vision"><a class="no-underline" href="#vision">Vision</a></h2>
<p>A self-describing, increasingly self-hosting learning ecosystem in which:</p>
<ul>
<li><strong>x64base is the stateful data substrate</strong> for tables, records, fields,
memos, indexes, relations, work areas, validation, and storage lifecycles.</li>
<li><strong>DotTalk++ is the executable campus language</strong> for commands, scripts,
observation, repeatable labs, and proof readback.</li>
<li><strong>HELP, metadata, contracts, SelfDoc, and MDO are institutional memory</strong>:
they explain, organize, validate, preserve, and publish what the system does.</li>
<li><strong>The Laboratory Campus is the configurable learning environment</strong> that
turns live project work into labs, lessons, cases, datasets, tools, and
learning paths.</li>
<li><strong>Students, developers, educators, and AI collaborators share governed
evidence</strong> instead of receiving separate simplified stories.</li>
</ul>
<p>The long-term goal is not merely a database product with training attached. It
is a place where the database, its development process, and its curriculum are
made from the same inspectable materials.</p>
<h2 id="the-campus-thesis"><a class="no-underline" href="#the-campus-thesis">The Campus Thesis</a></h2>
<pre><code class="language-text">Everything is data.
Everything can become a lesson.
Everything important should be inspectable.
</code></pre>
<p>That is a direction with proof gates, not a claim that every part is already
self-hosted. The current system still uses source files, YAML registries,
runtime transcripts, generated reports, and companion tools. Those assets move
toward x64base-backed curation through explicit import, validation, export, and
recovery proof.</p>
<h2 id="glass-but-real"><a class="no-underline" href="#glass-but-real">Glass but Real</a></h2>
<p>"Glass" means observable, explainable, and traceable. It does not mean fragile
or toy-like.</p>
<p>The engine and campus should:</p>
<ul>
<li>run reasonably well on supported paths;</li>
<li>expose state, cost, and failure honestly;</li>
<li>distinguish implemented, runtime-proven, source-defined, simulated,
historical, planned, and experimental material;</li>
<li>preserve reproducible evidence;</li>
<li>teach why mutation boundaries, reliability, backup, recovery, and validation
matter.</li>
</ul>
<h2 id="co-development-is-curriculum"><a class="no-underline" href="#co-development-is-curriculum">Co-development Is Curriculum</a></h2>
<p>The system develops recursively:</p>
<pre><code class="language-text">x64base behavior
-> DotTalk++ command or script
-> source, HELP, metadata, and contract evidence
-> SelfDoc preservation and MDO organization
-> manual, diagram, lab, lesson, and website derivative
-> review exposes drift, weak proof, or hidden behavior
-> findings return to the engine, language, metadata, and tests
</code></pre>
<p>A new feature becomes a state lesson. A bug becomes a lesson in assumptions and
proof. A documentation correction becomes a lesson in authority. A generator
becomes a lesson in how data becomes an explanation. Students learn not only
the finished answer, but how a real system is examined and strengthened.</p>
<h2 id="tools-are-part-of-the-campus"><a class="no-underline" href="#tools-are-part-of-the-campus">Tools Are Part of the Campus</a></h2>
<p>The campus progressively exposes the same tools used to build it: DotTalk++ and
DotScript, source review and AI-assisted exploration, SelfDoc, MDO/manualgen,
validators, diagram generators, portal registries, build systems, and
publication checks.</p>
<p>Beginners receive a usable path first. They can then open the glass and follow
the deeper tooling, evidence, and decisions behind it.</p>
<h2 id="the-ai-portal"><a class="no-underline" href="#the-ai-portal">The AI Portal</a></h2>
<p>The AI Portal is intended to give an AI collaborator a rapid, task-specific
introduction to the ecosystem. It follows curated, typed jumps between projects
and artifacts, checks authority and proof, and assembles a bounded context
packet for a series of related tasks.</p>
<p>The AI Portal lane is <strong>Alpha/Experimental</strong>. It is not production autonomous
memory, an independent source of truth, or permission to bypass human approval.
Generated context may be incomplete or stale; it must explain its evidence and
default to read-only preparation.</p>
<p><a href="/docs/labtalk/ai-portal">Read the AI Portal Alpha/Experimental lane</a>.</p>
<h2 id="one-campus-three-journeys"><a class="no-underline" href="#one-campus-three-journeys">One Campus, Three Journeys</a></h2>
<pre><code class="language-text">learner: concept -> example -> data -> command -> observation -> proof -> lesson
developer: request -> source -> change -> test -> proof -> documentation -> publication
AI collaborator: task -> curated context -> constraints -> action -> verification -> closeout
</code></pre>
<p>These are different entrances into the same evidence system.</p>
<h2 id="public-identity"><a class="no-underline" href="#public-identity">Public Identity</a></h2>
<blockquote>
<p>x64base is building a glass-but-real Laboratory Campus where the database is
infrastructure and subject, DotTalk++ is the teaching language, documentation
is executable institutional memory, and development itself becomes
proof-backed curriculum.</p>
</blockquote>
<p>The project is more than a product, but it is not less than one. The engine and
tools must remain useful, testable, and honestly described while the campus
grows around them.</p>

<h2 id="collaboration-direction"><a class="no-underline" href="#collaboration-direction">Collaboration Direction</a></h2>
<p>The campus welcomes review from computer science and database educators,
technical-writing instructors, general-education specialists, learning-science
researchers, practitioners, students, and maintainers.</p>
