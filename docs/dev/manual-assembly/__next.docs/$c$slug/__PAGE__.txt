6:T1865,<p>x64base makes a self-documentation claim: it should use its own metadata and
documentation infrastructure to describe, and increasingly prove, itself. This
page shows that claim working — how the developer manual is <em>assembled from the
system</em>, not typed by hand. It is a <strong>reviewed derivative</strong> of the assembly
manifest in the implementation tree, per the
<a href="/docs/dev/selfdoc-website-publication">simplex direction gates</a>.</p>
<p>The finished manual is on the shelf next to this explanation:
<a href="/downloads/current/developer-manual-latest.html"><strong>read it online</strong></a> →.</p>
<h2 id="from-forcing-pieces-together-to-assembling-a-declared-whole"><a class="no-underline" href="#from-forcing-pieces-together-to-assembling-a-declared-whole">From forcing pieces together to assembling a declared whole</a></h2>
<p>The first manual was made the way early software always is: build each piece,
then force the pieces into one artifact by hand — on the order of twenty
individually-invoked, human-gated steps, with no single assembler, and with no
table of contents, index, or glossary at all. It produced a good manual, but the
whole lived in an operator's memory, and staleness was found by hand.</p>
<p>The shift was not a bigger script. It was a change of stance: <strong>stop assembling
the manual by hand and start declaring what the manual is</strong>, then let a machine
assemble it and a gate keep it honest.</p>
<h2 id="the-bill-of-materials"><a class="no-underline" href="#the-bill-of-materials">The bill of materials</a></h2>
<p>Every part of the manual is declared, in reading order, in a single manifest
(<code>manual_assembly_manifest.yaml</code>). Each part records its class, its
source-of-record, a stable anchor, the region mode that bounds what the assembler
may touch, and the generator that produces it. The <code>class</code> and <code>direction</code>
vocabularies are shared <strong>verbatim</strong> with the
<a href="/docs/dev/website-documentation-matrix">website content manifest</a>, so the manual
and this site sit on one simplex/duplex spine.</p>
<p>The manual is three things, not one uniform blob:</p>

























<table><thead><tr><th>Layer</th><th>What it is</th><th>How it is produced</th></tr></thead><tbody><tr><td><strong>Spine</strong></td><td>Command reference, function reference, SET family, error/message catalog</td><td>Harvested from source, HELP, and metadata — mechanical</td></tr><tr><td><strong>Authored branch</strong></td><td>The narrative chapters (data model, indexing, expressions, dispatch, evidence)</td><td>Hand-authored / reviewed; teaches beyond what can be harvested</td></tr><tr><td><strong>Front &#x26; back matter</strong></td><td>Title, provenance, table of contents, glossary, index, colophon</td><td>Generated from the assembled whole — none of this existed before</td></tr></tbody></table>
<h2 id="the-assembler"><a class="no-underline" href="#the-assembler">The assembler</a></h2>
<p>One runner reads the manifest and emits the manual in a single pass — a loop over
parts, dispatching on each part's region mode:</p>
<ul>
<li><strong>whole-file</strong> — the assembler owns the region and regenerates it every build
(hand-edits forbidden): the title, TOC, function reference, error catalog, index.</li>
<li><strong>candidate</strong> — it regenerates a <em>review candidate</em>; the published text changes
only on human acceptance. This is how the authored/derived chapters stay gated.</li>
<li><strong>authored</strong> — it never rewrites the content; it only brackets it for tooling.</li>
<li><strong>append</strong> — it appends provenance/evidence snapshots and never overwrites them.</li>
<li><strong>bind</strong> — it binds a whole set into the reader: the 183 command pages, the
diagram set.</li>
</ul>
<p>The result of the first build: <strong>22 parts, ~13,800 lines</strong>, with 63 functions
harvested from source, 183 command pages bound, 12 diagrams bound from the
<a href="/docs/dev/important-documents#diagram-attachment-governance">diagram attachment matrix</a>,
and a generated table of contents, glossary, and index.</p>
<h2 id="the-drift-gate"><a class="no-underline" href="#the-drift-gate">The drift gate</a></h2>
<p>A companion check re-assembles the manual from current source and compares it,
region by region, to the committed build. The severity follows the part's class:</p>
<ul>
<li>a <strong>generated</strong> or <strong>bound</strong> region that no longer matches source → <strong>the build
fails</strong> (the same discipline the command-catalog and diagram checks already
apply);</li>
<li>a <strong>derived</strong>, <strong>authored</strong>, or <strong>reported</strong> region that changed → a
regenerate-or-review flag, non-blocking.</li>
</ul>
<p>So the manual cannot silently rot: if the engine changes and the manual is not
rebuilt, the gate says so.</p>
<h2 id="the-part-that-closes-the-loop"><a class="no-underline" href="#the-part-that-closes-the-loop">The part that closes the loop</a></h2>
<p>The manual's final page is a <strong>colophon</strong> the assembler writes itself — the
assembler version, the source commit, the tool versions, and the attested
machine. The manual records exactly how it assembled itself. That is the
self-documentation claim made literal: not asserted, but demonstrated on the last
page of the thing it describes.</p>
