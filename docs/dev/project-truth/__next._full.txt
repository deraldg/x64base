<h2 id="project-center-of-gravity"><a class="no-underline" href="#project-center-of-gravity">Project center of gravity</a></h2>
<p>The database engine is the center of gravity for x64base. The project should be
understood first through its table runtime, DBF-family storage work, x64 table
extensions, memo objects, index backends, work areas, relation graph, buffering,
locking, validation, and data-mutation rules.</p>
<p>The surrounding system is the teaching and analysis layer. DotTalk++, SelfDoc,
MDO, manuals, diagrams, the Laboratory Campus, and the website are important
because they make the database engine inspectable and teachable. They collect
the evidence, organize it, and explain it; they do not become a substitute for
runtime truth.</p>
<p>SelfDoc and MDO are now part of the system architecture. They started as
cooperating project-documentation conversations and are being shaped into a
repeatable publication loop:</p>
<pre><code class="language-text">source / runtime behavior
-> HELP / command metadata
-> comments / contracts / proof scripts
-> SelfDoc reports
-> MDO organization
-> manuals / diagrams / curriculum / website
</code></pre>
<p>This is the working partnership model for the project: implementation and
documentation develop together, with the database engine remaining the primary
truth source and the documentation system keeping the work visible.</p>
<h2 id="ownership-boundary"><a class="no-underline" href="#ownership-boundary">Ownership boundary</a></h2>
<p>The current repository boundary is deliberate:</p>
<ul>
<li><code>x64base</code> owns engine/runtime truth and shared front-end workbench code.</li>
<li>Arctic GUI/TUI work remains under the runtime/engine tree, not under a
separate campus repository.</li>
<li><code>LabTalk</code> is a consumer layer for portal, registries, labs, proofs,
assignments, and education-facing overlays.</li>
<li><code>DotTalk++</code> remains the runtime/manual identity, but source ownership is
still primarily centered on <code>x64base</code> until a cleaner split is justified.</li>
</ul>
<p>For publication naming:</p>
<ul>
<li><code>x64base</code> is the engine/platform name.</li>
<li><code>DotTalk++</code> is the runtime shell/manual/help identity.</li>
<li>the DotTalk++ artifact/manual-room site is the support surface for manuals,
references, generated documentation, proof packets, and downloads; it is
local preview only until a public repository/domain is approved</li>
<li><code>LabTalk</code> is the campus/education consumer layer.</li>
<li>historical/internal buckets such as <code>dottalk</code> may remain in slugs or folders,
but should not silently replace the published <code>DotTalk++</code> identity in prose or navigation.</li>
</ul>
<p>Use the lowest authority that can prove the claim:</p>
<ul>
<li>runtime/engine behavior -> <code>x64base</code></li>
<li>workbench surfaces over runtime truth -> <code>x64base</code></li>
<li>campus/portal/proof orchestration -> <code>labtalk</code></li>
</ul>
<h2 id="current-status"><a class="no-underline" href="#current-status">Current status</a></h2>
<p>DotTalk++ / x64base should be described as a working educational and research runtime in active beta, not as a finished commercial database product.</p>
<p>Current proven or active areas include:</p>
<ul>
<li>DBF-style table runtime, cursor state, records, fields, work areas, and table inspection.</li>
<li>x64-family table and metadata work.</li>
<li>INX, CNX, CDX, and LMDB-backed index work.</li>
<li>Memo subsystem with object-oriented architecture and canary coverage.</li>
<li>DotTalk++ command shell and DotScript automation.</li>
<li>HELP, command reflection, <code>CMDHELPCHK</code>, and metadata congruence work.</li>
<li>Relation-aware browsing through <code>ERSATZ</code> / <code>RBROWSE</code>.</li>
<li>Table listing plus record and tuple browsing through <code>SMARTLIST</code>, <code>SMARTBROWSE</code>, <code>SM</code>, <code>SMART</code>, <code>SIMPLEBROWSE</code>, <code>SB</code>, <code>BROWSE</code>, <code>BROWSER</code>, and <code>BROWSETUI</code>.</li>
<li>TableTalk is the table-buffering lane: <code>SET TABLE BUFFER</code>, <code>TABLE</code>, dirty/stale state, <code>COMMIT</code>, <code>ROLLBACK</code>, and buffered mutator behavior.</li>
<li>Data dictionary readers, catalog paths, and object resolver code.</li>
<li>Python binding experiments through <code>pydottalk</code>.</li>
<li>Indexing architecture across INX, CNX, CDX, LMDB, Open Index API boundaries, and backend diagnostics.</li>
<li>GUI work across wxWidgets, Python/Tk preview tools, pycrud, and Turbo Vision-style TUI surfaces.</li>
<li>GUI/front-end branding uses <code>DotTalk++ Workbench</code> as the family name.</li>
<li><code>Parallel GUI/TUI</code> is the public product lane for aligned graphical and terminal interfaces.</li>
<li><code>Arctic</code> remains the TUI code name.</li>
<li><code>Foxtalk</code> lineage names remain only where they still explain the subsystem story.</li>
</ul>
<p>See <a href="/docs/engine/feature-crosswalk">Engine Feature Crosswalk</a> for the current evidence-backed hierarchy and <a href="/docs/dev/current-lanes">Current Work Lanes</a> for the non-front-page lane inventory.</p>
<h2 id="build-configuration-truth"><a class="no-underline" href="#build-configuration-truth">Build configuration truth</a></h2>
<p>The top-level CMake options define the real build surface:</p>
<pre><code class="language-text">DOTTALK_WITH_TV        ON by default
DOTTALK_WITH_GUI       OFF by default
DOTTALK_WITH_WX        OFF by default
DOTTALK_TV_TESTS       ON by default
DOTTALK_WITH_INDEX     ON by default
DOTTALK_WITH_EDUCATION ON by default
BUILD_PYDOTTALK        OFF by default
DOTTALK_WITH_RELATIONS ON by default
DOTTALK_PROFILE        DEV or PROD
</code></pre>
<p>The build also wires <code>metacollect</code> as a developer executable for low-level metadata and reflection collection. Its CMake comment explicitly says it must not rebuild HELP data, mutate <code>CMDHELPCHK</code>, write DBFs, or mutate publication artifacts.</p>
<h2 id="command-surface-truth"><a class="no-underline" href="#command-surface-truth">Command surface truth</a></h2>
<p>The command reference shows active command surfaces including:</p>
<ul>
<li><code>BUILDLMDB</code> for materializing LMDB-backed tag databases from current table data.</li>
<li><code>CMDHELPCHK</code> for reflection and HELP/catalog validation.</li>
<li><code>ERSATZ</code> and <code>RBROWSE</code> for relation-aware browsing.</li>
<li><code>SMARTLIST</code>, <code>SIMPLEBROWSE</code> / <code>SB</code>, and <code>SMARTBROWSE</code> / <code>SM</code> / <code>SMART</code> for table, tuple, and record browsing.</li>
<li><code>WORKSPACE</code> commands for opening, saving, loading, and closing workspaces.</li>
<li><code>DDL</code> commands for schema-oriented DBF creation and validation work.</li>
<li><code>DRAWIO</code> helpers for opening diagram tooling, with metadata-generated diagrams still a future direction.</li>
</ul>
<h2 id="website-wording-rule"><a class="no-underline" href="#website-wording-rule">Website wording rule</a></h2>
<p>Use these words carefully:</p>
<ul>
<li>Use "active beta", "runtime-proven", "active", "partial", "experimental", and "canary" when that is the observed state.</li>
<li>Avoid presenting x64base as complete, commercial, or production-finished.</li>
<li>Avoid claiming DBF_64 and FPT64 are the only storage truth. The local tree contains DBF-style runtime work, x32 and x64-family work, memo systems, xindex backends, data dictionary code, and metadata catalogs.</li>
<li>Treat DotTalk++ as the canonical runtime and teaching shell, not merely a small scripting add-on.</li>
