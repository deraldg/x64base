<h2 id="source-of-truth"><a class="no-underline" href="#source-of-truth">Source of truth</a></h2>
<p>The authority order is:</p>
<ol>
<li>Runtime/source behavior in <code>D:\code\ccode</code>.</li>
<li>HELP, metadata, command registration, comments, <code>@dottalk.usage</code>, <code>@dottalk.contract</code>, contract scans, and canary evidence.</li>
<li>SelfDoc reports and manualgen manifests.</li>
<li>Reviewed manual sections.</li>
<li>Website pages in <code>D:\dev\x64base-site</code>.</li>
</ol>
<p>When a DotTalk++ command or function is added, the expected flow is:</p>
<pre><code class="language-text">source change
  -> command/function registration
  -> source header usage contracts and source comments
  -> HELP / metadata / comments evidence / contract scans
  -> SelfDoc and CMDHELPCHK validation
  -> manualgen inventory / validate / dry-run
  -> reviewed manual section
  -> website command/function/API pages
</code></pre>

<h2 id="promotion-rules"><a class="no-underline" href="#promotion-rules">Promotion rules</a></h2>
<ul>
<li>Do not auto-promote uncertain scanner output into public copy.</li>
<li>Preserve status labels: runtime-evidenced, source-evidenced, help-catalog-evidenced, planned, canary, review-needed.</li>
<li>Do not claim a lane is complete when the evidence says active beta, canary, or review-needed.</li>
<li>Keep generated reports local until reviewed, then summarize them on the website.</li>
<li>Command and function pages should cite the extraction snapshot and should be regenerated when HELP/CMDHELP/usage contracts change.</li>
<li>Commands that launch external apps, perform network requests, write files, write HELP DATA, or write DBFs must carry explicit <code>mutates:</code> and <code>risk:</code> fields before being promoted as stable docs.</li>
</ul>
<h2 id="procedural-curation"><a class="no-underline" href="#procedural-curation">Procedural curation</a></h2>
<p>The repeatable curation loop is:</p>
<pre><code class="language-text">scan source headers
-> harvest comments and usage contracts
-> rebuild or inspect HELP/CMDHELP surfaces
-> run CMDHELPCHK or report-only validators
-> run manualgen inventory/validate/dry-run
-> review generated sections
-> update website derivatives
</code></pre>
<p><code>CMDHELP</code>, <code>CMDHELPCHK</code>, <code>MAINT</code>, <code>DDICT</code>, <code>MANUAL</code>, <code>BBOX</code>, and contract tools all belong in this loop. Their documentation should state mutation boundaries explicitly so a report-only inspection command does not get mistaken for a promotion command.</p>
