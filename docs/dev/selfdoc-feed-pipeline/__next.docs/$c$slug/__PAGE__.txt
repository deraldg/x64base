6:T381b,<p>The SelfDoc feed is the repeatable path from implementation evidence to manuals,
diagrams, and public website pages. It exists so x64base and DotTalk++ can
describe themselves without turning memory, chat notes, or website prose into
technical truth.</p>
<p><a href="/images/sdlc/selfdoc-feed-pipeline-map-v2.svg"><img src="/images/sdlc/selfdoc-feed-pipeline-map-v2.svg" alt="SelfDoc evidence-to-publication pipeline with authority, promotion gates, reviewed outputs, and recursive drift feedback"></a></p>
<p>Select the diagram to inspect the full-resolution evidence and promotion map. The feedback path is intentional: publication review can report drift, but corrections return to the owning source or metadata lane.</p>
<p>The doctrine is intentionally strict:</p>
<ul>
<li>runtime proves;</li>
<li>source defines;</li>
<li>HELP explains;</li>
<li>metadata organizes;</li>
<li>SelfDoc preserves;</li>
<li>MDO curates;</li>
<li>manualgen packages;</li>
<li>the website publishes reviewed derivatives.</li>
</ul>
<h2 id="what-the-feed-does"><a class="no-underline" href="#what-the-feed-does">What The Feed Does</a></h2>
<p>The feed gives every public technical claim a path back to evidence. A command
page should not merely say that a command exists; it should be traceable to
registration, HELP, usage contracts, source comments, command-reference harvests,
runtime proof, or a reviewed manual section. A diagram should not be decorative;
it should be attached to the same evidence spine as the text it explains.</p>
<p>This matters most for the surfaces that can drift quickly:</p>

































<table><thead><tr><th>Surface</th><th>Why It Needs The Feed</th></tr></thead><tbody><tr><td>Commands and functions</td><td>Names, aliases, arguments, mutation behavior, and proof status change as the shell grows.</td></tr><tr><td>SET family and settings</td><td>Runtime defaults, scoped behavior, and command families need source-derived readback.</td></tr><tr><td>Data mutators</td><td><code>REPLACE</code>, <code>CALC</code>, <code>CALCWRITE</code>, <code>MULTIREP</code>, import/export, commit, rollback, and table-buffer commands need explicit dirty/stale/risk labels.</td></tr><tr><td>File formats</td><td>DBF, VFP, MS-DOS, DBF_64, memo, and index claims must trace back to headers, readers, writers, probes, or specs.</td></tr><tr><td>Workspaces and areas</td><td>Workspace wrappers, dbarea behavior, relations, buffers, and cursor control are runtime concepts, not brochure claims.</td></tr><tr><td>Diagrams and screenshots</td><td>Visuals are curated artifacts; they need owners, attachment rows, and review state.</td></tr></tbody></table>
<h2 id="authority-order"><a class="no-underline" href="#authority-order">Authority Order</a></h2>
<p>The authority order is:</p>
<ol>
<li>Runtime/source behavior in <code>implementation checkout</code>.</li>
<li>HELP, metadata, command registration, comments, <code>@dottalk.usage</code>,
<code>@dottalk.contract</code>, contract scans, and canary evidence.</li>
<li>SelfDoc reports and manualgen manifests.</li>
<li>Reviewed manual sections.</li>
<li>Website pages in <code>website source tree</code>.</li>
</ol>
<p>When a DotTalk++ command or function is added, the expected flow is:</p>
<pre><code class="language-text">source change
  -> command/function registration
  -> source header usage contracts and source comments
  -> HELP / metadata / comments evidence / contract scans
  -> SelfDoc and CMDHELPCHK validation
  -> manualgen inventory / validate / dry-run
  -> reviewed manual section
  -> website command/function/API pages
</code></pre>
<p>The website is allowed to organize, summarize, and teach. It is not allowed to
become the upstream authority for runtime behavior.</p>
<h2 id="feed-lanes"><a class="no-underline" href="#feed-lanes">Feed Lanes</a></h2>
<p>The current feed is not one monolithic generator. It is a set of lanes that
should converge before public promotion.</p>





















































<table><thead><tr><th>Lane</th><th>Inputs</th><th>Outputs</th><th>Promotion Gate</th></tr></thead><tbody><tr><td>Source/runtime lane</td><td>C++ source, headers, build files, scripts, runtime transcripts, canaries</td><td>implementation facts, proof notes, smoke outcomes</td><td>Must be source- or runtime-evidenced.</td></tr><tr><td>HELP lane</td><td>HELP, CMDHELP, CMDHELPCHK, command metadata, help tables</td><td>command descriptions, argument notes, missing-topic reports</td><td>Must preserve command identity and status.</td></tr><tr><td>Contract lane</td><td><code>@dottalk.usage</code>, <code>@dottalk.contract</code>, comments, mutation labels</td><td>usage contracts, risk flags, public command safety notes</td><td>Mutating commands must say what they mutate.</td></tr><tr><td>SelfDoc lane</td><td>scanners, inventories, crosswalk builders, report-only probes</td><td>CSV/MD reports, feature crosswalks, candidate findings</td><td>Uncertain findings stay labeled.</td></tr><tr><td>MDO lane</td><td>organized report shelves, section maps, diagram ledgers</td><td>manual sections, publication decisions, attachment maps</td><td>MDO curates; it does not erase source uncertainty.</td></tr><tr><td>Manualgen lane</td><td>inventories, validate runs, dry-run builds, accepted sections</td><td>reviewed manuals and downloadable/manual-ready artifacts</td><td>Validate before publication; note environment limits.</td></tr><tr><td>Website lane</td><td>reviewed manual sections, evidence summaries, public media</td><td>x64base.com pages and diagrams</td><td>Publication derivative only.</td></tr></tbody></table>

<h2 id="status-labels"><a class="no-underline" href="#status-labels">Status Labels</a></h2>
<p>Public pages should preserve status instead of smoothing it over.</p>





































<table><thead><tr><th>Label</th><th>Meaning</th></tr></thead><tbody><tr><td>runtime-evidenced</td><td>Observed in a runtime transcript, smoke test, canary, or generated proof output.</td></tr><tr><td>source-evidenced</td><td>Found in source, headers, command registration, parser code, or build files.</td></tr><tr><td>help-catalog-evidenced</td><td>Present in HELP/CMDHELP/help-data/catalog surfaces.</td></tr><tr><td>manual-reviewed</td><td>Accepted into a reviewed manual section or manualgen lane.</td></tr><tr><td>planned</td><td>A real lane or design goal, but not implemented.</td></tr><tr><td>canary</td><td>Exists as a probe, experiment, or limited proof path.</td></tr><tr><td>review-needed</td><td>Scanner output or memory note found something that still needs human or runtime confirmation.</td></tr></tbody></table>
<h2 id="promotion-rules"><a class="no-underline" href="#promotion-rules">Promotion Rules</a></h2>
<ul>
<li>Do not auto-promote uncertain scanner output into public copy.</li>
<li>Preserve status labels: runtime-evidenced, source-evidenced,
help-catalog-evidenced, manual-reviewed, planned, canary, review-needed.</li>
<li>Do not claim a lane is complete when the evidence says active beta, canary, or
review-needed.</li>
<li>Keep generated reports local until reviewed, then summarize them on the
website.</li>
<li>Command and function pages should cite the extraction snapshot and should be
regenerated when HELP/CMDHELP/usage contracts change.</li>
<li>Commands that launch external apps, perform network requests, write files,
write HELP DATA, or write DBFs must carry explicit <code>mutates:</code> and <code>risk:</code>
fields before being promoted as stable docs.</li>
<li>Treat source/manual-to-website as the normal simplex flow. Treat
website-to-manual as blocked unless the web artifact is separately maintained
and not derivable from the implementation checkout.</li>
<li>Treat <code>README.*</code> files as sacred orientation artifacts. If a README must be
replaced, preserve the previous version as a versioned artifact before
promotion.</li>
</ul>
<h2 id="procedural-curation"><a class="no-underline" href="#procedural-curation">Procedural Curation</a></h2>
<p><a href="/images/sdlc/parallel-lanes-procedural-curation-v2.svg"><img src="/images/sdlc/parallel-lanes-procedural-curation-v2.svg" alt="Six coordinated procedural-curation lanes showing collection, explanation, normalization, preservation, MDO review, packaging, artifacts, and promotion gates"></a></p>
<p>Select the diagram for a full-resolution inspection view. Each lane has a named
responsibility, a durable artifact, and an explicit gate; no lane can silently
promote its own output.</p>















































<table><thead><tr><th>Step</th><th>Owner</th><th>Durable Artifact</th><th>Required Gate</th></tr></thead><tbody><tr><td>Collect</td><td>Source/runtime maintainer</td><td>Evidence snapshot, transcript, canary, or source citation</td><td>Behavior is observed or source-evidenced.</td></tr><tr><td>Explain</td><td>HELP and contract owner</td><td>Command identity, usage, mutation, and risk semantics</td><td><code>CMDHELPCHK</code> and contract validation agree.</td></tr><tr><td>Normalize</td><td>Metadata curator</td><td>Registry row, alias/family mapping, status, and provenance</td><td>Identity is reconciled and uncertainty remains labeled.</td></tr><tr><td>Preserve</td><td>SelfDoc lane</td><td>Versioned report, crosswalk, proof link, or drift finding</td><td>Provenance is retained; scanner output is not auto-promoted.</td></tr><tr><td>Curate</td><td>MDO reviewer</td><td>Accept, defer, reject, and target-attachment decision</td><td>A human owner approves both claim and publication target.</td></tr><tr><td>Package</td><td>Manual and website maintainers</td><td>Reviewed section, diagram, page, and release metadata</td><td>Build, route checks, and public-content guard pass.</td></tr></tbody></table>
<p>The repeatable curation loop is:</p>
<pre><code class="language-text">scan source headers
-> harvest comments and usage contracts
-> rebuild or inspect HELP/CMDHELP surfaces
-> run CMDHELPCHK or report-only validators
-> run manualgen inventory/validate/dry-run
-> review generated sections
-> update website derivatives
</code></pre>
<p><code>CMDHELP</code>, <code>CMDHELPCHK</code>, <code>MAINT</code>, <code>DDICT</code>, <code>MANUAL</code>, <code>BBOX</code>, and contract tools
all belong in this loop. Their documentation should state mutation boundaries
explicitly so a report-only inspection command does not get mistaken for a
promotion command.</p>
<h2 id="manual-and-website-intake-gate"><a class="no-underline" href="#manual-and-website-intake-gate">Manual And Website Intake Gate</a></h2>
<p>Manual generation should not scrape public website prose as a shortcut for
technical truth. The website is a publication derivative; the manual should
consume the same upstream evidence that fed the website whenever that evidence
exists.</p>
<p>Allowed manual intake from the website is limited to artifacts that are:</p>
<ol>
<li>not derivable from source, HELP, metadata, SelfDoc, MDO, or manualgen;</li>
<li>intentionally maintained as website/public artifacts; and</li>
<li>labeled with provenance and owner.</li>
</ol>
<p>Examples include curated screenshots, public navigation/framing copy, contact
details, branding pages, hosted-download metadata, or a website-only media
artifact. Runtime claims, command syntax, file-format claims, index behavior,
data-mutator behavior, and engine feature claims must go back to the
implementation checkout and generated evidence lanes.</p>
<h2 id="diagram-lane"><a class="no-underline" href="#diagram-lane">Diagram Lane</a></h2>
<p>Diagrams are not decoration. They are attached explanatory views over the same
evidence spine.</p>
<p>Current rule:</p>
<pre><code class="language-text">prove or verify the lane
-> identify manual and website targets
-> attach a canonical diagram source
-> render/promote a website asset only after review
-> preserve the attachment row so the public image never becomes orphan truth
</code></pre>
<p>The current attachment ledger is:</p>
<ul>
<li><code>docs/manuals/developer/manualgen/reports/diagram-publication-attachment-matrix-v1.csv</code></li>
</ul>
<p>That ledger crosswalks canonical manual SVGs, LabTalk diagram sources, and
promoted site image assets so the publication surface stays traceable.</p>
