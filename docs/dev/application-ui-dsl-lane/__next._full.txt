17:T143c,<h2 id="position"><a class="no-underline" href="#position">Position</a></h2>
<p>DotTalk++ can already write database/runtime programs with commands and DotScript.
It does not yet provide a general application UI definition language like the
classic xBase environments did.</p>
<p>This lane is a proposed development path for moving from scripts that operate on
data to scripts that can define small database applications.</p>
<h2 id="current-boundary"><a class="no-underline" href="#current-boundary">Current Boundary</a></h2>
<p>DotTalk++ currently has:</p>
<ul>
<li>CLI command programs.</li>
<li>DotScript scripts.</li>
<li>Supplied GUI/TUI templates.</li>
<li>SmartList, SmartBrowser, Browse, and relation browser surfaces.</li>
<li>Arctic as the TUI code-name surface.</li>
<li>Parallel GUI/TUI as a supplied workbench lane.</li>
</ul>
<p>DotTalk++ does <strong>not</strong> yet have a user-facing UI definition language for custom
menus, windows, dialogs, popups, toolbars, form controls, or event-driven
application screens.</p>
<h2 id="proposed-syntax-seed"><a class="no-underline" href="#proposed-syntax-seed">Proposed Syntax Seed</a></h2>
<pre><code class="language-text">CREATE MENU MainMenu
CREATE WINDOW CustomerWin
CREATE DIALOG EditCustomer
DEFINE BUTTON Save
ON CLICK DO save_customer
</code></pre>
<p>This is not implemented syntax yet. It is a lane seed for design, source
contracts, examples, and eventual runtime proof.</p>
<h2 id="proposed-scope"><a class="no-underline" href="#proposed-scope">Proposed Scope</a></h2>





































<table><thead><tr><th>UI construct</th><th>Intended purpose</th></tr></thead><tbody><tr><td><code>CREATE MENU</code> / <code>ENDMENU</code></td><td>Define named application menus and menu items.</td></tr><tr><td><code>CREATE WINDOW</code> / <code>ENDWINDOW</code></td><td>Define top-level windows or workbench panels.</td></tr><tr><td><code>CREATE DIALOG</code> / <code>ENDDIALOG</code></td><td>Define modal or modeless dialogs.</td></tr><tr><td><code>DEFINE BUTTON</code></td><td>Define buttons and bind them to commands or scripts.</td></tr><tr><td><code>DEFINE FIELD</code></td><td>Bind form fields to DBF fields, variables, or expressions.</td></tr><tr><td><code>DEFINE GRID</code> / <code>DEFINE BROWSE</code></td><td>Place data grids and browses inside application screens.</td></tr><tr><td><code>ON CLICK</code>, <code>ON CHANGE</code>, <code>ON SELECT</code></td><td>Define event handlers that run DotTalk++ commands or DotScript routines.</td></tr></tbody></table>
<h2 id="backend-strategy"><a class="no-underline" href="#backend-strategy">Backend Strategy</a></h2>
<p>The same UI definition should be able to target more than one surface over time:</p>
<ul>
<li>Arctic TUI first, for keyboard-oriented menus, windows, dialogs, and command
forms.</li>
<li>Parallel GUI/TUI next, through shared engine services.</li>
<li>wxWidgets GUI later, where native windows and controls are needed.</li>
<li>Web UI later, when a browser-facing application surface is appropriate.</li>
</ul>
<p>The UI DSL should not duplicate table mechanics. It should bind to existing
engine services: work areas, relations, table buffers, commands, functions,
validation, messaging, and help metadata.</p>
<h2 id="comparison-context"><a class="no-underline" href="#comparison-context">Comparison Context</a></h2>
<p>Classic xBase environments often treated interface construction as part of the
application language:</p>
<ul>
<li>Visual FoxPro supported forms, menus, windows, popups, controls, and event
handlers.</li>
<li>dBASE Plus exposed form and menu designers that generated editable source.</li>
<li>Alaska Xbase++ supports character mode, GUI mode, hybrid mode, dialogs,
application windows, and menu systems.</li>
<li>Harbour and xHarbour rely heavily on GUI libraries such as MiniGUI, HwGUI,
wxHarbour, GTK, and Qt bindings.</li>
<li>XSharp can use .NET UI stacks such as WinForms and WPF.</li>
</ul>
<p>The DotTalk++ lane should be described as <strong>planned</strong>, not implemented, until
there is command registry evidence, runtime proof, examples, and generated
manual coverage.</p>
<h2 id="proof-gates"><a class="no-underline" href="#proof-gates">Proof Gates</a></h2>
<p>Before this lane is presented as active implementation, it needs:</p>
<ol>
<li>Syntax contract and examples.</li>
<li>Command registry entries.</li>
<li>TUI proof for menu, window, dialog, button, and event handler.</li>
<li>HELP/CMDHELP coverage.</li>
<li>SelfDoc metadata coverage.</li>
<li>Manualgen section.</li>
<li>Website comparison update.</li>
</ol>
<h2 id="website-wording"><a class="no-underline" href="#website-wording">Website Wording</a></h2>
<p>Use:</p>
<ul>
<li>"planned application UI DSL lane"</li>
<li>"proposed syntax seed"</li>
<li>"future path for menus, windows, dialogs, controls, and event handlers"</li>
<li>"targeting Arctic TUI first, with GUI/web backends later"</li>
</ul>
<p>Avoid:</p>
<ul>

<li>"custom GUI applications can be built without code today"</li>
<li>"drop-in replacement for Visual FoxPro forms"</li>
