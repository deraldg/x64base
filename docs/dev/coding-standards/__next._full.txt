<h2 id="code"><a class="no-underline" href="#code">Code</a></h2>
<ul>
<li>Prefer small functions with explicit ownership.</li>
<li>Use stable command, field, table, and lane names.</li>
<li>Keep runtime behavior and public documentation in sync.</li>
<li>Route filesystem, network, external-process, and database mutation through explicit command contracts.</li>
<li>Preserve existing worktree changes; do not use broad rewrites for narrow documentation or command updates.</li>
</ul>
<h2 id="command-header-contracts"><a class="no-underline" href="#command-header-contracts">Command Header Contracts</a></h2>
<p>Contracts are not decorative comments. They are the public source-of-truth layer
that lets DotTalk++ connect implementation, HELP, CMDHELP, SelfDoc, manualgen,
diagrams, website pages, and runtime proof. If a command can be run by a user,
called from DotScript, surfaced in HELP, included in CMDHELPCHK, or promoted to
the website, it needs a contract.</p>
<p>The contract should be close to the implementation, easy to harvest, and honest
about status. A reader should be able to tell whether the command is read-only,
session-mutating, table-mutating, filesystem-mutating, network-active,
externally active, or publication-active without reading the whole command body.</p>
<p>Every command intended for HELP, CMDHELP, SelfDoc, manualgen, or the website should have a leading <code>@dottalk.usage v1</code> block or an equivalent reviewed contract. Preferred fields:</p>
<pre><code class="language-text">@dottalk.usage v1
owner: DOT|COMMAND
command: COMMAND
aliases: ALIAS1, ALIAS2
category: ...
status: supported|experimental|sample-extension|...
noargs: usage|report|status|...
effect: report|display|evaluate|mutate|rebuild|session-control|mixed|network-or-file|...
mutates: none|session|shell-exit-state|table-data|table-buffer|memo|index|filesystem|helpdata|metadata|manualgen|sidecar|...
usage-access: COMMAND USAGE
summary:
  Short behavior statement.
usage:
  COMMAND USAGE
  COMMAND ...
notes:
  Runtime boundary notes and compatibility behavior.
risk:
  reads_files: yes|no
  writes_files: yes|no
  launches_external_app: yes|no
  network_access: yes|no
  mutates_table_data: yes|no
  writes_table_buffer: yes|no
  writes_memo: yes|no
  mutates_index: yes|no
  requires_open_table: yes|no
  requires_current_record: yes|no
related:
  HELP
  CMDHELPCHK
@dottalk.end
</code></pre>
<p>These headers are harvested into comments evidence, HELP/CMDHELP rows, validation reports, manualgen, diagrams, and website pages.</p>
<h3 id="required-contract-questions"><a class="no-underline" href="#required-contract-questions">Required Contract Questions</a></h3>
<p>Before a command is considered documented, its contract should answer:</p>
<ul>
<li>What is the canonical command name?</li>
<li>Are there aliases or compatibility spellings?</li>
<li>Is it current, planned, experimental, canary, sample-extension, or deprecated?</li>
<li>What happens with no arguments?</li>
<li>Does it read or write table records?</li>
<li>Does it write a table buffer instead of the physical DBF?</li>
<li>Does it mark fields dirty or stale?</li>
<li>Does it write memo payloads or memo object references?</li>
<li>Does it mutate indexes, index metadata, order state, or LMDB/CDX/CNX files?</li>
<li>Does it read or write ordinary files?</li>
<li>Does it launch an editor, browser, image viewer, shell, URL handler, SFTP/SSH tool, or other external app?</li>
<li>Does it touch HELP DATA, CMDHELPCHK, SelfDoc, manualgen, metadata, message catalogs, or publication artifacts?</li>
<li>What proof or smoke path should validate it?</li>
</ul>
<h3 id="mutation-classes"><a class="no-underline" href="#mutation-classes">Mutation Classes</a></h3>
<p>Use narrow mutation labels. Avoid saying only <code>mutates: yes</code>.</p>





















































<table><thead><tr><th>Class</th><th>Meaning</th></tr></thead><tbody><tr><td><code>none</code></td><td>Report-only or display-only.</td></tr><tr><td><code>session</code></td><td>Changes in-memory state such as selected area, order, filter, paths, language, or shell flags.</td></tr><tr><td><code>table-data</code></td><td>Writes physical DBF records, deletes/recalls records, appends rows, packs/zaps, or changes stored field values.</td></tr><tr><td><code>table-buffer</code></td><td>Writes buffered table state and may mark dirty or stale fields without immediate physical writeback.</td></tr><tr><td><code>memo</code></td><td>Writes memo payloads, memo object IDs, or memo sidecar/backing-store state.</td></tr><tr><td><code>index</code></td><td>Rebuilds, updates, attaches, selects, or mutates INX/CNX/CDX/LMDB index state.</td></tr><tr><td><code>filesystem</code></td><td>Creates, deletes, copies, fetches, edits, exports, imports, or rewrites ordinary files.</td></tr><tr><td><code>external-app</code></td><td>Opens an editor, image viewer, URL, shell, SFTP/SSH, browser, or platform launcher.</td></tr><tr><td><code>helpdata</code></td><td>Mutates HELP, CMDHELP, message, locale, or generated help catalogs.</td></tr><tr><td><code>metadata</code></td><td>Mutates Data Dictionary, SYS*, metadata, SelfDoc, comments, or contract tables.</td></tr><tr><td><code>manualgen</code></td><td>Mutates manual catalog, accepted manual sections, generated manuals, or publication shelves.</td></tr></tbody></table>
<h3 id="special-command-families"><a class="no-underline" href="#special-command-families">Special Command Families</a></h3>
<p>Some families require extra care because they can look harmless while crossing a
real boundary.</p>
<ul>
<li><strong>Data mutators</strong>: <code>REPLACE</code>, <code>REPLACE_MULTI</code>, <code>MULTIREP</code>, <code>CALCWRITE</code>, assignment-capable <code>CALC</code>, <code>APPEND</code>, <code>APPEND BLANK</code>, <code>DELETE</code>, <code>RECALL</code>, <code>PACK</code>, and <code>ZAP</code> must state physical-write versus buffered-write behavior.</li>
<li><strong>Index mutators</strong>: <code>INDEX</code>, <code>REINDEX</code>, <code>CDX</code>, <code>CNX</code>, <code>LMDB</code>, <code>BUILDLMDB</code>, <code>SET CDX</code>, <code>SET CNX</code>, <code>SET LMDB</code>, <code>SET ORDER</code>, and related compact forms must state whether table data, index files, order state, or backend environments are changed.</li>
<li><strong>Filesystem and external tools</strong>: <code>COPY</code>, <code>EXPORT</code>, <code>IMPORT</code>, <code>DDL FETCH</code>, <code>WEB</code>, <code>URL</code>, <code>IMAGE</code>, <code>EDIT</code>, shell commands, SFTP/SSH, archive, and zip commands must state path resolution, writes, external launch behavior, and network behavior.</li>
<li><strong>SelfDoc/help/manual commands</strong>: <code>HELP</code>, <code>CMDHELP</code>, <code>CMDHELPCHK</code>, <code>MAINT</code>, <code>DDICT</code>, <code>MANUAL</code>, <code>MANSTAR</code>, <code>BBOX</code>, message-manager commands, and contract scanners must say whether they are report-only or authorized to mutate catalogs.</li>
<li><strong>Session lifecycle</strong>: <code>QUIT</code>, <code>EXIT</code>, <code>CLOSE</code>, <code>WORKSPACE</code>, <code>SELECT</code>, <code>USE</code>, <code>SET</code>, and path/language settings must state whether they mutate only session state or also close files, release areas, or change persistent data.</li>
</ul>
<h3 id="dotref-and-registry-alignment"><a class="no-underline" href="#dotref-and-registry-alignment">DOTREF And Registry Alignment</a></h3>
<p>For user-facing commands, three surfaces should stay aligned:</p>
<ol>
<li>Runtime registration in the command registry.</li>
<li>Source contract in the implementation file.</li>
<li>Native DotTalk++ reference entry in <code>include/dotref.hpp</code>.</li>
</ol>
<p>Compatibility-only or historical material can live in <code>foxref.hpp</code>, but current
DotTalk++ commands belong in DOTREF. If a command is intentionally omitted from
DOTREF, the omission should be documented as internal, deprecated, or hidden.</p>
<p>Command-shell lambda registration is allowed, but lambda handlers still need command contracts, DOTREF review, and catalog visibility. Inline lambdas should stay small and delegate meaningful behavior to named helpers or <code>cmd_*.cpp</code> implementations when the command grows. See <a href="/docs/dev/naming-conventions">Naming Conventions</a>.</p>
<h2 id="comments-and-collection"><a class="no-underline" href="#comments-and-collection">Comments and Collection</a></h2>
<ul>
<li>Put command-level contracts at the top of the implementation file.</li>
<li>Keep usage examples ASCII-safe unless the file already requires Unicode.</li>
<li>Mark compatibility aliases explicitly.</li>
<li>State whether a command is read-only, report-only, filesystem-mutating, DBF-mutating, HELP-mutating, or externally active.</li>
<li>Use status labels such as <code>planned</code>, <code>canary</code>, <code>experimental</code>, or <code>review-needed</code> instead of burying uncertainty.</li>
<li>If a command writes HELP DATA, DBF files, sidecars, fetched files, or launches an external app, say so in <code>mutates:</code> and <code>risk:</code>.</li>
</ul>
<p>Current tooling includes <code>tools/comments/upsert_source_comment_contract.py</code> for staged source-comment imports and <code>tools/contracts/contract_scan.py</code> for contract-lane inventory.</p>
<h2 id="safeguards"><a class="no-underline" href="#safeguards">Safeguards</a></h2>
<ul>
<li>Validation commands should be report-only unless a specific work order authorizes writes.</li>
<li><code>CMDHELPCHK</code>, <code>MAINT</code>, <code>DDICT</code>, and <code>MANUAL</code> should remain read-only inspection surfaces unless their source contract changes.</li>
<li><code>CMDHELP BUILD</code> is allowed to mutate HELP DATA because that is its explicit command contract.</li>
<li><code>DDL FETCH</code>, <code>WEB FETCH</code>, <code>IMAGE</code>, <code>EDIT</code>, shell, and SFTP-style commands must document external app, network, and filesystem boundaries separately from table mutation.</li>
<li>Manualgen publication replacement requires review; dry-run artifacts are not automatic publications.</li>
</ul>
<h2 id="accessibility-and-inclusive-design-standards"><a class="no-underline" href="#accessibility-and-inclusive-design-standards">Accessibility and Inclusive Design Standards</a></h2>
<p>x64base is a scholastic system, so accessibility and inclusive design are
ground-level engineering requirements. Public pages, generated manuals,
diagrams, downloads, and UI surfaces should target WCAG 2.2 AA-style behavior
unless a platform constraint is explicitly documented.</p>
<p>Use these requirements for website work, generated documentation, GUI/TUI work,
manuals, and published artifacts:</p>
<ul>
<li>Non-text content needs a text alternative. Image links need real link text,
<code>alt</code> text, or an <code>aria-label</code>; complex diagrams need captions and a nearby
text summary.</li>
<li>Color must never be the only signal. Status should include text labels such
as <code>runtime-evidenced</code>, <code>planned</code>, <code>canary</code>, <code>review-needed</code>, or <code>public-ready</code>.</li>
<li>Normal prose must remain readable with browser zoom and operating-system
scaling. Avoid fixed pixel-only layouts that force horizontal scrolling for
ordinary reading.</li>
<li>Keyboard navigation is required for links, menus, tabs, cards, downloads,
forms, and command surfaces. Visible focus state is required.</li>
<li>Pages should use semantic landmarks and heading order: one logical <code>h1</code>,
meaningful <code>h2</code>/<code>h3</code> hierarchy, <code>header</code>, <code>nav</code>, <code>main</code>, and <code>footer</code> where
appropriate.</li>
<li>Icon-only controls need accessible names. Decorative icons should be hidden
from assistive technology.</li>
<li>Tables need real headers. Command catalogs, function catalogs, feature
matrices, and comparison tables should remain understandable to screen
readers.</li>
<li>Code blocks should not be the only explanation. Provide a short prose
summary for command examples, DotScript examples, and generated snippets.</li>
<li>Motion should respect <code>prefers-reduced-motion</code>; avoid flashing or decorative
animation that can interfere with reading.</li>
<li>Downloadable PDFs, DOCX files, large reports, and decks should have an HTML
or Markdown summary when possible, plus type, date, source/provenance, proof
status, and accessibility status.</li>
</ul>
<p>Accessibility status is part of provenance. Promoted artifacts should carry
labels such as:</p>
<pre><code class="language-text">accessibility:
