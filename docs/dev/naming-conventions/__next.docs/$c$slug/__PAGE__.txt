<h2 id="publication-naming-boundary"><a class="no-underline" href="#publication-naming-boundary">Publication naming boundary</a></h2>
<p>The project has both historical lineage names and current public product names.
Those are not interchangeable on publication surfaces.</p>
<p>Use this rule:</p>













































<table><thead><tr><th>Context</th><th>Preferred published name</th><th>Notes</th></tr></thead><tbody><tr><td>Engine, storage runtime, shared core, repository center of gravity</td><td><code>x64base</code></td><td>Use for DBF-family runtime truth, x64 table architecture, memo/index/work-area internals, and the main public engine repo.</td></tr><tr><td>Interactive runtime shell, DotScript/manual/help command surface</td><td><code>DotTalk++</code></td><td>This is the canonical runtime/manual identity. Use it in page titles, prose, manual sections, screenshots, and command-surface explanations.</td></tr><tr><td>Education/campus/portal/assignments/proof orchestration</td><td><code>LabTalk</code></td><td>Consumer/overlay layer, not engine truth.</td></tr><tr><td>GUI/TUI aligned workbench family</td><td><code>DotTalk++ Workbench</code></td><td>Public family name for aligned front ends.</td></tr><tr><td>TUI codename under the workbench lane</td><td><code>Arctic</code></td><td>Keep as codename/sub-lane, not the primary project identity.</td></tr><tr><td>Historical Fox lineage references</td><td><code>FoxPro</code>, <code>Foxtalk</code>, <code>foxref</code></td><td>Keep only where historical explanation or compatibility lineage matters. Do not let these become current product branding.</td></tr><tr><td>Historical route bucket / internal content grouping</td><td><code>dottalk</code></td><td>Acceptable as a slug or internal directory bucket for continuity, but not as the preferred published product name.</td></tr></tbody></table>
<h3 id="route-and-folder-rule"><a class="no-underline" href="#route-and-folder-rule">Route and folder rule</a></h3>
<p>The website may keep historical, short, or compatibility-safe route buckets such as:</p>
<pre><code class="language-text">content/docs/dottalk
/docs/dottalk/*
</code></pre>
<p>That does <strong>not</strong> mean the public-facing product name should revert to <code>dottalk</code>.</p>
<p>Publication rule:</p>
<ul>
<li>slugs and folder names may stay stable for continuity.</li>
<li>visible page titles, navigation labels, product copy, and prose should use the current published identity.</li>
<li>if a route bucket is historical, the page should still present the canonical name in the title and description.</li>
</ul>
<p>Therefore:</p>
<ul>
<li><code>/docs/dottalk/command-catalog</code> is acceptable as a stable route.</li>
<li>the page title should still say <code>Command Catalog</code> under the <code>DotTalk++</code> documentation family.</li>
<li>navigation labels should prefer <code>DotTalk++</code>, not bare <code>dottalk</code>, unless a historical note is intended.</li>
</ul>
<h3 id="drift-prevention-rule"><a class="no-underline" href="#drift-prevention-rule">Drift-prevention rule</a></h3>
<p>When a name appears in source, docs, or the site, ask which layer owns it:</p>
<ol>
<li>historical lineage</li>
<li>source-tree bucket</li>
<li>runtime/manual identity</li>
<li>engine/platform identity</li>
<li>consumer/education identity</li>
</ol>
<p>If the answer is only <code>historical lineage</code> or <code>source-tree bucket</code>, do not promote it into top-level branding by accident.</p>

<h2 id="necessary-directory-trees"><a class="no-underline" href="#necessary-directory-trees">Necessary directory trees</a></h2>
<p>The development tree is intentionally wider than the public product tree. The useful mental model is not "everything ships"; it is "source, proof, generated documentation, and staged publication each have a role."</p>
<h3 id="development-tree"><a class="no-underline" href="#development-tree">Development tree</a></h3>
<pre><code class="language-text">implementation checkout
|-- src\                 runtime source: CLI, xbase, indexes, memo, GUI bridges
|-- include\             public/internal headers and engine contracts
|-- dottalkpp\           runtime data, scripts, generated reports, maintenance lanes
|-- selfdoc\             documentation collection and source-contract probes
|-- labtalk\             Laboratory Campus plans, reports, products, and lessons
|-- tools\               manualgen, datadict, help, and support tooling
|-- scripts\             smoke scripts, build helpers, and DTS examples
|-- tests\               regression and manual test surfaces
|-- docs\                source-side specifications and design records
|-- apps\                app/workbench experiments
|-- pycrud\              Python CRUD/GUI lane
|-- py\                  Python helpers and experiments
|-- contracts\           contract and coding-standard support material
|-- cases\               case-study and proof material
|-- samples\             sample data/code surfaces
|-- results\             generated or captured run results
|-- build*\              local build outputs, not product truth
`-- third_party\         external dependencies and vendored support
</code></pre>
<h3 id="public-website-source-and-staged-product-tree"><a class="no-underline" href="#public-website-source-and-staged-product-tree">Public website source and staged product tree</a></h3>

<h2 id="command-implementation-convention"><a class="no-underline" href="#command-implementation-convention">Command implementation convention</a></h2>
<p>Command files should use <code>cmd_&#x3C;canonical_command>.cpp</code> where practical. The command's public identity should be aligned across:</p>
<ol>
<li>command registration in the shell.</li>
<li>the source contract near the implementation.</li>
<li>DOTREF entry in <code>include/dotref.hpp</code>.</li>
<li>HELP/CMDHELP/CMDHELPCHK rows.</li>
<li>website command catalog and manualgen output.</li>
</ol>
<p>Compatibility or historical references can live in <code>foxref.hpp</code>; current DotTalk++ commands belong in DOTREF unless intentionally hidden or internal.</p>
<h2 id="command-shell-lambda-registration"><a class="no-underline" href="#command-shell-lambda-registration">Command-shell lambda registration</a></h2>
<p>Some commands are registered directly in the command shell with lambda handlers rather than as standalone <code>cmd_*.cpp</code> functions. That is allowed, but it needs a convention because lambda registration is easy to lose in documentation scans.</p>
<p>Lambda-registered commands should:</p>
<ul>
<li>use an explicit canonical command name and alias list in the registration call.</li>
<li>include a nearby <code>@dottalk.usage v1</code> contract or reference a reviewed contract block.</li>
<li>state whether the lambda is report-only, session-mutating, table-mutating, filesystem-mutating, external-app-active, or network-active.</li>
<li>remain small; if the lambda grows beyond simple dispatch/glue, move behavior into a named helper or <code>cmd_*.cpp</code> implementation.</li>
<li>be visible to command catalog generation, DOTREF review, and CMDHELPCHK.</li>
<li>avoid hiding mutators, filesystem writers, or external app launchers inside inline lambdas.</li>
</ul>
<p>Preferred shape:</p>
<pre><code class="language-text">registry.registerCommand("COMMAND", aliases, [](ShellContext&#x26; ctx, Args args) {
  return run_named_command_helper(ctx, args);
});
</code></pre>
<p>Avoid putting complex business logic directly inside the lambda. The lambda should usually bind command syntax to a named implementation, not become the implementation.</p>
<h2 id="education-command-convention"><a class="no-underline" href="#education-command-convention">Education command convention</a></h2>
<p>Education commands use <code>edu_</code> when they are primarily teaching surfaces. If an education command is also a normal command-shell command, document both identities:</p>
<ul>
<li>source file prefix: <code>edu_</code></li>
<li>shell command: public command name such as <code>ASCII</code>, <code>RETRO</code>, <code>BBOX</code>, or <code>IDX</code></li>
<li>lesson lane: Laboratory Campus / education feature page</li>
<li>proof: transcript, screenshot, source contract, or <code>.dts</code> lab</li>
</ul>
<p>Student extension examples should live under extension paths and use explicit sample status. They should not overwrite core command names.</p>
<h2 id="function-convention"><a class="no-underline" href="#function-convention">Function convention</a></h2>
<p>Function implementations and registrations should use <code>fn_</code> or a clearly grouped function catalog source. Functions should state:</p>
<ul>
<li>canonical function name.</li>
<li>aliases, if any.</li>
<li>argument count and type expectations.</li>
<li>return type.</li>
<li>whether evaluation is pure or can read/mutate external state.</li>
<li>whether the function is core, sample-extension, or planned.</li>
</ul>
<p>Scalar functions used by mutating commands such as <code>CALC</code>, <code>CALCWRITE</code>, and <code>REPLACE</code> deserve extra review because expression behavior can become data mutation.</p>
<h2 id="script-and-proof-naming"><a class="no-underline" href="#script-and-proof-naming">Script and proof naming</a></h2>
<p><code>.dts</code> files should be named for intent and risk:</p>









































<table><thead><tr><th>Pattern</th><th>Use</th></tr></thead><tbody><tr><td><code>*_smoke.dts</code></td><td>Small smoke proof.</td></tr><tr><td><code>*_canary.dts</code></td><td>Probe for a specific behavior or regression.</td></tr><tr><td><code>*_regression*.dts</code></td><td>Regression script with expected behavior.</td></tr><tr><td><code>*_mutation*.dts</code></td><td>Script that changes data; should use sandbox/disposable data.</td></tr><tr><td><code>*_verify*.dts</code></td><td>Readback or validation script.</td></tr><tr><td><code>cmdhelpchk_*</code></td><td>HELP/CMDHELP/CMDHELPCHK evidence lane.</td></tr><tr><td><code>SOURCE_COMMENT_*</code></td><td>Source comment and contract collection lane.</td></tr><tr><td><code>MCC_*</code></td><td>My Community College dataset/proof lane.</td></tr></tbody></table>
<p>Every promoted <code>.dts</code> lesson should state:</p>
<ul>
<li>required dataset.</li>
<li>expected read/write behavior.</li>
<li>whether it is safe observation or mutation.</li>
<li>expected output or transcript anchor.</li>
<li>proof status.</li>
</ul>
<h2 id="documentation-convention"><a class="no-underline" href="#documentation-convention">Documentation convention</a></h2>
<p>Website and manual pages should avoid presenting active lanes as finished products. Use these status labels consistently:</p>
<ul>
<li>runtime-evidenced</li>
<li>source-evidenced</li>
<li>help-catalog-evidenced</li>
<li>generated-reviewed</li>
<li>planned-or-in-progress</li>
<li>review-needed</li>
</ul>
