6:T241a,<h2 id="not-code-first-documentation-later"><a class="no-underline" href="#not-code-first-documentation-later">Not code first, documentation later</a></h2>
<p>Co-development does not mean that an engine is built first and documented
afterward. <strong>x64base, DotTalk++, and their documentation infrastructure are
interdependent parts of one working system.</strong></p>
<ul>
<li><strong>x64base is the stateful database engine.</strong> It owns tables, fields, records,
memos, indexes, relations, work areas, buffers, validation, and storage
lifecycles.</li>
<li><strong>DotTalk++ is the engine's executable language and observation surface.</strong> It
names operations, invokes engine behavior, exposes state, runs DotScript, and
supplies the command boundary used by people, tests, GUIs, and automation.</li>
<li><strong>HELP, messages, metadata, contracts, SelfDoc, and MDO are the explanatory
and verification structure.</strong> They preserve what operations are called, how
they are used, what they mutate, their development status, and the evidence
supporting them.</li>
</ul>
<p>Documentation is therefore not merely a report <em>about</em> the engine. Its
registrations, contracts, HELP identities, status records, validation reports,
and missing-proof findings actively shape engine and runtime work. Source and
runtime remain authoritative, but documentation completeness is itself an
engineering test.</p>
<h2 id="the-engineruntimedocumentation-triad"><a class="no-underline" href="#the-engineruntimedocumentation-triad">The engine/runtime/documentation triad</a></h2>

























<table><thead><tr><th>Layer</th><th>Primary responsibility</th><th>Co-development dependency</th></tr></thead><tbody><tr><td>x64base engine</td><td>Own database state and storage behavior.</td><td>Exposes stable concepts, lifecycle hooks, validation state, and observable outcomes.</td></tr><tr><td>DotTalk++ runtime</td><td>Name, execute, script, and inspect engine operations.</td><td>Every command needs registration, HELP identity, contracts, messages, status, and proof paths.</td></tr><tr><td>Documentation system</td><td>Explain, inventory, crosswalk, validate, preserve, and publish.</td><td>Missing documentation reveals unstable names, hidden mutation boundaries, incomplete contracts, or insufficient evidence.</td></tr></tbody></table>
<p>None of these layers substitutes for another. Together they make the system
inspectable, teachable, testable, and increasingly self-describing.</p>
<h2 id="recursion-is-the-operating-model"><a class="no-underline" href="#recursion-is-the-operating-model">Recursion is the operating model</a></h2>
<pre><code class="language-text">x64base state and storage behavior
    -> DotTalk++ commands, scripts, inspection, and runtime evidence
    -> HELP + messages + registrations + usage/mutation contracts
    -> metadata normalization + SelfDoc preservation + MDO review
    -> manuals + diagrams + Laboratory Campus lessons + public pages
    -> review exposes drift, missing contracts, hidden behavior, or weak proof
    -> findings return to x64base, DotTalk++, metadata, and tests
    -> the next pass strengthens both behavior and explanation
</code></pre>
<p>The loop is recursive, but authority is not circular. Website prose does not
become engine truth merely because it is published. Generated material cannot
promote itself. Provenance, status labels, and human review govern every return
path.</p>
<h2 id="how-documentation-changes-implementation"><a class="no-underline" href="#how-documentation-changes-implementation">How documentation changes implementation</a></h2>
<p>The documentation system applies concrete engineering pressure:</p>
<ul>
<li>a command without stable registration and HELP identity is incomplete;</li>
<li>a mutator without usage, mutation, risk, stale, and dirty semantics is
incomplete;</li>
<li>an engine feature without an observable command or API path is difficult to
teach or prove;</li>
<li>a public claim without source or runtime provenance is blocked;</li>
<li>contradictory HELP, metadata, and behavior become a drift finding;</li>
<li>a missing failure transcript or canary becomes a proof requirement;</li>
<li>manual and website organization expose gaps in the engine's conceptual model.</li>
</ul>
<p>The constraint also works in reverse. Documentation cannot promise a
transaction, index, field type, API, or command behavior that x64base and
DotTalk++ do not actually support. That bidirectional constraint is the center
of co-development.</p>
<h2 id="three-cooperating-systems"><a class="no-underline" href="#three-cooperating-systems">Three cooperating systems</a></h2>
<h3 id="1-the-x64base-and-dottalk-engineruntime-pair"><a class="no-underline" href="#1-the-x64base-and-dottalk-engineruntime-pair">1. The x64base and DotTalk++ engine/runtime pair</a></h3>
<p>x64base supplies database state and behavior. DotTalk++ makes that behavior
readable, executable, scriptable, inspectable, and available to parallel
CLI/TUI/GUI work. Their source and captured runtime evidence are primary
technical truth.</p>
<h3 id="2-embedded-documentation-and-laboratory-campus"><a class="no-underline" href="#2-embedded-documentation-and-laboratory-campus">2. Embedded documentation and Laboratory Campus</a></h3>
<p>HELP, messages, metadata, contracts, SelfDoc, and MDO collect and preserve the
system's own definitions and evidence. Laboratory Campus, with LabTalk retained
as its project and historical name, turns reviewed material into diagrams,
datasets, cases, demonstrations, labs, and lessons.</p>
<h3 id="3-manuals-and-website-systems"><a class="no-underline" href="#3-manuals-and-website-systems">3. Manuals and website systems</a></h3>
<p>Generated manuals, <a href="https://x64base.com/">x64base.com</a>, and
<a href="https://dottalkpp.com/">dottalkpp.com</a> publish reviewed derivatives. They are
also active diagnostic surfaces: making the system coherent for readers reveals
omissions and contradictions that recurse into the next development pass.</p>
<h2 id="the-websites-are-development-instruments"><a class="no-underline" href="#the-websites-are-development-instruments">The websites are development instruments</a></h2>
<p>The two sites operate at different scales:</p>




















<table><thead><tr><th>Website</th><th>Primary role</th><th>Co-development feedback</th></tr></thead><tbody><tr><td>x64base.com</td><td>Main ecosystem and public integration surface for the engine, DotTalk++, products, architecture, Laboratory Campus, evidence, downloads, and governance.</td><td>Tests whether the entire system can be explained coherently without unsupported claims or hidden status.</td></tr><tr><td>dottalkpp.com</td><td>Deeper manual, DotScript, generated-reference, relationship, and artifact room.</td><td>Tests documentary depth, artifact continuity, cross-links, generated output, and whether detailed material remains navigable.</td></tr></tbody></table>
<p>A route that cannot be populated from governed evidence, a command family that
cannot be organized coherently, an orphan diagram, a broken cross-site
relationship, or contradictory status copy is a <strong>development finding</strong>. It
returns through MDO and SelfDoc to the responsible metadata, HELP, source, test,
manual, or website lane.</p>
<p>The websites do not redefine runtime truth. They test whether that truth is
complete, reachable, internally consistent, and teachable. In that sense,
x64base.com and dottalkpp.com are part of the recursive development apparatus,
not advertisements added after the work is done.</p>
<h2 id="governing-doctrine"><a class="no-underline" href="#governing-doctrine">Governing doctrine</a></h2>
<blockquote>
<p><strong>Runtime proves, source defines, HELP explains, metadata organizes, SelfDoc
preserves, MDO curates, and reviewed publication teaches.</strong></p>
</blockquote>
<h2 id="authority-and-safeguards"><a class="no-underline" href="#authority-and-safeguards">Authority and safeguards</a></h2>
<ul>
<li>Runtime and source remain authoritative for implementation claims.</li>
<li>SelfDoc and metadata preserve provenance rather than manufacturing proof.</li>
<li>Existing, developing, planned, and experimental work retain honest status.</li>
<li>Website review may identify a source defect; changing prose alone does not
repair it.</li>
<li>Manual and website exchange follows governed simplex and duplex gates.</li>
<li>Canonical <code>README.*</code> files are preserved and versioned when replaced.</li>
</ul>
