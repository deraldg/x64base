17:T277d,<h2 id="core-rule"><a class="no-underline" href="#core-rule">Core rule</a></h2>
<p><code>implementation checkout</code> is the implementation truth.</p>
<p><code>website source tree</code> is the public publication surface.</p>
<p>The site should present reviewed, provenance-aware derivatives of the runtime,
contracts, HELP, metadata, manuals, and diagrams. It should not invent
technical truth on its own.</p>
<h2 id="public-source"><a class="no-underline" href="#public-source">Public source</a></h2>
<ul>
<li>Public GitHub repo: <a href="https://github.com/deraldg/x64base">deraldg/x64base</a></li>
<li>Local implementation tree: <code>implementation checkout</code></li>
<li>Local DotTalk++ runtime tree: <code>DotTalk++ runtime tree</code></li>
<li>Local Laboratory Campus tree: <code>Laboratory Campus tree</code></li>
<li>Local site tree: <code>website source tree</code></li>
<li>Local runtime mirror: <code>local staging mirror</code></li>
<li>Public reviewed derivative: <a href="https://x64base.com/">x64base.com</a></li>
</ul>
<h2 id="observed-maintainer-flow"><a class="no-underline" href="#observed-maintainer-flow">Observed maintainer flow</a></h2>
<p>Use this as the normal direction of travel:</p>
<pre><code class="language-text">source/runtime:
implementation checkout -> local staging mirror -> GitHub repository

website:
website source tree -> build/public artifact -> GitHub Pages -> x64base.com
</code></pre>
<h2 id="shared-document-model"><a class="no-underline" href="#shared-document-model">Shared document model</a></h2>
<p>Use this split:</p>
<ul>
<li>Runtime/source/contracts/manual truth lives in <code>implementation checkout</code></li>
<li>Public website framing and navigation live in <code>website source tree</code></li>
<li><code>local staging mirror</code> is the clean staging mirror for source/runtime promotion</li>
<li>Built website output moves through reviewed build/public artifacts before
GitHub Pages publication</li>
<li>Shared diagrams and publication artifacts should be promoted from reviewed
SelfDoc/manual outputs, not rewritten from memory</li>
</ul>
<h2 id="repeatable-feed"><a class="no-underline" href="#repeatable-feed">Repeatable feed</a></h2>
<p>For the report-only source-to-website feed, see <a href="/docs/dev/selfdoc-feed-pipeline">SelfDoc Feed Pipeline</a>.</p>
<p>When commands or functions are added to DotTalk++, they should flow through registration, HELP, metadata, comments, usage contracts, SelfDoc/CMDHELPCHK validation, manualgen, and then the public command/function/API pages.</p>
<h2 id="dottalk-artifactmanual-room"><a class="no-underline" href="#dottalk-artifactmanual-room">DotTalk++ artifact/manual room</a></h2>
<p>The <a href="/docs/dev/public-site-architecture">Public Site Architecture</a> provides
separate diagrams for x64base.com and dottalkpp.com plus a governed connection
map showing their reciprocal links, independent release histories, and blocked
reverse-authority path.</p>
<p>The DotTalk++ artifact/manual room now supports this publication model from its
public HTTP home while SSL is still settling:</p>
<ul>
<li>Artifact room: <a href="http://dottalkpp.com/">http://dottalkpp.com/</a></li>
<li>Manuals: <a href="http://dottalkpp.com/manuals/">http://dottalkpp.com/manuals/</a></li>
<li>DotScript: <a href="http://dottalkpp.com/dotscript/">http://dottalkpp.com/dotscript/</a></li>
<li>Reference: <a href="http://dottalkpp.com/reference/">http://dottalkpp.com/reference/</a></li>
<li>Generated Docs: <a href="http://dottalkpp.com/generated/">http://dottalkpp.com/generated/</a></li>
<li>Downloads: <a href="http://dottalkpp.com/downloads/">http://dottalkpp.com/downloads/</a></li>
<li>Governance: <a href="http://dottalkpp.com/governance/">http://dottalkpp.com/governance/</a></li>
</ul>
<p>Its role is artifact support: larger manuals, generated reports, command and
function reference exports, proof packets, accessibility text versions,
downloadable bundles, and governance manifests. x64base.com remains the main
public website and the engine/ecosystem home.</p>
<p>The route and artifact manifest must remain reviewed, and the public-content
guard must pass before publication. Move these links to HTTPS only after the
certificate is verified.</p>
<p>Authority rule: the artifact/manual-room site is downstream from implementation
evidence, HELP, metadata, SelfDoc, MDO, and manualgen. It may feed selected
summary links back to x64base.com, but it must not become a substitute source
for command syntax, file formats, mutator behavior, indexing behavior, or
runtime truth.</p>
<h2 id="simplex-and-duplex-gates"><a class="no-underline" href="#simplex-and-duplex-gates">Simplex and duplex gates</a></h2>
<p>The website, manuals, and implementation tree are connected, but they do not
have equal authority.</p>
<p>Rule 1: most technical data on <code>x64base.com</code> must be derived, harvested,
calculated, or reviewed from the implementation checkout. The website may frame
and organize that information, but it should not become the source of runtime,
command, field-format, index, memo, HELP, or manual truth.</p>
<p>Rule 2: the manual may contribute reviewed sections, diagrams, and summaries to
the website. The manual must not consume website copy as technical truth unless
the website artifact is explicitly non-derivable from the implementation tree
and is maintained separately as a public artifact.</p>
<p>Rule 3: <code>README.*</code> files are sacred orientation artifacts. They may be
improved, annotated, or superseded, but a replacement must be versioned so the
older README remains recoverable and comparable. Do not casually overwrite a
README during website, manual, or staging work.</p>
<p>Use these gates:</p>




























































<table><thead><tr><th>Path</th><th>Gate Type</th><th>Rule</th></tr></thead><tbody><tr><td>implementation checkout -> manual</td><td>Simplex default</td><td>Source, HELP, metadata, comments, contracts, canaries, and SelfDoc/manualgen reports feed the manual.</td></tr><tr><td>implementation checkout -> website</td><td>Simplex default</td><td>Website pages should be derivatives of source/runtime/manual evidence, not hand-written replacements for it.</td></tr><tr><td>implementation checkout -> DotTalk++ artifact/manual room</td><td>Simplex default</td><td>Manuals, references, generated docs, downloads, proof packets, and governance should derive from source/runtime/HELP/metadata/SelfDoc/MDO/manualgen evidence.</td></tr><tr><td>manual -> website</td><td>Duplex, reviewed</td><td>Reviewed manual sections may promote into website summaries when proof labels and source anchors remain intact.</td></tr><tr><td>DotTalk++ artifact/manual room -> website</td><td>Duplex, reviewed</td><td>The artifact room may contribute selected summary links, route maps, and downloadable artifact notices to x64base.com after review.</td></tr><tr><td>website -> manual</td><td>Blocked by default</td><td>Do not let website prose become manual truth. Only ingest website-owned artifacts that are non-derivable elsewhere.</td></tr><tr><td>website -> DotTalk++ artifact/manual room</td><td>Blocked by default</td><td>Do not let ecosystem prose become technical manual truth unless the artifact is website-owned and non-derivable elsewhere.</td></tr><tr><td>website-owned artifact -> manual</td><td>Exception gate</td><td>Allowed for public-only artifacts such as curated screenshots, branding/UX copy, published contact details, or hosted downloads, when the artifact has an owner and provenance.</td></tr><tr><td>manual -> implementation checkout</td><td>Review gate</td><td>Manual discoveries should become source contracts, TODOs, tests, HELP changes, or SelfDoc tasks before they become runtime claims.</td></tr><tr><td>README replacement</td><td>Versioned artifact gate</td><td>Preserve the prior README version before replacing it; README history is part of project provenance.</td></tr></tbody></table>
<p>If a manual chat wants to "steal" from the website, it should first ask whether
the website text is derivative of the implementation tree. If yes, the manual
should return to the original source/manualgen/SelfDoc evidence. If no, the
manual may cite or ingest the website artifact only after recording why that
artifact is independently maintained.</p>
<h2 id="front-end-naming-rule"><a class="no-underline" href="#front-end-naming-rule">Front-end naming rule</a></h2>
<ul>
<li><code>DotTalk++ Workbench</code> is the family name for front-end/workbench surfaces</li>
<li><code>Parallel GUI/TUI</code> is the public product lane for aligned graphical and terminal interfaces</li>
<li><code>Arctic</code> is the code name for the TUI surface</li>
<li><code>TableTalk</code> is the table-buffering lane, not generic table browsing</li>
<li><code>Foxtalk</code> and fox-derived names remain valid lineage/subsystem names</li>
<li>Classic FoxPro-style syntax may remain for historical and educational reasons</li>
<li>DotTalk++ should not be described as a clone</li>
</ul>
<h2 id="route-bucket-naming-rule"><a class="no-underline" href="#route-bucket-naming-rule">Route-bucket naming rule</a></h2>
<ul>
<li>short route buckets such as <code>/docs/dottalk/*</code> may remain stable for continuity</li>
<li>those route buckets are not the preferred public brand</li>
<li>visible page titles, navigation labels, and prose should use <code>DotTalk++</code></li>
<li>historical names should be explained, not silently promoted into top-level branding</li>
</ul>
<h2 id="educational-goal"><a class="no-underline" href="#educational-goal">Educational goal</a></h2>
<p>The stack is meant to teach full-stack thinking from beginning to end:</p>
<ul>
<li>engine and file/runtime behavior</li>
<li>command shell and automation</li>
<li>HELP, metadata, and contracts</li>
<li>manuals and diagrams</li>
<li>GUI/TUI/front-end work</li>
<li>public website publication</li>
</ul>
