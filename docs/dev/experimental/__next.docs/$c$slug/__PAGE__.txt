<h2 id="hierarchical-work"><a class="no-underline" href="#hierarchical-work">Hierarchical Work</a></h2>
<p>The <code>HIER</code> command is more than an idea, but less than a supported general
hierarchical database product. Its source contract identifies it as a
<code>dev-hierarchy</code> command and warns that it can mutate hierarchy fields in the
current table.</p>
<p>Current command forms include:</p>
<pre><code class="language-text">HIER CREATE ROOT &#x3C;node_id>
HIER ADD CHILD &#x3C;parent_id> &#x3C;node_id>
HIER INSERT BETWEEN &#x3C;left_id> &#x3C;right_id> &#x3C;node_id>
HIER MOVE &#x3C;node_id> TO &#x3C;new_parent_id>
HIER DELETE &#x3C;node_id>
HIER DELETE SUBTREE &#x3C;node_id>
HIER REBUILD
HIER VALIDATE
HIER CHILDREN &#x3C;node_id>
HIER SUBTREE &#x3C;node_id>
</code></pre>
<p>This deserves controlled experiments around path rebuilding, subtree movement,
orphan detection, cycles, duplicate node identifiers, ordering, and rollback.
Until those experiments are repeatable, the website should call it a developer
command rather than a supported database model.</p>
<h2 id="dewey-work"><a class="no-underline" href="#dewey-work">Dewey Work</a></h2>
<p>The repository currently contains two related but distinct Dewey directions:</p>
<ol>
<li><strong>Dewey-style coordinates for hierarchy indexing.</strong> Prototype interfaces and
benchmark material explore subtree, direct-child, insert-between, and
path-to-root operations.</li>
<li><strong>Dewey Decimal as educational material.</strong> The current education translation
unit is explicitly an empty placeholder. It should not appear in the command
catalog as an available learner command.</li>
</ol>
<p>The naming overlap should not be allowed to blur the distinction between a
hierarchical coordinate scheme and a library-classification lesson. Each needs
its own contract, examples, and proof state.</p>
<h2 id="rushmore-work"><a class="no-underline" href="#rushmore-work">Rushmore Work</a></h2>
<p>The current source has an important precursor: <code>LOCATE</code> can use a simple CDX
fast path when a simple predicate matches the active tag. Complex predicates
fall back to selector and expression evaluation.</p>
<p>That is evidence of index-assisted search, but it is not enough to claim a
general Rushmore implementation. A broader Rushmore-inspired lane would need to
make these decisions explicit:</p>
<ul>
<li>which predicate shapes are optimizable;</li>
<li>whether multiple indexed predicates can be combined;</li>
<li>whether candidate-record sets or bitmaps are used;</li>
<li>how deleted, filtered, scoped, and ordered views interact;</li>
<li>when the engine falls back to sequential evaluation;</li>
<li>how the chosen plan is explained to the learner or developer;</li>
<li>how correctness and performance are measured separately.</li>
</ul>
<p>The initial research artifact should compare the existing simple-CDX path with
the fallback scan on fixed datasets. Naming can come later; behavior and proof
should come first.</p>
<h2 id="graduation-rules"><a class="no-underline" href="#graduation-rules">Graduation Rules</a></h2>
<p>An experiment should not move into supported documentation because a source
file exists. Promotion requires an appropriate subset of these gates:</p>
<ol>
<li><strong>Ownership</strong> - identify the engine, DotTalk++, workbench, SelfDoc, or campus
owner.</li>
<li><strong>Contract</strong> - document usage, effects, mutation, risk, prerequisites, and
unsupported cases.</li>
<li><strong>Build</strong> - prove the intended Windows and POSIX build surfaces.</li>
<li><strong>Runtime</strong> - run repeatable canaries against controlled data.</li>
<li><strong>Safety</strong> - use disposable data for mutating experiments and prove failure
behavior.</li>
<li><strong>Help</strong> - align runtime usage, DOTREF, CMDHELP, metadata, and command catalog.</li>
<li><strong>Measurement</strong> - record dataset, machine context, command, result, and
limitations for performance claims.</li>
<li><strong>Review</strong> - promote only after source, runtime, and documentation evidence
agree.</li>
</ol>
