c:Te4c,<h2 id="source-of-truth"><a class="no-underline" href="#source-of-truth">Source of truth</a></h2>
<p>Use <code>D:\code\ccode</code> as the authoritative local source tree for DotTalk++ / x64base work.</p>
<p>Use <code>D:\dev\x64base-site</code> as the website publication tree.</p>
<p>Use <code>C:\x64base\dottalk-webui\public-site</code> as the local runtime mirror for the
built website before GitHub Pages publication.</p>
<p>The site content was updated from a scan of:</p>
<ul>
<li>CMake build configuration.</li>
<li>command reference docs.</li>
<li>GUI docs under <code>docs/gui</code>.</li>
<li>database safety docs under <code>docs/database</code>.</li>
<li>source directories under <code>src/xbase</code>, <code>src/xindex</code>, <code>src/memo</code>, <code>src/cli</code>, <code>src/datadict</code>, and <code>bindings/pydottalk</code>.</li>
</ul>
<p>See <a href="/docs/dev/project-truth">Current Project Truth</a>.
See <a href="/docs/dev/selfdoc-website-publication">SelfDoc Website Publication</a>.</p>
<h2 id="working-principles"><a class="no-underline" href="#working-principles">Working principles</a></h2>
<ul>
<li>Preserve the difference between runtime-proven, active, partial, experimental, and canary features.</li>
<li>Treat DotTalk++ as the canonical runtime and command surface.</li>
<li>Treat x64base as the engine/runtime/data layer, not as a separate polished product unless the source tree proves that packaging.</li>
<li>Keep GUI descriptions top-down over the existing runtime. Do not imply wxWidgets, Tk, or TUI lanes own database semantics.</li>
<li>Treat the website as downstream of SelfDoc, contracts, manuals, and reviewed runtime docs.</li>
<li>Keep public GitHub links current where the site describes public work.</li>
<li>Avoid placeholder CLI commands when the local command reference has real commands.</li>
</ul>
<h2 id="shared-document-workflow"><a class="no-underline" href="#shared-document-workflow">Shared document workflow</a></h2>
<ul>
<li>Runtime/engineering truth belongs first in <code>D:\code\ccode</code>.</li>
<li>Publication-ready manuals, contracts, and diagrams should be reviewed there before site promotion.</li>
<li>The website should mirror or curate those reviewed materials, not replace them.</li>
<li>Website publication should flow through <code>D:\dev\x64base-site</code> to
<code>C:\x64base\dottalk-webui\public-site</code>, then to GitHub Pages and
<code>x64base.com</code>.</li>
<li>If a technical claim diverges between the two trees, fix <code>D:\code\ccode</code> first and then republish the site.</li>
</ul>
<h2 id="education-goal"><a class="no-underline" href="#education-goal">Education goal</a></h2>
<p>One explicit project goal is to learn front-end work through the x64base /
DotTalk++ / LabTalk stack.</p>
<p>That means learning:</p>
<ul>
<li>how runtime truth feeds UI truth,</li>
<li>how CLI, TUI, GUI, and website layers align,</li>
<li>how SelfDoc and manuals feed public-facing documentation,</li>
<li>how to build a full-stack educational database system from source to site.</li>
</ul>
<h2 id="build-boundaries"><a class="no-underline" href="#build-boundaries">Build boundaries</a></h2>
<p>Do not mutate DBFs, HELP data, metadata catalogs, or generated publication artifacts unless the task explicitly asks for that and the mutation path is understood.</p>
<p>The local CMake metacollect block calls out a safety boundary: no HELP DATA rebuild, no <code>CMDHELPCHK</code> mutation, and no DBF writes.</p>
