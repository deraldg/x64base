## Interpretation

The contracts lane is justified.

The repo already contains:

- many design and policy documents that function as contracts,
- source annotations that explicitly declare contracts,
- a large command usage-contract surface,
- manualgen/datadict/locale/maintenance policy artifacts,
- GUI/database contracts recently seeded.

Without a lane, these remain discoverable only by search, memory, or manual
review. With a lane, they can be inventoried, classified, registered, validated,
and promoted.

## First Drift Class

The first scan found many likely unregistered contract documents, especially
under:

- `docs/datadict`,
- `docs/locale`,
- `docs/maintenance`,
- `docs/manuals/developer/manualgen`,
- `docs/governance`,
- `docs/gui`.

This does not mean those documents are wrong. It means they need triage:

- active contract,
- historical report,
- superseded contract,
- rejected/closed candidate,
- or non-contract note with softened wording.

## Second Drift Class

The stricter registry crosswalk found active registry rows whose source is not
detected by the current scanner. This is usually because the source document is
real but its filename does not contain the scanner's contract-like keywords.

That is useful signal, not necessarily an error. These rows need one of:

- keep registered and accept that the scanner does not detect that source class,
- add explicit contract markers,
- add scanner rules for that source class,
- or move the source into a more obvious contract filename/path.

## Usage Marker Surface

The scanner found 261 source files with usage-marker evidence.

This confirms that usage contracts are currently the largest and most mature
contract family. They should become the reference implementation for contract
harvest, registry mapping, and HELP/manual promotion.
