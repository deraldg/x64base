1b:T59d,<h2 id="what-it-does"><a class="no-underline" href="#what-it-does">What it does</a></h2>
<p>TableTalk names the table-buffering lane exposed through DotTalk++.</p>
<p>Its subject is buffered table state:</p>
<ul>
<li>whether table buffering is ON or OFF</li>
<li>whether an area is clean or DIRTY</li>
<li>whether fields are fresh or STALE</li>
<li>which records have buffered changes</li>
<li>how <code>COMMIT</code> and <code>ROLLBACK</code> finish or discard pending edits</li>
<li>how mutators such as <code>REPLACE</code> and <code>CALCWRITE</code> interact with the buffer</li>
</ul>
<p>Real local command surfaces include:</p>
<pre><code class="language-text">SET TABLE BUFFER ON
SET TABLE BUFFER OFF
SET TABLE BUFFER ON ALL
SET TABLE BUFFER OFF ALL
TABLE
TABLE BUFFER STATUS
TABLE BUFFER DUMP
COMMIT
ROLLBACK
REPLACE &#x3C;field> WITH &#x3C;value>
CALCWRITE &#x3C;field> = &#x3C;expr>
</code></pre>
<p>These workflows are about table-buffer state, not generic table browsing. Use
SmartList, SmartBrowse, SimpleBrowse, and the browser pages for visible
listing/browsing surfaces.</p>
<p>Runtime evidence should show output like <code>buffer=ON</code>, <code>DIRTY</code>, <code>STALE [field]</code>,
and buffered row counts before promoting a TableTalk claim.</p>
<p>See <a href="/docs/dev/project-truth">Current Project Truth</a> and the canonical public
