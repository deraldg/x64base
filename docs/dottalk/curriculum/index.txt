<p>The current goal is to turn existing DotTalk++ examples into repeatable lessons with prerequisites, expected output, mutation warnings, and proof labels.</p>

<h2 id="module-1---tables-and-records"><a class="no-underline" href="#module-1---tables-and-records">Module 1 - Tables and records</a></h2>
<p>Learning goals:</p>
<ul>
<li>define records, fields, field types, work areas, and current record state.</li>
<li>inspect fixed-length records without hiding the file-format reality.</li>
<li>compare classic DBF, VFP DBF, and DBF_64 where evidence exists.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\canaries\dottalk_beta0_smoke_baseline_0_1.dts
dottalkpp\data\scripts\canaries\create_sample_dbf_matrix.dts
tests\manual\dotscript\core\MCC_TIER1_CORE_SURFACE_AND_INDEX_SAFE_V1.DTS
tests\manual\dotscript\focused\version.dts
</code></pre>
<p>Lesson shape:</p>
<ol>
<li>open a known table.</li>
<li>list fields and records.</li>
<li>inspect current area and cursor state.</li>
<li>compare a command transcript with the website command catalog.</li>
</ol>
<h2 id="module-2---memos"><a class="no-underline" href="#module-2---memos">Module 2 - Memos</a></h2>
<p>Learning goals:</p>
<ul>
<li>explain why memo fields are not ordinary fixed-width field payloads.</li>
<li>distinguish memo references, memo stores, memo objects, and memo integrity checks.</li>
<li>understand why FPT64/DTX work is a 64-bit architecture lane, not just a larger file.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\canaries\memo_shakedown.dts
dottalkpp\data\scripts\MEMO_X64_REOPEN_CANARY_20260513.dts
dottalkpp\data\scripts\memo_tests.dts
</code></pre>
<p>Next correction: publish a read-only memo lab before any mutation lab so students can learn pointers, blocks, and integrity without risking sample data.</p>
<h2 id="module-3---indexes"><a class="no-underline" href="#module-3---indexes">Module 3 - Indexes</a></h2>
<p>Learning goals:</p>
<ul>
<li>define key expressions, order state, tags, rebuilds, and verification.</li>
<li>compare education timing (<code>IDX</code>) with persistent index work (<code>CNX</code>, <code>CDX</code>, <code>LMDB</code>).</li>
<li>read diagnostics honestly, including mismatch or stale backend states.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\canaries\cnx_tag_smoke.dts
dottalkpp\data\scripts\canaries\order_engine_shakedown.dts
dottalkpp\data\scripts\canaries\smartlist_order_matrix.dts
dottalkpp\data\scripts\canaries\lmdb_open_shakedown.dts
tests\manual\dotscript\focused\CNX_TAG_SMOKE.dts
tests\manual\dotscript\focused\list_ordered_students_cnx_baseline_x32_v2.dts
</code></pre>
<p>Lesson shape:</p>
<ol>
<li>build or attach an index.</li>
<li>change order.</li>
<li>list ordered records.</li>
<li>record what changed in session state versus files.</li>
<li>compare a clean proof with a diagnostic proof.</li>
</ol>
<h2 id="module-4---relational-thinking"><a class="no-underline" href="#module-4---relational-thinking">Module 4 - Relational thinking</a></h2>
<p>Learning goals:</p>
<ul>
<li>distinguish selection, projection, relation traversal, tuple output, and joined readback.</li>
<li>use workspaces as wrappers over active DB areas.</li>
<li>inspect relation trees without pretending the system is a full SQL server.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\cases\ersatz_mini_test.dts

</code></pre>
<h2 id="module-5---dotscript-and-automation"><a class="no-underline" href="#module-5---dotscript-and-automation">Module 5 - DotScript and automation</a></h2>
<p>Learning goals:</p>
<ul>
<li>write repeatable command files.</li>
<li>use variables, comments, line continuation, <code>IF</code>, <code>LOOP</code>, <code>WHILE</code>, <code>UNTIL</code>, <code>SCAN</code>, and one-level nesting where supported.</li>
<li>separate safe observation scripts from mutating scripts.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\canaries\dt_var_date_concat_calc_smoketest_full_start.dts

<h2 id="module-6---mutation-safety"><a class="no-underline" href="#module-6---mutation-safety">Module 6 - Mutation safety</a></h2>
<p>Learning goals:</p>
<ul>
<li>explain physical writes versus buffered writes.</li>
<li>inspect dirty and stale state.</li>
<li>treat <code>REPLACE</code>, <code>CALC</code>, <code>CALCWRITE</code>, and <code>MULTIREP</code> as special commands requiring proof.</li>
<li>use disposable or sandbox datasets for mutation labs.</li>
</ul>
<p>Candidate <code>.dts</code> examples:</p>
<pre><code class="language-text">dottalkpp\data\scripts\canaries\mcc_mutation_trinity_disposable_driver.dts

</code></pre>
<h2 id="module-7---self-documenting-systems"><a class="no-underline" href="#module-7---self-documenting-systems">Module 7 - Self-documenting systems</a></h2>
