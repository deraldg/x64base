17:T26fd,<p><code>REPLACE</code>, <code>CALC</code>, <code>CALCWRITE</code>, and <code>MULTIREP</code>/<code>REPLACE_MULTI</code> deserve special handling in the documentation because they can change table data. They sit at the boundary between scalar expression evaluation and physical DBF/memo/index mutation.</p>
<p>This page is a reviewed website derivative of the current source contracts in <code>src/cli/cmd_replace.cpp</code>, <code>src/cli/cmd_calc.cpp</code>, <code>src/cli/cmd_calcwrite.cpp</code>, <code>src/cli/cmd_replace_multi.cpp</code>, and the table-buffer state code.</p>
<h2 id="command-roles"><a class="no-underline" href="#command-roles">Command roles</a></h2>








































<table><thead><tr><th>Command</th><th>Mutation role</th><th>Current source-backed behavior</th></tr></thead><tbody><tr><td><code>CALC &#x3C;expr></code></td><td>Read-only scalar evaluation</td><td>Evaluates an <code>xexpr</code> scalar or field-aware expression and prints Bool, Number, String, Date, or <code>.F.</code> fallback output.</td></tr><tr><td><code>CALC &#x3C;field> = &#x3C;expr></code></td><td>Conditional data mutation</td><td>If the left side is a real field in the open current area, <code>CALC</code> delegates to <code>CALCWRITE</code>. If not, it remains expression evaluation.</td></tr><tr><td><code>CALCWRITE &#x3C;field> = &#x3C;expr></code></td><td>Single-field expression write</td><td>Evaluates RHS with <code>xexpr</code>, normalizes for the target field type, handles currency and x64 memo fields, then writes or buffers the field.</td></tr><tr><td><code>REPLACE &#x3C;field> WITH &#x3C;value></code></td><td>Single-field replace</td><td>Resolves field by index/name, evaluates/dequotes RHS through the REPLACE expression path, validates/normalizes, handles x64 memo fields, then writes or buffers the field.</td></tr><tr><td><code>MULTIREP &#x3C;field> WITH &#x3C;value>, ...</code></td><td>Multi-field direct write</td><td>Alias surface for <code>REPLACE_MULTI</code>; parses multiple assignments, validates all assignments before write, then applies one record lock and one physical DBF write.</td></tr><tr><td><code>REPLACE_MULTI &#x3C;field> WITH &#x3C;value>, ...</code></td><td>Multi-field direct write</td><td>Canonical implementation behind <code>MULTIREP</code>; also used programmatically by UI surfaces such as staged browser saves.</td></tr></tbody></table>
<p>Scalar functions are not data mutators by themselves. They become part of mutation only when a mutating command uses them to compute a value that is written to a field.</p>
<h2 id="buffered-versus-direct-writes"><a class="no-underline" href="#buffered-versus-direct-writes">Buffered versus direct writes</a></h2>






























<table><thead><tr><th>Path</th><th>TABLE buffer ON</th><th>TABLE buffer OFF / direct</th></tr></thead><tbody><tr><td><code>REPLACE</code></td><td>Adds a table-buffer <code>CHANGE_UPDATE</code>, marks area dirty, marks changed field stale. <code>COMMIT</code> owns durable application.</td><td>Calls <code>DbArea::replaceFieldStored</code>, which owns the direct write path and index replace-snapshot behavior. If the visible value changes, the field is marked stale.</td></tr><tr><td><code>CALCWRITE</code></td><td>Adds a table-buffer <code>CHANGE_UPDATE</code>, marks area dirty, marks changed field stale. <code>COMMIT</code> owns durable application.</td><td>Calls <code>DbArea::replaceFieldStored</code>, preserving direct-write/index behavior. If the visible value changes, the field is marked stale.</td></tr><tr><td><code>CALC</code> assignment</td><td>Same as <code>CALCWRITE</code>, because assignment delegates to <code>CALCWRITE</code>.</td><td>Same as <code>CALCWRITE</code>, because assignment delegates to <code>CALCWRITE</code>.</td></tr><tr><td><code>MULTIREP</code> / <code>REPLACE_MULTI</code></td><td>Current implementation does <strong>not</strong> route through table buffering.</td><td>Direct-write only: validates all assignments, locks one record, performs one physical write, then applies before/after index snapshot maintenance.</td></tr></tbody></table>
<p>Current <code>REPLACE_MULTI</code> contract says buffering/<code>COMMIT</code>/<code>ROLLBACK</code> integration is deferred. Documentation must not imply that <code>MULTIREP</code> participates in table-buffer dirty state today.</p>
<h2 id="dirty-and-stale-state"><a class="no-underline" href="#dirty-and-stale-state">Dirty and stale state</a></h2>
<p><code>dirty</code> and <code>stale</code> are not synonyms.</p>





















<table><thead><tr><th>State</th><th>Meaning in this lane</th></tr></thead><tbody><tr><td><code>DIRTY</code></td><td>The table buffer contains pending changes that have not been committed. <code>REPLACE</code> and <code>CALCWRITE</code> set dirty when table buffering is ON.</td></tr><tr><td><code>STALE</code></td><td>A field or area needs attention because current visible/index/cache state may need refresh or rebuild. <code>REPLACE</code> and <code>CALCWRITE</code> mark changed fields stale in buffered mode. Direct writes mark stale when the visible value changes or when index maintenance fails, depending on the command path.</td></tr><tr><td><code>fresh</code> / <code>clean</code></td><td>Reported state after buffer clearing, successful rebuild/refresh paths, or explicit table-buffer state commands. These labels should be verified by runtime output before being used as proof.</td></tr></tbody></table>
<p>For <code>MULTIREP</code>/<code>REPLACE_MULTI</code>, changed fields are marked <code>STALE</code> only if direct index maintenance fails. It does not mark the table buffer <code>DIRTY</code>.</p>
<h2 id="runtime-evidence-example"><a class="no-underline" href="#runtime-evidence-example">Runtime evidence example</a></h2>
<p>This transcript demonstrates TableTalk/table-buffer behavior for <code>REPLACE</code>:</p>
<pre><code class="language-text">. set table buffer on
TABLE BUFFER: ON (area 8)
. tup
50000000 | Taylor | Derald | 19921225 | X | CSCI | 20250220 | 2.97 | quinn.taylor0@student.mcc.edu
. replace dob with 19931225
REPLACE: invalid date for field.
. replace dob with "19931225"
. table
TABLE BUFFER: occupied areas only
  enabled=1 dirty=1 stale=1
  Area 8: DBF=STUDENTS.DBF | buffer=ON | mode=RAM | DIRTY | STALE [DOB] | rows: 1 changes (1 recnos)
</code></pre>
<p>Observed behavior:</p>
<ul>
<li><code>SET TABLE BUFFER ON</code> enables buffering for the current area.</li>
<li>Unquoted numeric <code>19931225</code> fails date validation for <code>DOB</code>.</li>
<li>Quoted <code>"19931225"</code> is accepted by the <code>REPLACE</code> path.</li>
<li><code>TABLE</code> reports the current area as <code>buffer=ON</code>, <code>DIRTY</code>, and <code>STALE [DOB]</code>.</li>
<li>The buffered edit is recorded as one pending row change, not immediately promoted as clean storage.</li>
</ul>
<h2 id="type-and-memo-handling"><a class="no-underline" href="#type-and-memo-handling">Type and memo handling</a></h2>
<p>The mutator commands validate and normalize before storage:</p>
<ul>
<li>Date fields accept canonical <code>YYYYMMDD</code> and supported date forms such as <code>TODAY</code>.</li>
<li>Logical fields normalize accepted true/false spellings into DBF logical storage.</li>
<li>Numeric and float fields enforce field width and decimals.</li>
<li>Int32, double, and currency fields have strict parsing paths.</li>
<li>Currency pair fields run through the currency helper validation/normalization path.</li>
<li>x64 memo fields write memo payloads through the memo backend and store memo reference/token text in the DBF field.</li>
</ul>
<p>This is why these commands must be documented as storage mutators, not just expression or display commands.</p>
<h2 id="index-maintenance"><a class="no-underline" href="#index-maintenance">Index maintenance</a></h2>





















<table><thead><tr><th>Command</th><th>Index behavior</th></tr></thead><tbody><tr><td><code>REPLACE</code></td><td>Direct mode uses <code>DbArea::replaceFieldStored</code>, the engine mutation funnel for record write and active index replace snapshots.</td></tr><tr><td><code>CALCWRITE</code></td><td>Direct mode also uses <code>DbArea::replaceFieldStored</code> so expression writes do not bypass active index maintenance.</td></tr><tr><td><code>MULTIREP</code> / <code>REPLACE_MULTI</code></td><td>Captures active tag keys before the multi-field edit, performs one DBF write, captures after state, and applies the replace snapshot once. If that maintenance fails, changed fields are marked stale.</td></tr></tbody></table>
<p>Documentation should keep single-field and multi-field mutation paths separate until the source paths converge.</p>
<h2 id="required-proof-before-promotion"><a class="no-underline" href="#required-proof-before-promotion">Required proof before promotion</a></h2>
<p>Before calling a mutator behavior “complete” in the manual or product pages, capture proof for:</p>
<ul>
<li><code>TABLE BUFFER ON</code>, <code>REPLACE</code>, <code>TABLE BUFFER STATUS</code>, <code>COMMIT</code>, and <code>ROLLBACK</code>.</li>
<li><code>TABLE BUFFER ON</code>, <code>CALCWRITE</code>, <code>TABLE BUFFER STATUS</code>, <code>COMMIT</code>, and <code>ROLLBACK</code>.</li>
<li><code>CALC</code> expression-only output versus <code>CALC &#x3C;field> = &#x3C;expr></code> mutation delegation.</li>
<li><code>TABLE BUFFER OFF</code> direct writes through <code>REPLACE</code> and <code>CALCWRITE</code>, including indexed-field updates.</li>
<li><code>MULTIREP</code>/<code>REPLACE_MULTI</code> direct write, one-lock/one-write behavior, validation failure before physical write, and stale marking on index-maintenance failure.</li>
<li>Memo-field write and clear behavior for x64 memo fields.</li>
</ul>
