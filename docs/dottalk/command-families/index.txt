18:T201e,<p>This page groups command families that are easy to miss in the alphabetical catalog. The catalog remains the row-level reference; this page records the documentation lanes.</p>
<h2 id="tables-lists-and-browsers"><a class="no-underline" href="#tables-lists-and-browsers">Tables, Lists, and Browsers</a></h2>
<p>Table inspection and browsing are a family, not one command:</p>













































<table><thead><tr><th>Surface</th><th>Current role</th></tr></thead><tbody><tr><td><code>SMARTLIST</code> / <code>SL</code></td><td>Filter-aware, order-aware table listing with projections, tuple output, deleted-record modes, and predicate filtering.</td></tr><tr><td><code>SMARTBROWSE</code> / <code>SM</code> / <code>SMART</code></td><td>Smart browser surface over table/tuple/workspace data.</td></tr><tr><td><code>SIMPLEBROWSE</code> / <code>SB</code></td><td>Simple browser surface; current shortcut resolver maps <code>SB</code> to <code>SIMPLEBROWSE</code>.</td></tr><tr><td><code>BROWSE</code></td><td>Legacy/global browse entrypoint forwarded to the refactored browse module.</td></tr><tr><td><code>BROWSER</code></td><td>Minimal interactive browser with list, display, goto, edit, help, and quit actions.</td></tr><tr><td><code>BROWSETUI</code> / <code>BT</code></td><td>Text-mode browser UI that delegates table operations to command handlers.</td></tr><tr><td><code>BROWSETV</code></td><td>Turbo Vision browser lane where enabled.</td></tr><tr><td><code>RBROWSE</code> / <code>ERSATZ</code></td><td>Relation-aware browsing.</td></tr><tr><td><code>TABLE</code>, <code>TABLE_BUFFER</code>, <code>TABLEMETA</code></td><td>Table buffering, stale/dirty state, and metadata diagnostics.</td></tr></tbody></table>
<p>Use <code>SMARTBROWSE</code>, <code>SM</code>, <code>SMART</code>, <code>SIMPLEBROWSE</code>, <code>SB</code>, and <code>SMARTLIST</code> for the current smart/list/browser surfaces.</p>
<h2 id="data-mutators"><a class="no-underline" href="#data-mutators">Data Mutators</a></h2>
<p><code>REPLACE</code>, <code>CALC</code>, <code>CALCWRITE</code>, and <code>MULTIREP</code>/<code>REPLACE_MULTI</code> are the mutation-sensitive command family. They deserve separate treatment from ordinary scalar functions and report commands.</p>

























<table><thead><tr><th>Command</th><th>Documentation boundary</th></tr></thead><tbody><tr><td><code>CALC</code></td><td>Read-only when evaluating an expression; mutating only when assignment targets a real field and delegates to <code>CALCWRITE</code>.</td></tr><tr><td><code>CALCWRITE</code></td><td>Expression-backed single-field mutation; honors TABLE buffering when enabled and direct write/index maintenance when disabled.</td></tr><tr><td><code>REPLACE</code></td><td>Single-field mutation; honors TABLE buffering when enabled and direct write/index maintenance when disabled.</td></tr><tr><td><code>MULTIREP</code> / <code>REPLACE_MULTI</code></td><td>Multi-field direct-write path; validates all assignments, uses one record lock and one physical write, and does not currently participate in TABLE buffer dirty state.</td></tr></tbody></table>
<p>See <a href="/docs/dottalk/data-mutators">Data Mutators</a> for buffered/unbuffered behavior, stale/dirty state, memo mutation, scalar evaluation, and index-maintenance boundaries.</p>
<h2 id="ddl-and-schema"><a class="no-underline" href="#ddl-and-schema">DDL and Schema</a></h2>
<p><code>DDL</code> is the schema-oriented command family:</p>
<pre><code class="language-text">DDL FETCH &#x3C;url> TO &#x3C;file>
DDL VALIDATE &#x3C;schema.json> USING &#x3C;validator.json>
DDL CREATE DBF &#x3C;out.dbf> FROM &#x3C;schema.json>
DDL CREATE DBF &#x3C;out.dbf> FROM &#x3C;schema.json> EMIT SIDECARS
DDL CREATE DBF &#x3C;out.dbf> FROM &#x3C;schema.json> SEED CSV &#x3C;path.csv>
DDL CREATE DBF &#x3C;out.dbf> FROM &#x3C;schema.json> SEED BLANK &#x3C;n>
</code></pre>
<p>Current contract notes: <code>FETCH</code> can perform network access and write schema-side files; <code>CREATE DBF</code> writes DBF output and optional sidecars; existing output is refused unless <code>OVERWRITE</code> is supplied; <code>SEED CSV</code> is recognized by the contract but not complete in the current drop-in.</p>
<h2 id="data-dictionary"><a class="no-underline" href="#data-dictionary">Data Dictionary</a></h2>
<p><code>DDICT</code> is the active Data Dictionary inspection surface:</p>
<pre><code class="language-text">DDICT STATUS
DDICT TABLES
DDICT FIELDS &#x3C;table-or-alias>
DDICT TAGS &#x3C;table-or-alias>
DDICT REL &#x3C;object> [IN|OUT|BOTH]
DDICT EVIDENCE &#x3C;object>
</code></pre>
<p>The current source contract is read-only. It preserves DD* aliases such as <code>DDOBJECT</code>, <code>DDATTR</code>, <code>DDEDGE</code>, and <code>DDEVID</code>, while resolving authoritative <code>DATA_DICTIONARY_*</code> names.</p>
<h2 id="external-app-and-network-boundaries"><a class="no-underline" href="#external-app-and-network-boundaries">External App and Network Boundaries</a></h2>
<p>These commands cross the runtime boundary and need explicit risk notes:</p>

































<table><thead><tr><th>Command</th><th>Boundary</th></tr></thead><tbody><tr><td><code>EDIT</code> / <code>TEXT</code></td><td>Launch or use configured editor behavior.</td></tr><tr><td><code>IMAGE</code></td><td>Reads image metadata; <code>IMAGE &#x3C;file></code> opens the OS image viewer on Windows.</td></tr><tr><td><code>WEB</code></td><td>Opens URLs, performs HTTP <code>GET</code>/<code>HEAD</code>, or writes fetched content to a file.</td></tr><tr><td><code>SFTP</code></td><td>Remote file-transfer lane.</td></tr><tr><td><code>PSHELL</code>, <code>!</code>, <code>BANG</code></td><td>Host shell / external command lane.</td></tr><tr><td><code>ZIP</code></td><td>Archive read/write/extract lane through the configured backend.</td></tr></tbody></table>
<p>Documentation must separate external-process, network, viewer/browser launch, and filesystem-write risks from database mutation.</p>
<h2 id="selfdoc-help-maintenance-and-manuals"><a class="no-underline" href="#selfdoc-help-maintenance-and-manuals">SelfDoc, Help, Maintenance, and Manuals</a></h2>









































<table><thead><tr><th>Command</th><th>Role</th></tr></thead><tbody><tr><td><code>HELP</code></td><td>Routes user help across command, function, FoxPro, PowerShell, SQL, beta, predicate, and education surfaces.</td></tr><tr><td><code>CMDHELP</code></td><td>Builds or reports HELP DATA catalogs; <code>CMDHELP BUILD</code> mutates HELP DATA DBFs.</td></tr><tr><td><code>CMDHELPCHK</code></td><td>Validates HELP/catalog/reflection artifacts; default behavior is report-only.</td></tr><tr><td><code>MAINT</code></td><td>Read-only maintenance inspector for lanes, cookbooks, contracts, boundaries, and docs.</td></tr><tr><td><code>BBOX</code></td><td>Teaches the Blackbox model and contract lane relationships.</td></tr><tr><td><code>MANUAL</code></td><td>Read-only inspector for accepted MAN* manualgen catalog evidence.</td></tr><tr><td><code>MANSTAR</code></td><td>Manualgen/publication support lane.</td></tr><tr><td><code>DRAWIO</code></td><td>Diagram generation/export lane.</td></tr></tbody></table>
<p>The repeatable curation path is:</p>
<pre><code class="language-text">source behavior
-> @dottalk.usage / @dottalk.contract / source comments
-> comments and SRC* evidence
-> HELP DATA and CMDHELP
-> CMDHELPCHK validation
-> SelfDoc reports and manualgen artifacts
-> reviewed manual section
-> website derivative
</code></pre>
