17:T1887,<p>DotScript is the repeatable script form of the DotTalk++ command language. It is for command files, labs, smoke runs, validation passes, CSV/DBF import-export exercises, and automation that should be readable by students and maintainers.</p>
<h2 id="script-shape"><a class="no-underline" href="#script-shape">Script shape</a></h2>
<p>A DotScript file is a sequence of DotTalk++ commands:</p>
<pre><code class="language-text">USE students
STRUCT
LIST
</code></pre>
<p>Scripts should be written as documentation-quality artifacts: clear names, small sections, explicit assumptions, and proof-friendly output.</p>
<h2 id="variables"><a class="no-underline" href="#variables">Variables</a></h2>
<p>Variables carry values between commands and expressions. The exact runtime syntax should track the local parser, but the language guide reserves this product surface for:</p>
<pre><code class="language-text">SET VAR tableName = "students"
SET VAR maxRows = 25
</code></pre>
<p>Variables are expected to support command substitution and repeatable scripts without hard-coding every value in every line.</p>
<h2 id="dotscript-nesting"><a class="no-underline" href="#dotscript-nesting">DOTSCRIPT nesting</a></h2>
<p>The current runner supports a main script plus one subscript. Deeper nesting is intentionally blocked by the runtime and should not be shown in public examples as available behavior.</p>
<h2 id="comments"><a class="no-underline" href="#comments">Comments</a></h2>
<p>Comments are part of the language, not an afterthought. They let scripts explain the lab, database assumption, or canary being run.</p>
<pre><code class="language-text">* Open the current lab table.
USE students

// Inspect structure before mutation.
STRUCT
</code></pre>
<p>Comment forms should be accepted consistently by the script runner before they are used in published examples.</p>
<h2 id="line-continuation"><a class="no-underline" href="#line-continuation">Line continuation</a></h2>
<p>Long commands need a documented continuation form so scripts can stay readable.</p>
<pre><code class="language-text">INDEX ON last_name + first_name ;
  TAG student_name
</code></pre>
<p>The continuation marker should be treated as part of the canonical grammar once the parser behavior is locked.</p>
<h2 id="if--else--endif"><a class="no-underline" href="#if--else--endif">IF / ELSE / ENDIF</a></h2>
<p><code>IF</code> / <code>ELSE</code> / <code>ENDIF</code> uses the shell's shared boolean expression evaluator and control-flow stack.</p>
<pre><code class="language-text">IF recno() = 1
  ? "first record"
ELSE
  ? "not first"
ENDIF
</code></pre>
<h2 id="loop--endloop"><a class="no-underline" href="#loop--endloop">LOOP / ENDLOOP</a></h2>
<p><code>LOOP</code> / <code>ENDLOOP</code> is the general repetition form. The command handler supports numeric repetition forms such as <code>LOOP 5</code>, <code>LOOP 5 TIMES</code>, and <code>LOOP FOR &#x3C;n></code>.</p>
<pre><code class="language-text">SET VAR n = 0
LOOP 10
  ? n
  SET VAR n = n + 1
ENDLOOP
</code></pre>
<p>Required documentation topics:</p>
<ul>
<li>loop condition evaluation.</li>
<li>loop variable mutation.</li>
<li>numeric repetition labels and quiet mode.</li>
<li>error handling inside a loop.</li>
</ul>
<h2 id="while--endwhile"><a class="no-underline" href="#while--endwhile">WHILE / ENDWHILE</a></h2>
<p><code>WHILE</code> / <code>ENDWHILE</code> buffers and executes a loop while a boolean expression remains true.</p>
<pre><code class="language-text">WHILE recno() &#x3C; 10
  ? recno()
  SKIP
ENDWHILE
</code></pre>
<h2 id="until--enduntil"><a class="no-underline" href="#until--enduntil">UNTIL / ENDUNTIL</a></h2>
<p><code>UNTIL</code> / <code>ENDUNTIL</code> buffers and executes a loop until a boolean expression becomes true.</p>
<pre><code class="language-text">UNTIL eof()
  ? recno()
  SKIP
ENDUNTIL
</code></pre>
<h2 id="scan--endscan"><a class="no-underline" href="#scan--endscan">SCAN / ENDSCAN</a></h2>
<p>Scan-style loops are table-oriented. They should make record traversal readable and close to the xBase lineage.</p>
<pre><code class="language-text">USE students
SCAN
  ? id, last_name, first_name
ENDSCAN
</code></pre>
<p>Required documentation topics:</p>
<ul>
<li>current work area selection.</li>
<li>filter and scope interaction.</li>
<li>deleted-record handling.</li>
<li>index order interaction.</li>
<li>current runtime limitation: only one <code>SCAN</code> block is buffered at a time; nested <code>SCAN</code> during <code>ENDSCAN</code> is not currently supported.</li>
</ul>
<h2 id="csv-and-dbf-importexport"><a class="no-underline" href="#csv-and-dbf-importexport">CSV and DBF import/export</a></h2>
<p>DotScript examples can drive the current CSV import/export commands:</p>
<pre><code class="language-text">USE students
IMPORT students.csv
EXPORT TO students_out.csv CSV
</code></pre>
<p><code>IMPORT</code> reads CSV into the current open DBF by matching CSV headers to field names case-insensitively. <code>EXPORT</code> writes the current table, or a named open work area, to CSV by default.</p>
<h2 id="workspaces"><a class="no-underline" href="#workspaces">Workspaces</a></h2>
<p>DotScript must document workspace behavior explicitly. A workspace is a wrapper over active areas. Each area is a <code>DbArea</code>-style runtime object with table state, cursor state, indexes, relations, and metadata. Scripted workspace save/load should therefore be documented as object state management, not just a file convenience.</p>
<h2 id="memos-and-custom-field-types"><a class="no-underline" href="#memos-and-custom-field-types">Memos and custom field types</a></h2>
<p>The memo system is object-oriented and should be scriptable as a first-class data surface. In addition to student code hooks, x64base should document custom field type hooks so laboratories and downstream users can add field semantics without forking the core engine.</p>
<h2 id="proof-labels"><a class="no-underline" href="#proof-labels">Proof labels</a></h2>
<p>Every published DotScript example should be labeled as one of:</p>
<ul>
<li>runtime-proven.</li>
<li>active beta.</li>
<li>canary.</li>
<li>planned.</li>
</ul>
