# AI Friendly Lane Manifest v1

Status: seed manifest.
Scope: DotTalk++ / LabTalk AI-assisted work capture, curation, and promotion.

## Definition

AI Friendly is the lane that captures the best human/AI collaboration and turns
it into durable project assets without duplicating work already covered by
contracts, SelfDoc, MDO/manualgen, HELP, CMDHELPCHK, proof reports, source
comments, or LabTalk registries.

## Doctrine

```text
Source defines.
Runtime proves.
HELP explains.
Metadata organizes.
CMDHELPCHK validates.
SelfDoc preserves provenance.
Manualgen assembles reviewed manuals.
Contracts preserve durable decisions.
AI Friendly captures, distills, routes, and improves collaboration.
```

## Why This Lane Exists

DotTalk++ and LabTalk have used AI-assisted development, review, documentation,
and planning since 2025. A large amount of useful material exists in retained
interactions, notes, generated reports, plans, and partial documentation.

The project benefits when the best of that material is curated into stable
assets and the rest is clearly left as raw source material, scratch, or archive.

AI capability is expanding faster than any single DotTalk++ subsystem or
LabTalk campus structure. This lane must therefore stay flexible and act as a
routing layer rather than a rigid product module.

## Non-Goals

- Do not create a second contract system.
- Do not create a second SelfDoc or MDO pipeline.
- Do not preserve raw chat as project authority.
- Do not promote AI claims without source, proof, or honest design-intent status.
- Do not repeat existing reports, registries, source comments, HELP rows, or
  manualgen artifacts.

## Lane Inputs

- retained AI conversations
- project notes created from AI sessions
- agent bootstrap and handoff documents
- proposed contracts and source-comment candidates
- review summaries and repair sessions
- LabTalk lab, lesson, and case candidates
- proof/report summaries
- drift findings and recovery patterns
- reusable prompts, checklists, and working patterns

## Lane Outputs

| Output | Destination |
| --- | --- |
| durable rule | `docs/contracts` and contract registry |
| command behavior or usage rule | source `@dottalk.usage`, HELP, CMDHELP, CMDHELPCHK |
| source-documentation improvement | SelfDoc/comment lane |
| reviewed publication material | MDO/manualgen |
| teaching sequence | LabTalk lab/case/lesson registry |
| runtime claim | proof transcript or test/report |
| AI operating pattern | `docs/agents` or this lane |
| discarded or stale item | archive/rejected status with reason |

## Evidence Classes

Use the existing project vocabulary wherever possible:

- chat-intended
- draft
- design-intended
- source-defined
- runtime-proven
- HELP-documented
- metadata-staged
- publication-ready
- student-ready
- superseded
- rejected

If an item does not fit one of these classes, classify it as `review-needed`
instead of inventing a new authority state.

## Operating Principle

Before adding a new artifact, ask:

```text
Can this be handled by an existing contract, source comment, HELP surface,
SelfDoc report, MDO/manualgen artifact, LabTalk registry, or proof transcript?
```

If yes, route it there. If no, add the smallest AI Friendly artifact needed to
preserve and classify it.
