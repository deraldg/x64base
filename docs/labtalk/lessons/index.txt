17:T12e2,<p>The Laboratory Campus lesson platform is the place where LabTalk turns live
system work, source evidence, cases, datasets, and career experience into
organized lessons.</p>
<p>It is intentionally registry-first. Local lesson records begin in:</p>
<pre><code class="language-text">labtalk/registries/lessons.yaml
</code></pre>
<p>Lesson drafts live under:</p>
<pre><code class="language-text">labtalk/lessons/student
labtalk/lessons/career
</code></pre>
<h2 id="lesson-rule"><a class="no-underline" href="#lesson-rule">Lesson Rule</a></h2>
<p>Campus lessons should preserve the proof-aware path:</p>
<pre><code class="language-text">concept -> app -> dataset -> command -> proof -> case -> lesson
</code></pre>
<p>That means a lesson should not be detached prose. It should say what concept it
teaches, which app or lab exposes it, which dataset or artifact is involved,
which command or workflow the learner can inspect, what proof exists, and what
review gate remains.</p>
<h2 id="lesson-lanes"><a class="no-underline" href="#lesson-lanes">Lesson Lanes</a></h2>






























<table><thead><tr><th>Lane</th><th>Purpose</th><th>Current status</th></tr></thead><tbody><tr><td>Student lessons</td><td>Learner-facing or learner-candidate lessons with setup, steps, observations, questions, and proof links.</td><td>Draft and proof-linked seed lessons exist.</td></tr><tr><td>Career lessons</td><td>Lessons learned while building x64base, DotTalk++, LabTalk, proof systems, documentation systems, and legacy-system recovery paths.</td><td>Draft shelf created.</td></tr><tr><td>Instructor review</td><td>Notes for setup safety, expected observations, assessment, and classroom use.</td><td>Planned as lessons mature.</td></tr><tr><td>External suggestions</td><td>Ideas and review notes from educators, technical writers, database practitioners, students, and collaborators.</td><td>Open through the suggestion path.</td></tr></tbody></table>
<h2 id="seed-lesson-records"><a class="no-underline" href="#seed-lesson-records">Seed Lesson Records</a></h2>









































<table><thead><tr><th>Lesson</th><th>Lane</th><th>Proof state</th><th>Next gate</th></tr></thead><tbody><tr><td>Records, Fields, and Tables</td><td>Student</td><td><code>runtime_observed</code></td><td>Add reviewed worksheet questions and HELP/CMDHELP readback.</td></tr><tr><td>Comments to Contracts to HELP</td><td>Student</td><td><code>validated</code></td><td>Convert proof readback into student questions and instructor notes.</td></tr><tr><td>Data Systems History Trail</td><td>Student</td><td><code>case_registered</code></td><td>Use the storyboard PPTX as local source material while separating live runtime demos from reviewed history and simulations.</td></tr><tr><td>Proof-First Development Lessons</td><td>Career</td><td><code>source_defined</code></td><td>Add dated build notes and one before/after example.</td></tr><tr><td>Turning Legacy Systems into Learning Labs</td><td>Career</td><td><code>case_registered</code></td><td>Add source-memory boundaries and reviewed public wording.</td></tr></tbody></table>
<p>The LabTalk / DotTalk++ systems storyboard deck already fits this lesson model
in two ways: as the source artifact for the
<a href="/docs/labtalk/database-evolution">Database Evolution Path</a>, and as a
resume-style career artifact for the <a href="/docs/labtalk/career-lessons">Career Lessons</a>
lane. The hosted site summarizes the deck and keeps the large PPTX in the local
artifact set instead of bundling it into the public website source.</p>
