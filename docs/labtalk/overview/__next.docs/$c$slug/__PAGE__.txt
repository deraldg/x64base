7:T1865,<p>The <strong>Laboratory Campus</strong> is the learning-lab and collaboration framing for the
x64base / DotTalk++ ecosystem. LabTalk® remains the project mark and legacy name,
but the campus is the public frame to emphasize.</p>
<p>It is not just a single app. It is an alpha hybrid of planning material,
implemented pieces, in-development labs, datasets, runtime demonstrations, help
systems, GUI experiments, and proof artifacts that make computing systems easier
to study and easier to collaborate on.</p>

<h2 id="cases-and-storyboard"><a class="no-underline" href="#cases-and-storyboard">Cases and Storyboard</a></h2>
<p>The campus case-study material now has a visible entry point:
<a href="/docs/labtalk/cases-storyboard">Cases and Storyboard</a>. That page surfaces the
LabTalk / DotTalk++ systems storyboard deck, case families, and the proof-aware
path from source memory to reviewed case notes, datasets or simulations,
runtime/UI demonstrations, proof artifacts, and lessons.</p>
<h2 id="laboratory-campus"><a class="no-underline" href="#laboratory-campus">Laboratory Campus</a></h2>
<p>Think of it as a campus:</p>
<ul>
<li><strong>Runtime Systems Lab</strong> - DotTalk++ commands, scripts, work areas, tables, records, indexes, and relations.</li>
<li><strong>Historical Data Systems Lab</strong> - punch cards, COBOL, CODASYL, xBase, SQL, ERP, cloud computing, and AI-era data literacy.</li>
<li><strong>Self-Documenting Systems Lab</strong> - comments, contracts, HELP, CMDHELP, CMDHELPCHK, metadata, and proof readback.</li>
<li><strong>Dataset Library</strong> - small inspectable datasets for lessons and repeatable labs.</li>
<li><strong>Case Library</strong> - engineering and historical cases that connect live demonstrations to real system stories.</li>
<li><strong>GUI, pycrud, and Portal Lab</strong> - local launchers, CRUD/database literacy tools, dashboards, and front-end experiments that expose the same underlying evidence.</li>
</ul>
<p>The first public history path is the <a href="/docs/labtalk/database-evolution">Database Evolution Path</a>,
which integrates the LabTalk / DotTalk++ storyboard deck as a teaching artifact
from punch cards and COBOL through xBase, enterprise data, cloud computing, and AI.</p>
<p>The <a href="/docs/labtalk/education-features">Education Features</a> page now exposes
source-defined teaching surfaces such as <code>IDX</code> timed index/sort labs, student
command/function hooks, <code>SORT</code> lab behavior, and pre/post polling hooks.</p>
<p>The <a href="/docs/labtalk/runtime-evidence">Runtime Evidence Gallery</a> records reviewed screenshots from Arctic, the local campus portal, workspace loading, table browsing, tuple output, and GUI/TUI readback for use by the manual and website.</p>
<p>The <a href="/docs/labtalk/lessons">Lesson Platform</a> now separates student lesson seeds
from career lessons learned while building the system. The site is also open to
structured input through <a href="/docs/labtalk/suggest-a-lesson">Suggest a Lesson</a>.</p>

<h2 id="repository-boundary"><a class="no-underline" href="#repository-boundary">Repository Boundary</a></h2>
<p>The Laboratory Campus is a <strong>consumer layer</strong>, not a replacement authority for
the engine.</p>
<ul>
<li><code>x64base</code> remains the owner of engine/runtime truth, including command
semantics, cursor state, relations, indexing, and the Arctic GUI/TUI
workbench surfaces.</li>
<li><code>DotTalk++</code> remains the best runtime/manual identity for the command shell and
teaching shell surfaces.</li>
<li><code>LabTalk</code> owns the portal, registries, labs, proofs, assignments, and campus
overlays that consume runtime truth.</li>
</ul>
<p>That means the campus may launch, explain, and sequence runtime behavior, but
it should not silently redefine the behavior that belongs to the runtime.</p>
<h2 id="core-pattern"><a class="no-underline" href="#core-pattern">Core Pattern</a></h2>
<p>Campus material should follow a proof-aware path:</p>
<p><a href="/docs/labtalk/cases-storyboard"><img src="/images/labtalk/laboratory-campus-proof-path-v2.svg" alt="Laboratory Campus proof-aware learning path"></a></p>
<pre><code class="language-text">concept -> app -> dataset -> command -> proof -> case -> lesson
</code></pre>
<p>This keeps lessons connected to live or reviewed evidence instead of becoming
detached prose.</p>
<h2 id="current-status"><a class="no-underline" href="#current-status">Current Status</a></h2>
<p>The Laboratory Campus is an active alpha direction. Public x64base pages should
describe it as a campus model, collaboration area, and learning-lab program, not
as a finished packaged distribution unless that packaging has been explicitly
released.</p>
<h2 id="academic-and-research-direction"><a class="no-underline" href="#academic-and-research-direction">Academic and Research Direction</a></h2>
<p>The <a href="/docs/labtalk/academic-positioning">Academic Positioning and Research Agenda</a>
is supporting Laboratory Campus material, not the campus hero or primary entry
point. It frames candidate work across database education, learning sciences,
HCI, end-user programming, literate computing, explainable systems, and
computing education research while separating implemented system evidence from
educational hypotheses that still require literature review and empirical
study.</p>
