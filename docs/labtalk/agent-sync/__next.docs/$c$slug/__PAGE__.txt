6:T2b38,<h2 id="purpose"><a class="no-underline" href="#purpose">Purpose</a></h2>
<p>This page is the <strong>doc-only live portal</strong>: a compact, dated snapshot of the current
governance surface for AI development partners who work through the browser and cannot
read the maintainer's local drive. The authoritative source of truth is the maintainer's
private development tree (not published); the GitHub <code>AI_PORTAL.md</code> only moves on full
engine snapshots. <strong>This page updates on the website's own cadence, so it is the
freshest public state an outside agent can reach.</strong></p>
<p>If you are an external agent (ChatGPT or similar): read this page first, treat it as
authoritative over any older GitHub portal state, and do your work as standalone,
compile-only packets that hand back to the maintainer's in-repo agent for integration.
<strong>To reply, use the Pseudo-Chat return lane below</strong> — the asynchronous (not real-time)
channel by which your answers are logged back onto this page.</p>
<blockquote>
<p><strong>Freshness:</strong> 2026-07-21a. When in doubt, the maintainer's reconciliation against the
private development tree wins. Do not treat this page as autonomous authority or as
permission to bypass a proof gate.</p>
<p><em>Changed since 2026-07-20c:</em> added the <strong>Pseudo-Chat return lane</strong> — this page is now a
two-way (asynchronous, closeout-cadence) channel: external-partner replies are logged
here and close the Open questions, which are now a tracked table. A fuller state refresh
(BETA-1 stabilization + regression net, the script-header contract lane) is queued for
the next closeout.</p>
</blockquote>
<h2 id="working-agreement"><a class="no-underline" href="#working-agreement">Working agreement</a></h2>
<ul>
<li><strong>No GitHub pushes, no staging-repo writes, no commits</strong> by outside agents — dev-only
on the current branch. Authoritative integration into the maintainer's development
tree is the in-repo agent's step.</li>
<li><strong>Ship standalone, compile-only packets</strong> with a passing test, an explicit
provisional-decisions list, and a hard stop before engine integration.</li>
<li><strong>Prove before you claim.</strong> Status is PLANNED → PARTIAL → SUPPORTED only on runtime
proof. Runtime proves, source defines, HELP explains.</li>
</ul>
<h2 id="doctrine-to-honor"><a class="no-underline" href="#doctrine-to-honor">Doctrine to honor</a></h2>
<ul>
<li><strong>Character cross-compatibility.</strong> Programs interoperate as characters. The wire is
chars; in-memory <code>double</code> / <code>int64</code> / <code>ExactDecimal</code> are projections. No private binary
interchange; exact numerics carry a <code>canonical_text()</code>.</li>
<li><strong>Errors derive from messaging (AIF-036).</strong> Parse/value errors become message-catalog
<code>MessageId</code> + severity so <code>stop_on_error</code> governs them — not free-form strings at
contract time.</li>
<li><strong>Representative by design + Rule of Three (AIF-037).</strong> One central API per concern,
one checked helper; a third copy becomes a function. Reuse the field-codec registry
(AIF-030); do not fork parallel tables.</li>
<li><strong>HELP/SelfDoc metadata (AIF-025)</strong> on every public command/function.</li>
<li><strong>Evidence-gated lanes:</strong> lanes → milestones → proof gates → closeouts; prove one
vertical before freezing a contract.</li>
</ul>
<h2 id="canonical-value--signed-off-2026-07-20"><a class="no-underline" href="#canonical-value--signed-off-2026-07-20">Canonical Value — SIGNED OFF (2026-07-20)</a></h2>
<p>The canonical-value decision that gates the tuple contract is <strong>accepted</strong>:</p>
<ul>
<li><strong>Canonical domain value = DotTalk Value (DTV), <code>dottalk::value::Value</code></strong> — exact,
six-state, char-canonical, and the single superset.</li>
<li><strong>Two-tier model:</strong> DTV is the domain / storage / tuple / interchange truth; the
engine's <code>double</code>-based value (<code>xexpr::Value</code> / <code>EvalValue</code>) is DTV's <em>evaluation
projection</em> — fast and deliberately lossy, not a rival. No fourth value type.</li>
<li><strong>Accepted amendments:</strong> (A) DTV carries both <code>Integer</code> (int64) and <code>UnsignedInteger</code>
(uint64) — RECNO64 is unsigned, so values up to <code>18446744073709551615</code> never pass
through <code>double</code>; (B) the canonical wire is tagged + versioned (<code>DTV1|state|kind| payload</code>, fully reversible); (C) comparison is semantic (kind/state-aware), not lexical
on the wire form; (D) the <code>type_code → ValueKind</code> map lives in the field/type registry,
not inside <code>Value</code>; (E) array Nil → DTV <code>Null</code> (DBF blanks stay <code>Blank</code>); (F) adapters
return a <code>ProjectionResult</code> with an explicit <code>ProjectionLoss</code> — no invisible exact→double.</li>
<li><strong>Rules:</strong> convert only at named seams, through characters where fidelity matters;
never reconstruct domain truth from an evaluation value.</li>
<li><strong>Phase-0 COMPLETE (8/8):</strong> the DTV foundation is integrated into the authoritative
tree as an isolated <code>dottalk_value</code> lib (real <code>MemoRef</code>) and the <strong>MSVC build is
green</strong>. The tuple lane may now open; the TupleCell contract is frozen only after the
wire + comparison + adapter proofs below.</li>
</ul>
