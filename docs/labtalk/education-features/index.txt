<h2 id="timed-sort-and-index-labs"><a class="no-underline" href="#timed-sort-and-index-labs">Timed Sort and Index Labs</a></h2>
<p>The strongest current timing surface is <code>IDX</code>, not plain <code>SORT</code>.</p>
<p><code>IDX</code> is explicitly described in source as a memory-only educational index lab. It accepts commands such as:</p>
<pre><code class="language-text">IDX ON LNAME TAG lname_std
IDX ON LNAME TAG lname_bubble BUBBLE
IDX ON LNAME TAG lname_bubble2 SORT BUBBLE DESC
IDX LIST
IDX DROP &#x3C;tag>
</code></pre>
<p>Its output path records:</p>
<ul>
<li>records scanned and indexed,</li>
<li>deleted records skipped,</li>
<li>total build elapsed time,</li>
<li>sort elapsed time,</li>
<li>comparisons,</li>
<li>swaps.</li>
</ul>
<p>That makes it suitable for a campus algorithm lab where students compare <code>STD</code> versus <code>BUBBLE</code> against the same table.</p>
<p>Plain <code>SORT</code> is also lab-ready, but it currently reports operation counts rather than elapsed time. It is useful for teaching DBF output creation, key expressions, <code>FOR</code>/<code>WHILE</code> predicates, field projection, deleted-record handling, and <code>UNIQUE</code> suppression.</p>
<h2 id="reference-retro-and-blackbox-commands"><a class="no-underline" href="#reference-retro-and-blackbox-commands">Reference, Retro, and Blackbox Commands</a></h2>
<p>Current education-visible commands include:</p>





































<table><thead><tr><th>Command</th><th>Campus use</th></tr></thead><tbody><tr><td><code>ASCII</code></td><td>Teach character codes, control characters, extended ASCII, ranges, and table rendering.</td></tr><tr><td><code>RETRO</code></td><td>Show ASCII-safe retro system screens for lineage and classroom context.</td></tr><tr><td><code>BBOX</code></td><td>Teach the Blackbox model: data enters a processing system and information comes out.</td></tr><tr><td><code>BOOLEAN</code></td><td>Teach predicate evaluation through xexpr.</td></tr><tr><td><code>CASE</code></td><td>Browse education case-study catalog entries.</td></tr><tr><td><code>CHRISTMAS</code></td><td>Demonstrate controlled colored ASCII output.</td></tr><tr><td><code>COBOL</code>, <code>CODASYL</code>, <code>NORMALIZE</code>, <code>SIX</code></td><td>Tie xBase learning to older database/toolchain concepts and normalization/index labs.</td></tr></tbody></table>
<p>These are campus curriculum and proof surfaces that should connect to command catalog rows, transcripts, and SelfDoc reports.</p>
<h2 id="student-code-hooks"><a class="no-underline" href="#student-code-hooks">Student Code Hooks</a></h2>
<p>The extension policy separates core and learner code:</p>
<pre><code class="language-text">built-in commands -> central registry
student/custom commands -> src/ext/cmd self-registration
function families -> function catalog or self-registration pattern
</code></pre>
<p>Current examples:</p>

























<table><thead><tr><th>Hook</th><th>Registered Surface</th><th>Lesson</th></tr></thead><tbody><tr><td><code>cmd_student_hello.cpp</code></td><td><code>STUDENTHELLO</code>, <code>SHELLO</code></td><td>Add a custom command without touching the protected command registry.</td></tr><tr><td><code>cmd_student_echo.cpp</code></td><td><code>STUDENTECHO</code>, <code>SECHO</code></td><td>Add aliases and command-local usage behavior.</td></tr><tr><td><code>fn_student_text_autoreg.cpp</code></td><td><code>STU_UPPER</code>, <code>STU_REPEAT</code></td><td>Add expression-style functions with argument checks and small evaluators.</td></tr></tbody></table>
<p>This is a strong campus pattern: students can first inspect a command/function in the website catalog, then add a controlled extension, then prove it with HELP/CMDHELP and a runtime transcript.</p>
<h2 id="prepost-polling"><a class="no-underline" href="#prepost-polling">Pre/Post Polling</a></h2>
<p>The current pre/post polling files are small but important:</p>
<pre><code class="language-text">pre_poll()  -> [POLL PRE]
post_poll() -> [POLL POST]
</code></pre>
<p>That gives the campus a visible seam for later labs about command lifecycle observation. The next useful campus slice is:</p>
<pre><code class="language-text">before command -> pre poll -> command execution -> post poll -> transcript -> SelfDoc evidence
</code></pre>
<p>Until a full <code>POLLING</code> command is implemented, the website should describe this as a source-defined hook surface and placeholder integration lane, not a finished user-facing polling subsystem.</p>
<h2 id="vertical-documentation-flow"><a class="no-underline" href="#vertical-documentation-flow">Vertical Documentation Flow</a></h2>
<p>The Laboratory Campus should feed these pages from the existing vertical stack:</p>
<pre><code class="language-text">source comments and @dottalk.usage
-> comments/SRC* evidence
-> HELP DATA and CMDHELP
-> CMDHELPCHK validation
-> SelfDoc/manualgen artifacts
-> website pages and campus lessons
</code></pre>
<p>The initial website implementation uses source-derived pages:</p>
<ul>
<li><a href="/docs/dottalk/command-catalog">Command Catalog</a></li>
<li><a href="/docs/dottalk/function-catalog">Function Catalog</a></li>
</ul>
<p>The next implementation step is to replace manual extraction with a repeatable generator that reads the same evidence tables and reports used by SelfDoc.</p>
