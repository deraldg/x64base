<h2 id="timed-sort-and-index-labs"><a class="no-underline" href="#timed-sort-and-index-labs">Timed Sort and Index Labs</a></h2>
<p>The strongest current timing surface is <code>IDX</code>, not plain <code>SORT</code>.</p>
<p><code>IDX</code> is explicitly described in source as a memory-only educational index lab. It accepts commands such as:</p>
<pre><code class="language-text">IDX ON LNAME TAG lname_std
IDX ON LNAME TAG lname_bubble BUBBLE
IDX ON LNAME TAG lname_bubble2 SORT BUBBLE DESC
IDX LIST
IDX DROP &#x3C;tag>
</code></pre>
<p>Its output path records:</p>
<ul>
<li>records scanned and indexed,</li>
<li>deleted records skipped,</li>
<li>total build elapsed time,</li>
<li>sort elapsed time,</li>
<li>comparisons,</li>
<li>swaps.</li>
</ul>
<p>That makes it suitable for a LabTalk algorithm lab where students compare <code>STD</code> versus <code>BUBBLE</code> against the same table.</p>
<p>Plain <code>SORT</code> is also lab-ready, but it currently reports operation counts rather than elapsed time. It is useful for teaching DBF output creation, key expressions, <code>FOR</code>/<code>WHILE</code> predicates, field projection, deleted-record handling, and <code>UNIQUE</code> suppression.</p>
<h2 id="student-code-hooks"><a class="no-underline" href="#student-code-hooks">Student Code Hooks</a></h2>
<p>The extension policy separates core and learner code:</p>
<pre><code class="language-text">built-in commands -> central registry
student/custom commands -> src/ext/cmd self-registration
function families -> function catalog or self-registration pattern
</code></pre>
<p>Current examples:</p>

























<table><thead><tr><th>Hook</th><th>Registered Surface</th><th>Lesson</th></tr></thead><tbody><tr><td><code>cmd_student_hello.cpp</code></td><td><code>STUDENTHELLO</code>, <code>SHELLO</code></td><td>Add a custom command without touching the protected command registry.</td></tr><tr><td><code>cmd_student_echo.cpp</code></td><td><code>STUDENTECHO</code>, <code>SECHO</code></td><td>Add aliases and command-local usage behavior.</td></tr><tr><td><code>fn_student_text_autoreg.cpp</code></td><td><code>STU_UPPER</code>, <code>STU_REPEAT</code></td><td>Add expression-style functions with argument checks and small evaluators.</td></tr></tbody></table>
<p>This is a strong LabTalk pattern: students can first inspect a command/function in the website catalog, then add a controlled extension, then prove it with HELP/CMDHELP and a runtime transcript.</p>
<h2 id="prepost-polling"><a class="no-underline" href="#prepost-polling">Pre/Post Polling</a></h2>
<p>The current pre/post polling files are small but important:</p>
<pre><code class="language-text">pre_poll()  -> [POLL PRE]
post_poll() -> [POLL POST]
</code></pre>
<p>That gives LabTalk a visible seam for later labs about command lifecycle observation. The next useful campus slice is:</p>
<pre><code class="language-text">before command -> pre poll -> command execution -> post poll -> transcript -> SelfDoc evidence
</code></pre>
<p>Until a full <code>POLLING</code> command is implemented, the website should describe this as a source-defined hook surface and placeholder integration lane, not a finished user-facing polling subsystem.</p>
