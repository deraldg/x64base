<h2 id="teaching-thesis"><a class="no-underline" href="#teaching-thesis">Teaching Thesis</a></h2>
<p>Students who understand data, records, reports, and systems are better prepared
to work with AI. The campus should make that path visible as a living
laboratory, not only as a history lecture.</p>

<h2 id="database-families-map"><a class="no-underline" href="#database-families-map">Database Families Map</a></h2>
<p>The evolution path also needs a high-level map of modern database families. This
is not a replacement for the case-study flow below. It is a taxonomy students
can use while the campus builds evidence-backed cases over time.</p>
<p>Current case coverage is intentionally narrower than the full database
landscape. The public case-study path should only expand when we have reviewed
source material, a dataset or simulation, a command/UI demonstration, and a
proof artifact for that database family.</p>



















































































<table><thead><tr><th>Family</th><th>Structural Idea</th><th>Common Strengths</th><th>Campus Status</th></tr></thead><tbody><tr><td>Relational / RDBMS</td><td>Schemas, tables, rows, columns, keys, SQL, joins, constraints, and transactions.</td><td>Strong consistency, ACID transactions, mature tooling, ERP/finance/HR/inventory workflows.</td><td>Core teaching lane. x64base models table, tuple, order, relation, and business-record thinking without claiming to be a full SQL server.</td></tr><tr><td>Document stores</td><td>Semi-structured JSON/BSON-style documents with flexible shape.</td><td>Nested records, dynamic schema, content/user/event data.</td><td>Context lane. Useful when comparing schema overlays and metadata, but not yet a case-study family.</td></tr><tr><td>Key-value stores</td><td>Associative key to value storage, often optimized for simple lookup.</td><td>Speed, caching, session state, embedded lookup, backend indexing.</td><td>Context lane. LMDB-backed index work gives x64base a key-value-adjacent backend story, but the public case flow should not present a separate key-value case yet.</td></tr><tr><td>Wide-column / column-family stores</td><td>Sparse distributed tables grouped by column family.</td><td>Horizontal scale and high write throughput.</td><td>Future context only. No current public case study.</td></tr><tr><td>Graph databases</td><td>Nodes, edges, and properties optimized for relationship traversal.</td><td>Relationship queries, recommendation, fraud, dependency analysis.</td><td>Conceptual comparison lane. RelTalk and relation browsing can introduce relationship thinking without claiming graph-database implementation.</td></tr><tr><td>Columnar analytical databases</td><td>Column-oriented storage for scan, compression, and aggregate workloads.</td><td>BI, warehousing, dashboards, large aggregations.</td><td>Future context only. x64base orders and filters can support analytical lessons, but not a columnar engine claim.</td></tr><tr><td>Time-series databases</td><td>Timestamped, append-heavy data with retention and range-query patterns.</td><td>Metrics, monitoring, sensors, financial ticks.</td><td>Future context only. No current public case study.</td></tr><tr><td>Search engines</td><td>Inverted indexes for full-text search and relevance ranking.</td><td>Search, faceting, log and document indexing.</td><td>Future context only. Index lessons may mention the family, but search-engine behavior is not a current case.</td></tr><tr><td>Object databases</td><td>Direct persistence of objects and object graphs.</td><td>Object-oriented domain models and embedded/specialized apps.</td><td>Context lane. x64base is an object-oriented hybrid internally, but DBF-family records remain the public teaching anchor.</td></tr><tr><td>Distributed SQL / NewSQL</td><td>Relational semantics with horizontal and multi-region distribution.</td><td>ACID plus cloud/global scale.</td><td>Future context only. No current public case study.</td></tr><tr><td>Ledger / blockchain databases</td><td>Append-only or cryptographically verifiable record chains.</td><td>Auditability, tamper resistance, compliance trails.</td><td>Future context only. Do not add to case flow until a reviewed audit/provenance lesson exists.</td></tr><tr><td>In-memory databases</td><td>Data held primarily in RAM for low-latency workloads.</td><td>Real-time analytics and high-speed transactional paths.</td><td>Context lane. Useful when discussing buffers and runtime state, but not a separate case family yet.</td></tr></tbody></table>
<p>In x64base terms, the current work sits at the intersection of a few families:</p>
<ul>
<li><strong>Relational</strong>: schema, tuples, work areas, relations, orders, and business records.</li>
<li><strong>Embedded / file-based</strong>: DBF-family heritage, local tables, memos, indexes, and inspectable files.</li>
<li><strong>Key-value-adjacent</strong>: LMDB-backed index storage used as an implementation backend, not as the user-facing data model.</li>
<li><strong>Document-adjacent</strong>: JSON and metadata overlays as schema and documentation aids, not as a replacement for DBF-family table truth.</li>
<li><strong>Analytical-adjacent</strong>: filters, orders, browses, and reports can support analytical lessons, but x64base should not be described as a columnar OLAP database.</li>
</ul>
<p>The concise public description is: x64base is an educational, object-oriented
hybrid in the xBase/DBF lineage with relational, embedded, metadata, and
backend-indexing ideas. It should not be described as a full polyglot database
engine until runtime evidence and case coverage justify that stronger claim.</p>
<h2 id="deck-storyboard"><a class="no-underline" href="#deck-storyboard">Deck Storyboard</a></h2>
<p>The current deck organizes the path as a classroom sequence:</p>
<ol>
<li><strong>Foundations: COBOL and connected computers</strong> - business data, records, and ARPANET as preparation for AI literacy.</li>
<li><strong>Case study: JUMPS Army system</strong> - institutional scale, traceability, auditability, and consequences of error.</li>
<li><strong>Unisys / CODASYL COBOL at Alcoa</strong> - owner-member sets and enterprise relationships.</li>
<li><strong>xBase as a major platform</strong> - dBASE, Clipper, FoxPro, Visual FoxPro, and Microsoft interoperability.</li>
<li><strong>Earthkids to CAREPAX</strong> - daycare administration, receivables, vaccination scheduling, and market-fit lessons.</li>
<li><strong>Digital transfer, ERP, and industrial scale</strong> - TitleSCAN, document imaging, SQL workflows, auto-ID, transactions, and semiconductor process data.</li>
<li><strong>DotTalk++ / Laboratory Campus and the AI future</strong> - the xBase future that did not happen, revived as a learning lab for explainable data systems.</li>
</ol>
<h2 id="lab-implementation-pattern"><a class="no-underline" href="#lab-implementation-pattern">Lab Implementation Pattern</a></h2>
<p>Every era should become a small repeatable lab:</p>
<pre><code class="language-text">historical concept -> source artifact -> dataset -> command or UI -> proof transcript -> lesson
</code></pre>
<p>This lets the campus simulate or demonstrate a horizontal slice through database
history. Some slices can be live. Others can be carefully simulated when the
original hardware, software, or data is unavailable.</p>
