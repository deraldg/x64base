6:T2141,<h2 id="what-the-ai-portal-is"><a class="no-underline" href="#what-the-ai-portal-is">What the AI Portal Is</a></h2>
<p>The AI Portal is a machine-facing fast-start and task-context system for AI
development partners working with x64base.</p>
<p>It is <strong>not</strong> a student portal for accessing an AI service. Its job is to help a
new or resumed AI understand project authority, contracts, runtime behavior,
safety gates, and proof requirements without relying on lost chat history.</p>
<p>It enters through the maintained DotTalk++, maintenance, LabTalk, and PLDC
lifecycles. It does not create a parallel AI workflow or let publication bypass
an SDLC proof gate.</p>
<p>The public repository entry point is:</p>
<p><a href="https://github.com/deraldg/x64base/blob/main/AI_PORTAL.md">Open the x64base AI Portal on GitHub</a></p>
<h2 id="mandatory-status"><a class="no-underline" href="#mandatory-status">Mandatory Status</a></h2>
<p>The AI Portal lane is <strong>Alpha/Experimental</strong>. It is not autonomous project
memory, an independent source of truth, or permission to bypass maintainer
review.</p>
<pre><code class="language-text">task request
-> select relevant seeds
-> identify authority and contracts
-> explain every selected connection
-> expose missing or contradictory context
-> assemble a bounded fast-start packet
-> prepare guarded work and proof requirements
</code></pre>
<h2 id="current-fast-start-seeds"><a class="no-underline" href="#current-fast-start-seeds">Current Fast-Start Seeds</a></h2>
<p>The current repository seeds cover:</p>
<ul>
<li>development locations, branch continuity, and publication authority;</li>
<li>owning SDLC, current lane, truth/proof state, risk, and next gate;</li>
<li>mandatory contract review before source-code mutation;</li>
<li>DotTalk++ and DotScript learning/readiness;</li>
<li>outside-AI change-package intake;</li>
<li>task recipes, permission boundaries, and proof expectations.</li>
</ul>
<h2 id="current-implementation-inventory"><a class="no-underline" href="#current-implementation-inventory">Current Implementation Inventory</a></h2>
<p>The present foundation includes:</p>
<ul>
<li>repo-local AI orientation and assimilation documents;</li>
<li>LabTalk project, app, lesson, concept, proof, and portal registries;</li>
<li>a local registry browser and launcher;</li>
<li>read-only <code>MAINT AI</code> orientation commands;</li>
<li>runtime transcripts and truth-audit reports;</li>
<li>a machine-readable AI report-audit envelope on every new closeout (provider,
product, model or <code>not_exposed</code>, access mode, git baseline, and authorization
scope);</li>
<li>a closeout-updates-startup gate: a session that changes lane state also updates
the AI-facing startup documents and drops a dated closeout;</li>
<li>a document-as-you-work discipline: material steps are recorded as they land,
and evidence not captured in the moment is treated as not proven;</li>
<li>committed drift checks that gate source-derived website pages — the command and
function catalogs, error codes, and locale surfaces — against engine source
before a publish;</li>
<li>the APH-0 through APH-6 hardening plan.</li>
</ul>
<p>These pieces are useful and increasingly connected, but they do not yet
constitute validated autonomous project memory.</p>
<h2 id="hardening-gates"><a class="no-underline" href="#hardening-gates">Hardening Gates</a></h2>





































<table><thead><tr><th>Gate</th><th>Required outcome</th></tr></thead><tbody><tr><td>APH-0</td><td>Preserve the lane, remove unexplained drift, make startup reproducible, and establish tests.</td></tr><tr><td>APH-1</td><td>Turn flat registry metadata into a validated graph of typed, directional synapses.</td></tr><tr><td>APH-2</td><td>Compile deterministic, bounded, read-only task-context packets—the first useful MVP.</td></tr><tr><td>APH-3</td><td>Add reviewed capabilities, risk classes, approvals, containment, cancellation, and proof transcripts.</td></tr><tr><td>APH-4</td><td>Make task preparation and jump explanation the primary portal experience.</td></tr><tr><td>APH-5</td><td>Curate the graph and packet history in x64base with deterministic portable exports and recovery.</td></tr><tr><td>APH-6</td><td>Turn portal behavior into DotTalk++ lessons and measure cold-start task performance.</td></tr></tbody></table>
<h2 id="safety-and-authority"><a class="no-underline" href="#safety-and-authority">Safety and Authority</a></h2>
<ul>
<li>Read-only preparation is the default.</li>
<li>A registry record cannot grant itself execution authority.</li>
<li>Missing or contradictory evidence must be visible before action.</li>
<li>Source mutation requires the applicable contracts to be read first.</li>
<li>Mutating operations require explicit, separately reviewed capabilities and
maintainer authorization.</li>
<li>A generated packet is working context, not a new source of runtime truth.</li>
<li>Website summaries remain downstream reviewed derivatives.</li>
</ul>
<h2 id="working-with-chatgpt-gemini-grok-or-copilot"><a class="no-underline" href="#working-with-chatgpt-gemini-grok-or-copilot">Working With ChatGPT, Gemini, Grok, or Copilot</a></h2>
<p>An outside AI should receive:</p>
<ol>
<li>the GitHub AI Portal entry;</li>
<li>the exact public commit used as its baseline;</li>
<li>the target file or subsystem;</li>
<li>the permitted and excluded scope;</li>
<li>the expected proof.</li>
</ol>
<p>The request and returned package should also name the owning lifecycle, current
SDLC lane, truth state, proof state, risk class, and next gate.</p>
<p>It should return a unified patch package with a manifest, contracts read, risks,
and a test plan. That package is reviewed and selectively applied in the
maintainer-controlled authoritative development tree identified by the current
AI Portal location seed.</p>
<h2 id="authority-and-publication-flow"><a class="no-underline" href="#authority-and-publication-flow">Authority and Publication Flow</a></h2>
<pre><code class="language-text">authoritative development tree
  source mutation and runtime proof
        |
        | proven relevant files
        v
curated Git publication staging
  reviewed public projection
        |
        v
github.com/deraldg/x64base
  public snapshot used by outside AI systems
</code></pre>
<p>The website is a separate downstream publication. When verified behavior
changes public documentation, its separately maintained source is updated and
built only after the source change is proven.</p>
<h2 id="frontal-memory-and-inspectability"><a class="no-underline" href="#frontal-memory-and-inspectability">Frontal Memory and Inspectability</a></h2>
<p>The working metaphor is <strong>frontal memory</strong>: a durable, repo-backed entrance
that helps an AI recover project context and prepare a bounded task without
pretending that a model retains authoritative memory between sessions.</p>
<p>The mechanism must remain inspectable. Every selected artifact should explain
why it was included, what authority it has, what evidence supports it, what
permissions apply, and what remains missing or contradictory.</p>
<h2 id="future-seed-connector"><a class="no-underline" href="#future-seed-connector">Future Seed Connector</a></h2>
<p>The next prototype should stay small and inspectable: take a task statement,
select only the necessary seeds through explicit registry relationships,
explain why each seed was selected, and show missing context, conflicts,
permissions, and readiness.</p>
<p>It does not need a student chat UI, embeddings, or autonomous execution to
prove the fast-start concept.</p>
<h2 id="campus-role"><a class="no-underline" href="#campus-role">Campus Role</a></h2>
<p>The AI Portal is both an engineering tool and a Laboratory Campus lesson. Its
graph records, context budgets, validation failures, safety gates, transcripts,
and evaluation results should become inspectable DotTalk++ and Campus material.</p>
<p>This makes AI participation part of the same co-development loop as engine
work, documentation, proof, and curriculum—without pretending the experimental
portal is already complete.</p>
