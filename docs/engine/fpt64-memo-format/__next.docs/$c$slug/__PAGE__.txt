6:Tdc9,<p>Memo (<code>M</code>) fields in a DBF hold variable-length text or binary payloads. The DBF
record stores only a small <strong>reference</strong>; the payload lives outside the
fixed-length record. This page is a source-derived summary of the engine's memo
layer (<code>include/memo/memo64.hpp</code> and the memo backend).</p>
<h2 id="two-memo-backends"><a class="no-underline" href="#two-memo-backends">Two memo backends</a></h2>
<ul>
<li><strong>x64 tables (<code>DBF_VERSION_64</code>) use <code>memo64</code> — an LMDB-backed store.</strong> This is
the current x64base memo format; it is not a classic <code>.fpt</code> file.</li>
<li><strong>Legacy DBF/VFP tables use the classic FPT/DBT sidecar</strong> — fixed-size memo
blocks in a companion file, with a block pointer stored in the record. x64base
reads these for compatibility; new x64 tables use <code>memo64</code>.</li>
</ul>
<h2 id="the-memo-reference-in-the-dbf-record"><a class="no-underline" href="#the-memo-reference-in-the-dbf-record">The memo reference (in the DBF record)</a></h2>
<p>An <code>M</code> field in an x64 table stores a <strong><code>MemoRef64</code></strong>: an 8-byte unsigned
<code>memo_id</code>. A <code>memo_id</code> of <code>0</code> means an empty memo. The payload is fetched from the
memo64 store by that id.</p>
<h2 id="the-memo64-store-lmdb"><a class="no-underline" href="#the-memo64-store-lmdb">The memo64 store (LMDB)</a></h2>
<p><code>memo64</code> (version 1) keeps two named databases in an LMDB environment beside the
table:</p>
<ul>
<li>a <strong>values</strong> database keyed by <code>memo_id</code>, holding a value header followed by the
payload bytes;</li>
<li>a <strong>meta</strong> database holding control values such as <code>next_memo_id</code> (the id
allocator).</li>
</ul>
<p>Each stored value begins with a <strong>16-byte value header</strong>:</p>

























<table><thead><tr><th>Field</th><th>Width</th><th>Meaning</th></tr></thead><tbody><tr><td>flags</td><td>4 bytes</td><td>text / binary / compressed classification</td></tr><tr><td>reserved</td><td>4 bytes</td><td>reserved for future use</td></tr><tr><td>length</td><td>8 bytes</td><td>payload byte count</td></tr></tbody></table>
<p>Payload flags distinguish <strong>text</strong>, <strong>binary</strong>, and <strong>compressed</strong> memos, so the
store can hold both character memos and arbitrary blobs.</p>
<h2 id="integrity"><a class="no-underline" href="#integrity">Integrity</a></h2>
<ul>
<li>A reference is valid when it is <code>0</code> (empty) or resolves to a value in the store.</li>
<li>The value header's <code>length</code> bounds the payload read.</li>
<li>Memo/reference integrity failures raise the canonical <code>fpt64</code> facility error
(<code>e_fpt_block_invalid</code>, <code>fpt64/0x0001</code>) — see <a href="/docs/engine/error-codes">Error Codes</a>.</li>
</ul>
<h2 id="practical-tooling"><a class="no-underline" href="#practical-tooling">Practical tooling</a></h2>
<ul>
<li>Use <strong>TupTalk</strong> to inspect record memo references.</li>
<li>Use <strong>SmartList</strong> / export tooling for field-audit extracts.</li>
<li>Use <strong>TableTalk</strong> when memo edits interact with buffered table state and
<code>COMMIT</code>.</li>
</ul>
