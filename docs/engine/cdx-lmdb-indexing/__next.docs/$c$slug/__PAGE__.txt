c:T1221,<h2 id="teaching-model"><a class="no-underline" href="#teaching-model">Teaching Model</a></h2>
<p>The LabTalk explanation should start with the learner-visible model:</p>
<ol>
<li>A table stores records.</li>
<li>A CDX container stores named logical tags for that table.</li>
<li>An active order changes navigation without changing physical record order.</li>
<li>LMDB is the backend index environment that makes the built order durable and fast.</li>
</ol>
<p>That distinction matters. CDX is the container, creator, and manager surface for indexes in the xBase lineage. LMDB is the embedded database technology used underneath the modern x64base implementation.</p>
<h2 id="source-evidence"><a class="no-underline" href="#source-evidence">Source Evidence</a></h2>
<p>The current source tree shows this split in several places:</p>
<ul>
<li><code>include/cdx/cdx.hpp</code> defines the minimal CDX container contract: header I/O, table binding, tag directory read/write, and add/drop tag operations.</li>
<li><code>src/cli/cmd_cdx.cpp</code> exposes CDX container management commands.</li>
<li><code>src/cli/cmd_buildlmdb.cpp</code> builds or rebuilds the LMDB backing store for a CDX container.</li>
<li><code>src/cli/cmd_setorder.cpp</code> documents the policy: v64 tables use a CDX container backed by LMDB; public CDX paths resolve under <code>INDEXES</code>, while backend LMDB paths resolve under <code>LMDB</code>.</li>
<li><code>docs/cases/CASE_ENG_010_INDEX_NAVIGATION_CDX_LMDB.md</code> states the teaching rule directly: CDX is the user-facing logical container, and LMDB is a backend implementation detail.</li>
</ul>
<h2 id="operational-flow"><a class="no-underline" href="#operational-flow">Operational Flow</a></h2>
<pre><code class="language-text">DBF_64 / table area
  -> CDX container
       -> tag directory
       -> table binding
       -> logical order names
  -> BUILDLMDB
       -> LMDB environment
       -> named backend databases for tags
  -> SET ORDER / SEEK / INDEXSEEK / LIST_LMDB
       -> ordered navigation and diagnostics
</code></pre>
<p>In the lab campus, this gives us a clean horizontal slice. A student can see record storage, index metadata, backend materialization, and navigation behavior as separate layers of the same system.</p>
<h2 id="command-surface"><a class="no-underline" href="#command-surface">Command Surface</a></h2>





































<table><thead><tr><th>Command area</th><th>Role</th></tr></thead><tbody><tr><td><code>CDX</code></td><td>Create and inspect the logical CDX container and tag directory.</td></tr><tr><td><code>SETCDX</code></td><td>Attach a CDX container to the current work area.</td></tr><tr><td><code>BUILDLMDB</code></td><td>Materialize or rebuild the LMDB backend for the current CDX/tag model.</td></tr><tr><td><code>SETLMDB</code></td><td>Attach or configure the LMDB backend path where needed.</td></tr><tr><td><code>SET ORDER</code></td><td>Activate a logical tag/order for navigation.</td></tr><tr><td><code>INDEXSEEK</code></td><td>Seek through index-backed navigation.</td></tr><tr><td><code>LIST_LMDB</code> / <code>LMDBDUMP</code> / <code>LMDB_UTIL</code></td><td>Inspect backend state for diagnostics and advanced lessons.</td></tr></tbody></table>
<h2 id="why-this-fits-labtalk"><a class="no-underline" href="#why-this-fits-labtalk">Why This Fits LabTalk</a></h2>
<p>LabTalk is a laboratory campus, so the index system should be presented as a set of rooms:</p>
<ul>
<li><strong>Records room</strong> - physical table rows and record numbers.</li>
<li><strong>Container room</strong> - CDX headers, table binding, and tag directory.</li>
<li><strong>Backend room</strong> - LMDB environment, named databases, and rebuild lifecycle.</li>
<li><strong>Navigation room</strong> - <code>SET ORDER</code>, <code>SEEK</code>, <code>SKIP</code>, <code>TOP</code>, and <code>BOTTOM</code>.</li>
<li><strong>Diagnostics room</strong> - command output, case stories, runtime proofs, and SelfDoc contracts.</li>
</ul>
<p>This lets the same feature serve CS101, database history, systems programming, and advanced runtime inspection without changing the underlying architecture.</p>
