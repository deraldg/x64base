17:T170a,<h2 id="positioning"><a class="no-underline" href="#positioning">Positioning</a></h2>
<p>x64base should not be described as "the first 64-bit xBase" or as a replacement
for every existing xBase tool. The stronger claim is narrower and more durable:</p>
<p><strong>x64base is an experimental 64-bit DBF/xBase architecture project that
preserves the classic table-oriented workflow while exploring how DBF-style
files, memos, indexes, metadata, and runtime behavior can evolve beyond
historical structural limits.</strong></p>
<p>That framing keeps the project honest. It distinguishes x64base from compiler
projects, migration platforms, commercial modernization tools, DBF component
libraries, and legacy-compatible application frameworks.</p>
<h2 id="what-the-ecosystem-already-has"><a class="no-underline" href="#what-the-ecosystem-already-has">What The Ecosystem Already Has</a></h2>
<p>xBase did not disappear. It continued into several branches:</p>



































<table><thead><tr><th>Branch</th><th>Examples</th><th>General role</th></tr></thead><tbody><tr><td>Open-source compiler/runtime</td><td><a href="https://harbour.github.io/">Harbour</a>, xHarbour</td><td>Preserve and extend Clipper/xBase-compatible programming models.</td></tr><tr><td>Commercial modernization</td><td><a href="https://www.alaska-software.com/landing/clipper/start.cxp">Alaska Xbase++</a>, dBASE, FlagShip</td><td>Support Windows development, migration, modernization, or commercial xBase workflows.</td></tr><tr><td>.NET migration/language work</td><td>XSharp</td><td>Bring xBase-style language work into .NET-oriented environments.</td></tr><tr><td>DBF engines and libraries</td><td>Apollo, libxbase/Xbase64, Python DBF libraries</td><td>Provide DBF access, conversion, tooling, or embedded data-engine behavior.</td></tr><tr><td>Legacy historical systems</td><td>Visual FoxPro, CA-Clipper, dBASE ancestors</td><td>Define the practical and historical vocabulary that current projects inherit.</td></tr></tbody></table>
<p>Harbour describes itself as a cross-platform xBase implementation with compiler,
runtime libraries, UI, database, and I/O backends. Alaska Xbase++ markets itself
as a Clipper-compatible platform for 32- and 64-bit Windows. Microsoft lists
Visual FoxPro 9.0 extended support as ending on January 13, 2015. These facts
matter because they keep x64base from making unnecessary "only" or "first"
claims.</p>
<h2 id="where-x64base-fits"><a class="no-underline" href="#where-x64base-fits">Where x64base Fits</a></h2>
<p>Most current tools preserve, modernize, migrate, or interoperate with classic
xBase compatibility.</p>
<p>x64base asks a narrower engineering question:</p>
<p><strong>What would an open 64-bit DBF/xBase architecture look like if designed for
modern systems while keeping the classic workflow recognizable?</strong></p>
<p>That question points to a different center of gravity:</p>
<ul>
<li>DBF-style table and header evolution.</li>
<li>Memo architecture beyond older sidecar assumptions.</li>
<li>Index design that can represent classic intent while supporting newer backends.</li>
<li>Metadata space for longer names, field/type evolution, diagnostics, and
self-documentation.</li>
<li>A command-driven runtime that remains teachable and inspectable.</li>
</ul>
<h2 id="historical-limits"><a class="no-underline" href="#historical-limits">Historical Limits</a></h2>
<p>Classic DBF-family formats were designed around assumptions from earlier
computing eras. One visible example is the table header record-count field: the
dBASE file-structure reference describes bytes 4-7 as a 32-bit number containing
the number of records in the table.</p>
<p>x64base should frame this as structural research, not as a blanket scale claim.
The point is not "infinite scale." The point is that the old table, memo,
header, metadata, and index assumptions can be studied and reworked explicitly.</p>
<h2 id="what-to-say"><a class="no-underline" href="#what-to-say">What To Say</a></h2>
<p>Use:</p>
<ul>
<li>"experimental 64-bit DBF-style architecture"</li>
<li>"xBase-inspired runtime and workflow"</li>
<li>"research and engineering project"</li>
<li>"designed to explore larger table, memo, metadata, and index structures"</li>
<li>"education, exploration, and modernization-oriented"</li>
</ul>
<p>Avoid:</p>
<ul>
<li>"the first 64-bit xBase"</li>
<li>"the only 64-bit xBase"</li>
<li>"successor to all xBase systems"</li>
<li>"fully compatible with every DBF/xBase dialect"</li>
<li>"production-ready FoxPro/Harbour/Xbase++ replacement"</li>
</ul>
<h2 id="website-treatment"><a class="no-underline" href="#website-treatment">Website Treatment</a></h2>
<p>The public website should show this as context, not as the top-level hero. The
best placement is a lower homepage section that links here, paired with this
documentation page as the sourced explanation.</p>
