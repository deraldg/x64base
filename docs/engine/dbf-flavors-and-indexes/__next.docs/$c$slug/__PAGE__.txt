7:Tdce,<p>x64base is an object-oriented hybrid. The runtime centers on <code>DbArea</code> and <code>XBaseEngine</code> objects, while workspaces wrap active areas and preserve table, cursor, index, relation, memo, and metadata state. The storage layer is not only <code>DBF_64</code>; the current headers define a three-part DBF family.</p>
<h2 id="header-trinity"><a class="no-underline" href="#header-trinity">Header trinity</a></h2>

























<table><thead><tr><th>Flavor</th><th>Header evidence</th><th>Runtime position</th></tr></thead><tbody><tr><td>Classic / MS-DOS DBF</td><td><code>include/xbase.hpp</code>, <code>include/xbase_vfp.hpp</code>; versions <code>0x03</code>, <code>0x83</code>, and FoxPro memo <code>0xF5</code></td><td>Classic-like V32 area. Uses 10-byte descriptor field names and classic/FoxPro descriptor rules.</td></tr><tr><td>Visual FoxPro DBF</td><td><code>include/xbase_vfp.hpp</code>; versions <code>0x30</code>, <code>0x31</code>, <code>0x32</code></td><td>VFP-compatible V64 area with newer field/data type handling, including currency-oriented work and VFP descriptor extras.</td></tr><tr><td>x64 DBF_64</td><td><code>include/xbase_64.hpp</code>; version <code>0x64</code></td><td>x64 extension layer with large-header metadata, vector table/field names, fallback descriptor tokens, and x64 field metadata.</td></tr></tbody></table>
<p><code>include/xbase.hpp</code> is the neutral runtime contract. <code>include/xbase_vfp.hpp</code> bridges classic/MS-DOS, FoxPro, and VFP descriptor formats. <code>include/xbase_64.hpp</code> owns the x64 extension structures, vector metadata, and fallback name-resolution policy.</p>
<h2 id="index-defaults"><a class="no-underline" href="#index-defaults">Index defaults</a></h2>
<p>The command handlers make the current default behavior explicit:</p>























<table><thead><tr><th>Area flavor</th><th><code>SET ORDER</code> default container</th><th><code>SET INDEX</code> accepted containers</th><th>Notes</th></tr></thead><tbody><tr><td>V32 classic/MS-DOS/FoxPro-like</td><td><code>&#x3C;table>.cnx</code></td><td><code>.inx</code> or <code>.cnx</code></td><td><code>INX</code> is single-order attachment. Tag activation through <code>SET ORDER</code> is handled through CNX, not INX.</td></tr><tr><td>V64 VFP/x64</td><td><code>&#x3C;table>.cdx</code></td><td><code>.cdx</code></td><td>CDX is the command-facing container; LMDB is the physical backend where the CDX/LMDB environment is in use.</td></tr></tbody></table>
<p>This should be documented as current implementation behavior, not a permanent restriction. The Open Index API lane is the right place to describe future backend-neutral expansion.</p>
<h2 id="name-policy"><a class="no-underline" href="#name-policy">Name policy</a></h2>
<p>x64 DBF metadata can carry authoritative table and field names beyond classic DBF descriptor limits. When x64 metadata is missing, the runtime falls back to descriptor tokens and mangled names. That fallback is intentional: old tools can still see a legal DBF-shaped file, while x64-aware code can resolve the richer vector metadata.</p>
