17:T29a2,<h2 id="principles"><a class="no-underline" href="#principles">Principles</a></h2>
<ul>
<li><strong>Runtime truth first</strong> - command behavior and source boundaries matter more than aspirational product language.</li>
<li><strong>Object-oriented hybrid runtime</strong> - <code>DbArea</code>, <code>XBaseEngine</code>, workspace, memo, index, relation, and metadata objects carry state while preserving DBF-family interoperability.</li>
<li><strong>Visible database layers</strong> - tables, records, fields, indexes, relations, work areas, metadata, HELP, and validation are teaching surfaces.</li>
<li><strong>Separation of concerns</strong> - table storage, memo payloads, indexes, command handling, metadata, HELP, validation, and UI lanes are separate responsibilities.</li>
<li><strong>Proof language</strong> - distinguish runtime-proven behavior from active, partial, experimental, and canary work.</li>
<li><strong>Co-development</strong> - the implementation and documentation system are developed together. SelfDoc and the Master Documentation Organizer are part of the architecture, not a separate publishing chore.</li>
<li><strong>Open cross-platform architecture</strong> - the runtime should remain portable across Windows, WSL/Linux, Ubuntu, and macOS while keeping UI and index integrations behind open boundaries.</li>
</ul>

<p>Select the diagram to inspect the full-resolution SVG. This is a system-context
map rather than an entity-relationship model: its arrows describe authority,
execution, evidence, consumption, and reviewed promotion. Status labels keep
active runtime behavior separate from alpha, partial, experimental, and reserved

<p>The most important boundary is that the Laboratory Campus consumes and organizes
DotTalk++ and x64base evidence. It may launch, sequence, explain, and teach the
runtime, but it does not redefine engine or command behavior. The curriculum is
an evolving crosswalk and collection of proof-backed paths; it is not yet a
fixed seven-module syllabus.</p>
<h2 id="database-core-and-teaching-system"><a class="no-underline" href="#database-core-and-teaching-system">Database core and teaching system</a></h2>
<p>The database engine is the core architecture. Tables, records, fields, work
areas, indexes, memos, relations, buffering, locking, validation, and mutation
behavior are the project root.</p>
<p>DotTalk++, SelfDoc, MDO, manualgen, diagrams, and the Laboratory Campus are the
analysis and teaching architecture around that root. Their purpose is to keep
the engine observable: commands can be run, behavior can be proved, contracts
can be collected, documentation can be generated, and lessons can remain tied to
runtime truth.</p>
<p>This is why the documentation system is part of the architecture. It is not only
a website workflow. It is the mechanism that keeps the database work, the
teaching work, and the analyst work aligned.</p>
<h2 id="major-subsystems"><a class="no-underline" href="#major-subsystems">Major subsystems</a></h2>
<p>The local project is organized around these implementation areas:</p>
<ol>
<li><code>src/xbase</code> - table runtime, DbArea behavior, cursor state, field handling, locks, record views, and the DBF flavor trinity.</li>
<li><code>src/xindex</code> - INX/CNX/CDX/LMDB index infrastructure and command bridges.</li>
<li><code>src/memo</code> - memo store, memo manager, memo object, memo reference, and memo canaries.</li>
<li><code>src/cli</code> - DotTalk++ command surface, HELP, browsers, workspace commands, DDL, DrawIO, validation, and scripts.</li>
<li><code>src/datadict</code> - data dictionary readers, object resolution, catalog paths, and catalog bridge work.</li>
<li><code>src/xexpr</code> and <code>src/cli/expr</code> - expression and function evaluation.</li>
<li><code>bindings/pydottalk</code> - Python binding work and smoke probes.</li>
<li><code>docs/gui</code> and GUI source lanes - GUI-neutral services, wxWidgets frontend work, Python/Tk preview tools, and TUI alignment.</li>
</ol>
<h2 id="workspaces-and-areas"><a class="no-underline" href="#workspaces-and-areas">Workspaces and areas</a></h2>
<p>Workspaces are wrappers over active areas. An area is a <code>DbArea</code>-style object that owns the active table state, cursor position, selected order, relation hooks, field metadata, memo bindings, and validation context.</p>
<p>This matters for documentation and APIs:</p>
<ul>
<li>workspace save/load is object state management.</li>
<li>scripted workspace operations should describe the area objects they affect.</li>
<li>GUI panes should bind to workspace and area state instead of duplicating table state.</li>
<li>relation and index views should be derived from active area objects.</li>
</ul>
<h2 id="dbf-flavor-trinity"><a class="no-underline" href="#dbf-flavor-trinity">DBF flavor trinity</a></h2>
<p>The local headers document three active storage dialects: classic/MS-DOS DBF, Visual FoxPro DBF, and x64 <code>DBF_64</code>. Classic and FoxPro-like files are bridged through the VFP descriptor layer; VFP files carry newer field/data type compatibility; x64 files add large-header metadata, vector table/field names, and fallback descriptor mangling.</p>
<p>See <a href="/docs/engine/dbf-flavors-and-indexes">DBF Flavors and Index Defaults</a>.</p>
<h2 id="memo-and-field-extension-model"><a class="no-underline" href="#memo-and-field-extension-model">Memo and field extension model</a></h2>
<p>The memo subsystem is object-oriented: memo stores, memo managers, memo objects, and memo references need their own documentation lane.</p>
<p>In addition to student code hooks, x64base should expose custom field type hooks. Custom field types let education labs, domain experiments, and downstream products add semantics such as validators, display formatting, import/export rules, and editor behavior without patching the core DBF runtime.</p>
<h2 id="open-api-boundaries"><a class="no-underline" href="#open-api-boundaries">Open API boundaries</a></h2>
<p>The engine should maintain explicit open boundaries for:</p>
<ul>
<li>Open Index API - index creation, rebuild, verification, lookup, order selection, and backend-neutral lifecycle calls.</li>
<li>Open GUI API - GUI-neutral services for table browsing, workspace inspection, command execution, validation reports, and editor/view synchronization.</li>
<li>DotScript - repeatable command files over the same shell/runtime concepts.</li>
</ul>
<h3 id="student-and-custom-index-extension-lane"><a class="no-underline" href="#student-and-custom-index-extension-lane">Student and Custom Index Extension Lane</a></h3>
<p>The source reserves index space for students, researchers, and downstream users
to experiment without colliding with the core INX, CNX, and CDX/LMDB families.
The current surface is intentionally mixed-status:</p>



































<table><thead><tr><th>Surface</th><th>Current evidence</th><th>Status</th></tr></thead><tbody><tr><td><code>SIX</code></td><td>Registered command with <code>CREATE</code>, <code>BUILD</code>, and <code>INFO</code> over the local single-tag index stub.</td><td>Active educational stub</td></tr><tr><td><code>SCX</code></td><td>Student/local compound index-file lab family and the current second custom slot.</td><td>Active educational stub</td></tr><tr><td><code>REINDEX CUSTOM</code></td><td>Explicitly runs <code>SIX + SCX</code>; <code>REINDEX ALL</code> excludes them by design.</td><td>Active opt-in dispatcher</td></tr><tr><td><code>SNX</code></td><td>Header/container/catalog groundwork exists, but <code>edu_snx.cpp</code> exports no command and shell registration is disabled.</td><td>Reserved and partial</td></tr><tr><td>User-supplied backend attachment</td><td>Intended adapter boundary for custom indexing systems through the Open Index API.</td><td>Planned contract</td></tr></tbody></table>
<p>This proves an extension laboratory and reserved namespace, not yet a universal
plug-in ABI. The goal is for a student or user to implement an index family,
attach it through a documented lifecycle interface, exercise it against a
<code>DbArea</code>, compare behavior and timing, and preserve results as proof artifacts
without changing the core index guarantees.</p>
<h2 id="current-build-surface"><a class="no-underline" href="#current-build-surface">Current build surface</a></h2>
<p>The top-level CMake project is <code>DotTalkpp</code> version <code>0.6</code>, C++20.</p>
<p>Important options:</p>
<pre><code class="language-text">DOTTALK_WITH_TV
DOTTALK_WITH_GUI
DOTTALK_WITH_WX
DOTTALK_WITH_INDEX
DOTTALK_WITH_EDUCATION
DOTTALK_WITH_RELATIONS
BUILD_PYDOTTALK
DOTTALK_PROFILE
</code></pre>
<p>The current Windows release payload keeps <code>dottalkpp.exe</code>, <code>lmdb.dll</code>, and
<code>sqlite3.dll</code> separate. LMDB and SQLite are streamlined dynamic dependencies;
the larger <code>dottalkpp.exe</code> is the compiled command/runtime host containing the
shell, engine services, scripting, HELP/SelfDoc, metadata, browser, relation,
workspace, and TUI-enabled command surfaces.</p>
<p>See <a href="/docs/engine/runtime-footprint">Runtime Footprint and Dependencies</a>.</p>
<h2 id="current-focus"><a class="no-underline" href="#current-focus">Current focus</a></h2>
<p>The current useful work is not broad feature expansion. The local tree points toward validation, congruence, GUI/runtime bridging, metadata reports, CDX/LMDB lifecycle cleanup, memo proof, and keeping browser/workspace smoke paths honest.</p>
<p>For the index teaching model, see <a href="/docs/engine/cdx-lmdb-indexing">CDX and LMDB Indexing</a>. CDX is the logical container and command-facing manager; LMDB is the physical backend used for built index environments.</p>
