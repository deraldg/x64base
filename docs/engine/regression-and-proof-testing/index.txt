17:T103b,<p>x64base verifies behavior three ways: a <strong>curated regression launcher</strong> inside
the shell, a <strong>hash-bound proof-transcript</strong> discipline for durable evidence, and
build-integrated <strong>CTests</strong>. The disposable <a href="/docs/engine/pinocchio-benchmarks">Pinocchio</a>
lane adds scale and fault-injection runs on top.</p>
<h2 id="the-regression-launcher"><a class="no-underline" href="#the-regression-launcher">The <code>REGRESSION</code> launcher</a></h2>
<p><code>REGRESSION</code> is a curated launcher, not a separate test executor — it runs
reviewed DotScript smoke/regression files through the normal
<a href="/docs/dottalk/dotscript-language-guide">DotScript</a> runner. Subcommands:</p>
<ul>
<li><code>REGRESSION LIST</code> — show the curated stable entrypoints (not every historical
script on disk).</li>
<li><code>REGRESSION SHOW NONDESTRUCTIVE</code> — show what a named suite will run.</li>
<li><code>REGRESSION RUN INDEX_X64</code> — run one curated suite by name.</li>
<li><code>REGRESSION ALL</code> — run the curated default suite in declared order.</li>
</ul>
<p>Each suite bootstraps its own environment. Because scripts may mutate data and
session state, <code>REGRESSION</code> is a read-and-execute surface with no built-in
transaction or rollback — suites are expected to be self-contained.</p>
<h2 id="curated-suites"><a class="no-underline" href="#curated-suites">Curated suites</a></h2>









































<table><thead><tr><th>Suite</th><th>Focus</th></tr></thead><tbody><tr><td><code>NONDESTRUCTIVE</code></td><td>Read-only smoke across core commands.</td></tr><tr><td><code>INDEX_X32</code></td><td>32-bit INX / CNX index behavior.</td></tr><tr><td><code>INDEX_X64</code></td><td>64-bit CDX / LMDB index behavior.</td></tr><tr><td><code>X64_METRICS</code></td><td>x64 matrix metrics and boundary canary.</td></tr><tr><td><code>LANGUAGE</code></td><td>Localization shakedown — <code>es</code>/<code>fr</code>/<code>de</code>/<code>it</code> USAGE rendering across the localized command surface.</td></tr><tr><td><code>HARVEST</code></td><td>Documentation/metadata harvest shakedown.</td></tr><tr><td><code>CURSOR</code></td><td>Cursor / ordered-navigation regression.</td></tr><tr><td><code>RELJOIN</code></td><td>Relational join enumeration regression.</td></tr></tbody></table>
<p>Dev-only warning and reproduction canaries stay outside this curated surface
unless they are intentionally promoted into it.</p>
<h2 id="proof-transcripts"><a class="no-underline" href="#proof-transcripts">Proof transcripts</a></h2>
<p>Correctness-critical changes are bound to a <strong>hash-verified teed transcript</strong>: the
run's full output is captured to a log and its SHA-256 recorded, so a result is a
durable artifact rather than a recollection. Recent examples include the
table-buffer WAL durability phases and the Pinocchio scale-verb fixes; the
repository retains dozens of these run logs under <code>labtalk/proofs/runs/</code>.</p>
<p>This is the same discipline the documentation pipeline uses: a claim is
<code>runtime-proven</code> only when a captured, hash-bound transcript backs it.</p>
<h2 id="build-integrated-tests"><a class="no-underline" href="#build-integrated-tests">Build-integrated tests</a></h2>
<p>The CMake build registers <strong>CTests</strong> (via <code>include(CTest)</code> / <code>add_test</code>) that run
as part of the build — for example the xBase physical-order proof matrix and the
pydottalk contract smoke — so structural regressions surface at build time, not
only through the shell launcher.</p>
<h2 id="scale-and-fault-injection--pinocchio"><a class="no-underline" href="#scale-and-fault-injection--pinocchio">Scale and fault injection — Pinocchio</a></h2>

stress-test surface: million-row datasets, ordered-navigation and scale-verb
benchmarks, and the Phase 2 fault-injection work (the buffered-lane durability
WAL is the first delivered slice). Its data, tables, and indexes stay isolated
