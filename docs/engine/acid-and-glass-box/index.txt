<h2 id="the-central-distinction"><a class="no-underline" href="#the-central-distinction">The central distinction</a></h2>
<p>A glass-box educational architecture does not inherently weaken ACID.
Visibility and transactional guarantees are separate dimensions. DotTalk++ can
show buffers, stale fields, locks, commit boundaries, and recovery evidence
without changing the correctness of the mechanisms underneath them.</p>
<p>The useful question is not whether the engine exposes its state. It is whether
each operation, storage lane, and failure boundary provides a tested guarantee.</p>

























<table><thead><tr><th>Finding</th><th>Meaning</th><th>Example</th></tr></thead><tbody><tr><td>Educational tradeoff</td><td>Complexity is exposed or staged deliberately for learning.</td><td>Showing dirty and stale fields before commit.</td></tr><tr><td>Implementation gap</td><td>A required mechanism is absent or incomplete.</td><td>A persistent cross-store recovery journal remains a stub.</td></tr><tr><td>Transactional guarantee</td><td>Repeatable evidence establishes a scoped behavior.</td><td>A crash/reopen test proves an acknowledged write survives.</td></tr></tbody></table>
<h2 id="two-important-corrections"><a class="no-underline" href="#two-important-corrections">Two important corrections</a></h2>
<h3 id="buffered-state-is-not-persistent-inconsistency"><a class="no-underline" href="#buffered-state-is-not-persistent-inconsistency">Buffered state is not persistent inconsistency</a></h3>
<p>A tuple-buffer edit can be dirty or stale while the persistent DBF and its
indexes remain unchanged. The risk boundary occurs when a direct mutation or
commit updates one participating store without updating, invalidating,
rebuilding, or recovering the others.</p>
<p>The current <code>COMMIT</code> contract says partial commit is possible and that CDX/LMDB
rebuild is outside the command. These are explicit engineering and test
boundaries, not evidence for a broad compliance or failure claim.</p>
<h3 id="inspection-does-not-weaken-isolation"><a class="no-underline" href="#inspection-does-not-weaken-isolation">Inspection does not weaken isolation</a></h3>
<p>Showing a user their own pending changes does not weaken isolation. Isolation
depends on session visibility, lock scope, versioning, conflict detection, and
concurrent operation ordering. Buffering and per-record commit-lock mechanisms
exist in source; the effective isolation model still requires captured
multi-session tests.</p>
<h2 id="acid-by-property"><a class="no-underline" href="#acid-by-property">ACID by property</a></h2>



































<table><thead><tr><th>Property</th><th>Current rating</th><th>What exists</th><th>What must still be proved</th></tr></thead><tbody><tr><td>Atomicity</td><td>Partial (buffered lane)</td><td>Pre-commit table-buffer rollback, per-record commit locking, and a durable redo log with an atomic commit marker + crash recovery for the buffered lane (proven 2026-07-19).</td><td>The same across non-buffered/direct paths and a multi-store DBF + memo + index transaction.</td></tr><tr><td>Consistency</td><td>Unverified</td><td>Visible dirty/stale state and validation mechanisms.</td><td>Invariants after direct mutation, successful commit, failed commit, and recovery.</td></tr><tr><td>Isolation</td><td>Unverified</td><td>Buffered state and commit locking mechanisms.</td><td>Dirty-read, repeatable-read, lost-update, conflict, and visibility behavior.</td></tr><tr><td>Durability</td><td>Partial (buffered lane)</td><td>LMDB-local commits, memo flush, and a crash-recoverable <code>fsync</code>'d <code>.tbj</code> redo log for buffered table transactions (proven 2026-07-19).</td><td>A portable DBF <code>fsync</code>, post-commit index reconciliation, and acknowledgement/reopen/power-loss behavior for every non-buffered store.</td></tr></tbody></table>
<h2 id="scope-before-score"><a class="no-underline" href="#scope-before-score">Scope before score</a></h2>
<p>ACID ratings belong to an operation and storage lane, not to a product name
alone.</p>
<ul>
<li>buffered x64 mutation</li>
<li>direct mutation</li>
<li>LMDB-local mutation</li>
<li>DBF + LMDB combined operation</li>
<li>DBF + memo combined operation</li>
<li>legacy index lane</li>
</ul>
<p>A strong local guarantee cannot be generalized to a multi-store operation it
does not cover.</p>
<h2 id="update--2026-07-19-buffered-lane-durability-table-buffer-wal"><a class="no-underline" href="#update--2026-07-19-buffered-lane-durability-table-buffer-wal">Update — 2026-07-19: buffered-lane durability (table-buffer WAL)</a></h2>
<p>The table-buffer <code>COMMIT</code> / <code>ROLLBACK</code> path gained a durable <strong>write-ahead redo
log</strong>. Buffered writes append to an <code>fsync</code>'d <code>.tbj</code> sidecar; <code>COMMIT</code> writes an
atomic marker and syncs it <strong>before</strong> applying to the DBF; recovery-on-open
replays a committed log or discards an uncommitted one. Three hash-verified
proofs cover the durable writer, recovery replay, and buffered <code>DELETE</code>.</p>
<p>This is the first <strong>proof-backed</strong> mechanism for the buffered lane's Atomicity
and Durability, where the commit/rollback journal was previously a stub. It is
deliberately scoped: it does <strong>not</strong> address concurrency or isolation, and two
gaps remain — a portable DBF <code>fsync</code> after replay, and CDX/LMDB index
reconciliation after a buffered commit (an indexed table still needs a
<code>REINDEX</code>). This evidence is now recorded as a re-rating in
<code>acid_assessment_beta-1.md</code>, moving the buffered-lane Atomicity and Durability to
<strong>Partial</strong> in the table above — a scoped gain, not full ACID.</p>
<h2 id="source-defined-limits"><a class="no-underline" href="#source-defined-limits">Source-defined limits</a></h2>
<ul>
<li><code>COMMIT</code> documents <code>partial_commit_possible: yes</code>.</li>
<li><code>COMMIT</code> does not rebuild CDX or LMDB indexes.</li>
<li>the buffered-lane commit/rollback journal is <strong>no longer a stub</strong> — a durable
<code>fsync</code>'d <code>.tbj</code> redo log with crash recovery landed 2026-07-19 (see the update
note above); a coordinated cross-store (DBF + memo + index) recovery journal
is still open.</li>
<li>LMDB-local transactions do not prove a coordinated DBF + memo + index
transaction.</li>
</ul>
<p>These observations explain why the formal rating remains <code>Unverified</code>. They do
not diminish the value of the glass-box design; they identify exactly what the
project must test and harden.</p>
<h2 id="acid-as-a-lesson"><a class="no-underline" href="#acid-as-a-lesson">ACID as a lesson</a></h2>
<p>ACID is both an engineering objective and a Laboratory Campus lesson. A learner
should be able to predict a guarantee, observe participating state, inject a
safe failure into disposable data, reopen the stores, and assign an
evidence-backed rating. A failed test is useful when preserved honestly because
it converts an abstract acronym into a concrete requirement.</p>
