17:T2055,<h2 id="snapshot"><a class="no-underline" href="#snapshot">Snapshot</a></h2>
<p>This page records the observed Windows release payload from the local build on
July 7, 2026:</p>

























<table><thead><tr><th>File</th><th align="right">Size</th><th>Role</th></tr></thead><tbody><tr><td><code>dottalkpp.exe</code></td><td align="right">6,675,456 bytes</td><td>DotTalk++ command shell, x64base runtime surface, scripting, HELP, browser, relation, metadata, and TUI-enabled command host.</td></tr><tr><td><code>lmdb.dll</code></td><td align="right">82,432 bytes</td><td>LMDB runtime used as the physical backend for LMDB-backed index environments.</td></tr><tr><td><code>sqlite3.dll</code></td><td align="right">1,077,760 bytes</td><td>SQLite runtime used by SQLite and SQL bridge lanes.</td></tr></tbody></table>
<p>The important point is that LMDB and SQLite are streamlined external runtime
dependencies. They are not what makes <code>dottalkpp.exe</code> larger.</p>
<h2 id="dynamic-dependencies"><a class="no-underline" href="#dynamic-dependencies">Dynamic dependencies</a></h2>
<p>PE inspection of <code>dottalkpp.exe</code> shows that the executable imports:</p>
<ul>
<li><code>sqlite3.dll</code></li>
<li><code>lmdb.dll</code></li>
<li>Windows system libraries such as <code>KERNEL32.dll</code>, <code>USER32.dll</code>, <code>SHELL32.dll</code>,
<code>ADVAPI32.dll</code>, <code>WINHTTP.dll</code>, and <code>WS2_32.dll</code></li>
<li>MSVC runtime libraries such as <code>MSVCP140.dll</code>, <code>VCRUNTIME140.dll</code>,
<code>VCRUNTIME140_1.dll</code>, and Universal CRT DLLs</li>
</ul>
<p>That means SQLite and LMDB are dynamically loaded support libraries. Their code
is not statically folded into the DotTalk++ executable.</p>
<h2 id="why-the-executable-is-larger"><a class="no-underline" href="#why-the-executable-is-larger">Why the executable is larger</a></h2>
<p><code>dottalkpp.exe</code> is larger because it is the full application runtime, not a thin
launcher. The release build links the command shell and internal libraries that
make DotTalk++ the canonical human-facing surface for x64base.</p>
<p>The build links or includes:</p>
<ul>
<li>DotTalk++ command handlers across the <code>cmd_*</code> family.</li>
<li><code>src/xbase</code> table runtime, work areas, cursor state, field handling, locking,
record views, and DBF flavor handling.</li>
<li><code>src/xindex</code> index infrastructure and CDX/LMDB command bridges.</li>
<li><code>src/memo</code> memo manager, memo object/reference/store concepts, and memo proof
surfaces.</li>
<li>DotScript parsing, variables, control flow, expression evaluation, and scalar
function support.</li>
<li>HELP, SelfDoc, command reflection, function catalogs, metadata mining,
message catalogs, and manual/report generation support.</li>
<li>Relation, tuple, workspace, DDL, import/export, CSV, SQL, data dictionary, and
validation command lanes.</li>
<li>Browser, SmartList, table buffering, dirty/stale state, and TUI integration
hooks.</li>
<li>Turbo Vision/TUI support for the Arctic terminal lane.</li>
</ul>
<p>This is why the executable is best described as a compact native runtime host
with a broad command/documentation surface, not as a minimal embedded database
library.</p>

<p>The observed release executable contains these main PE sections:</p>








































<table><thead><tr><th>Section</th><th align="right">Approximate size</th><th>Meaning</th></tr></thead><tbody><tr><td><code>.text</code></td><td align="right">5.36 MB</td><td>Compiled code. <p>Most of the footprint is therefore DotTalk++ code and read-only runtime data.</p>

itself.</p>
<h2 id="build-flags-observed"><a class="no-underline" href="#build-flags-observed">Build flags observed</a></h2>
<p>The local release build used optimization and dynamic MSVC runtime linking:</p>
<pre><code class="language-text">/O2
/Ob2
/MD
/std:c++20
NDEBUG
</code></pre>
<p>It also enabled several DotTalk++ feature flags:</p>
<pre><code class="language-text">DOTTALK_WITH_DEV=1
DOTTALK_PROFILE_DEV=1
DOTTALK_EXTRA_DIAGNOSTICS=1
DOTTALK_WITH_RELATIONS=1
DOTTALK_XEXPR_AVAILABLE=1
DOTTALK_WITH_INDEX=1
DOTTALK_TV_AVAILABLE=1
DOTTALK_SQLITE_AVAILABLE=1
</code></pre>
<p>Those flags explain why the current release build includes developer,
diagnostic, relation, expression, index, TUI, and SQLite command surfaces. A
smaller distribution build could be created later by intentionally separating
developer/selfdoc tooling, command catalogs, TUI support, or optional surfaces,
but the current beta build keeps them together because they are part of the
proof and documentation loop.</p>
<h2 id="documentation-rule"><a class="no-underline" href="#documentation-rule">Documentation rule</a></h2>
<p>When documenting the runtime footprint:</p>
<ul>
<li>Do say that DotTalk++ is a compact native runtime compared with framework-heavy
app stacks.</li>
<li>Do say that LMDB and SQLite are streamlined external DLL dependencies.</li>
<li>Do not imply that <code>dottalkpp.exe</code> is only a database kernel.</li>
<li>Do not blame the size on SQLite or LMDB; they are dynamically imported.</li>
<li>Do explain that the executable contains the command shell, engine services,
scripting, SelfDoc/HELP, metadata, browsers, relations, and TUI hooks.</li>
</ul>
