17:T2618,<p>Pinocchio is the disposable engine stress-test lane used to answer a practical
question: can the educational database behave like a real engine at
million-row scale? Its generated data, tables, indexes, and LMDB stores remain
isolated from protected project data.</p>
<h2 id="current-proof-state"><a class="no-underline" href="#current-proof-state">Current proof state</a></h2>
<p>The retained Phase 1 workload contains 1,000,000 student rows and 5,501,358
enrollment rows. The original read battery found that ordered <code>TOP</code> navigation
was scanning and sorting the table even when a CDX/LMDB order was active. A
development correction routed ordered navigation through the LMDB cursor.</p>
<p>The following values are historical results, not a fresh website benchmark:</p>

<h2 id="addendum--record-addressing-at-the-billion-plus-boundary-2026-07-19"><a class="no-underline" href="#addendum--record-addressing-at-the-billion-plus-boundary-2026-07-19">Addendum — record addressing at the billion-plus boundary (2026-07-19)</a></h2>
<p>The results above measure <em>throughput</em> at million-row scale. A separate question
is <em>addressing</em>: can the engine locate and read a record whose number is past the
old 32-bit ceiling (<code>2^31 - 1</code>, about 2.1 billion)? As of 2026-07-19 the answer is
yes — proven end to end <strong>without</strong> materializing a multi-billion-row table:</p>
<ul>
<li>A synthetic boundary unit test (<code>dottalkpp_recno64_boundary_test</code>) resolves
distinct record numbers at <code>INT32_MAX</code>, <code>+1</code>, <code>+2</code>, <code>UINT32_MAX</code>, and
<code>UINT32_MAX + 1</code>, while the legacy 32-bit accessors return <code>-1</code> at overflow.</li>
<li>A <strong>real sparse-file end-to-end test</strong> (<code>dottalkpp_recno64_sparse_e2e_test</code>)
sets an x64 table's <code>record_count</code> to <code>2^31 + 2</code>, writes real records at recno
<code>1</code>, <code>2^31 + 1</code>, and <code>2^31 + 2</code> into a <strong>sparse</strong> file — the ~18 GiB gap is a
filesystem hole, a few KB physical — reopens through the engine, and reads the
two records past <code>2^31</code> <strong>distinctly off disk</strong> in about 0.5 s.</li>
</ul>
<p>So record <em>addressing</em> is proven past <code>2^31</code> (past ~2.1 billion). What is <strong>not</strong>
proven here is a <em>fully populated</em> multi-billion-row table: that is a data-volume
and throughput question — the million-row results above are the current throughput
evidence — not an addressing-correctness one. The path-by-path status lives in
<a href="/docs/engine/x64-capacity-math">64-Bit Capacity Math</a>.</p>
<h2 id="phase-13--scale-verb-hardening-2026-07-19"><a class="no-underline" href="#phase-13--scale-verb-hardening-2026-07-19">Phase 1.3 — scale-verb hardening (2026-07-19)</a></h2>
<p>Beyond ordered navigation, the read and write command verbs affected by indexing
and scale were profiled and hardened at the same 1M / 5.5M workload. Each result
is bound to a hash-verified teed transcript; all are development-only, not
promoted.</p>





























<table><thead><tr><th>Verb</th><th align="right">Before</th><th align="right">After</th><th>Fix</th></tr></thead><tbody><tr><td><code>SEEK</code> near-last key (5.5M ENROLL)</td><td align="right">56.99 s</td><td align="right">0.0033 s</td><td>Keyed LMDB range-seek (<code>MDB_SET_RANGE</code>, O(log n)) with DBF re-verify, replacing a linear index walk (1M STUDENTS near-last: 16.57 s → 0.0020 s).</td></tr><tr><td><code>DELETE FOR</code> under active order (~9,999 rows)</td><td align="right">65.4 s</td><td align="right">37.8 s</td><td>Bulk write-transaction batching: the ~25 s per-row index-commit surcharge collapses to one commit per 10,000-row chunk; the remaining ~40 s is the inherent predicate scan.</td></tr><tr><td><code>AGGS ALL</code> (COUNT/SUM/AVG/MIN/MAX, 1M)</td><td align="right">75.4 s (four scans)</td><td align="right">19.5 s (one scan, ~4×)</td><td>Single-pass multi-aggregate.</td></tr></tbody></table>
<p>A companion fix normalized unquoted string literals in <code>SET FILTER</code> and aggregate
<code>FOR</code> predicates — the source of the earlier <code>COUNT SET FILTER</code> returning 0 while
<code>COUNT FOR</code> returned the correct count.</p>
<h2 id="durability--table-buffer-write-ahead-log-phase-2-slice"><a class="no-underline" href="#durability--table-buffer-write-ahead-log-phase-2-slice">Durability — table-buffer write-ahead log (Phase 2 slice)</a></h2>
<p>The deferred Phase 2 durability question has its first answer. Table-buffer
<code>COMMIT</code> / <code>ROLLBACK</code> are now <strong>crash-recoverable</strong> through a durable, <code>fsync</code>'d
<code>.tbj</code> redo log with recovery-on-open, proven across three hash-verified phases
(durable writer, recovery replay, and buffered <code>DELETE</code> through the log). This is
scoped to the buffered table lane and does <strong>not</strong> claim full ACID — see
<a href="/docs/engine/acid-and-glass-box">ACID and the Glass-Box Engine</a>. Concurrency
fault injection remains the open Phase 2 work.</p>
<h2 id="machine-identity"><a class="no-underline" href="#machine-identity">Machine identity</a></h2>
<p>The individual run transcripts did not self-record the machine, so the raw
measurements carry no embedded hardware label. The maintainer has since
<strong>attested</strong> that <strong>all</strong> runs were done on one machine — every run through
2026-07-19, including the Phase 1 stress test (2026-07-15), the ordered-navigation
before/after runs (2026-07-18), the scale-verb and table-buffer durability runs
(2026-07-19), and the RECNO64 boundary and sparse-file addressing tests
(2026-07-19). That profile is preserved with the benchmark evidence
(<code>PINOCCHIO_MACHINE_PROFILE_CURRENT_V1.json</code>) and bound as <code>MAINTAINER_ATTESTED</code>:</p>
<ul>
<li><strong>Alienware m16 R2</strong> (notebook), BIOS 1.21.0.</li>
<li><strong>Intel Core Ultra 9 185H</strong> (Meteor Lake) — 16 cores / 22 threads (6 P + 8 E +
2 LP), 5.1 GHz P-core turbo, 24 MB L3, 45 W.</li>
<li><strong>64 GB DDR5-5600</strong> SO-DIMM (2 × 32 GB), 4 × 32-bit channels, JEDEC CL46, no XMP/EXPO.</li>
<li><strong>NVIDIA RTX 4070 Laptop (8 GB)</strong> + Intel Arc integrated graphics; Intel AI Boost NPU.</li>
<li>Project tree and DBF/CDX/LMDB fixtures on a <strong>Samsung 970 EVO 1 TB NVMe (<code>D:</code>)</strong>.</li>
<li><strong>Windows 11 Professional</strong> (x64), Build 26200.8875.</li>
</ul>
<p>Honesty note: this is a deliberate maintainer attestation made after the runs,
recorded as <code>MAINTAINER_ATTESTED</code> rather than silently auto-attached to each row.
Future runs should still capture their own per-run machine profile beside the raw
build and measurement transcripts; comparable rows require the machine label,
engine state, dataset size, operation, timer source, and evidence paths.</p>
