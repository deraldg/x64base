17:Tef4,<p>This page names the public API boundaries the project should protect as the implementation matures. It is not a frozen ABI. It is the policy-level surface for open architecture work.</p>
<h2 id="core-engine-api"><a class="no-underline" href="#core-engine-api">Core engine API</a></h2>
<ul>
<li>open/close table</li>
<li>iterate records</li>
<li>get/set fields</li>
<li>memo read/write</li>
<li>inspect structure and field metadata</li>
<li>expose typed errors and diagnostics</li>
</ul>
<h2 id="workspace-and-area-api"><a class="no-underline" href="#workspace-and-area-api">Workspace and area API</a></h2>
<p>Workspaces are wrappers over active areas. Each area is a <code>DbArea</code>-style object with table state, cursor state, indexes, relations, field metadata, memo references, and validation context.</p>
<p>The workspace API should expose:</p>
<ul>
<li>create, select, close, save, and load workspace.</li>
<li>enumerate areas.</li>
<li>inspect active area state.</li>
<li>bind relation and index views to area state.</li>
<li>run validation against one area or a workspace set.</li>
</ul>
<h2 id="open-index-api"><a class="no-underline" href="#open-index-api">Open Index API</a></h2>
<ul>
<li>create index</li>
<li>rebuild index</li>
<li>verify index</li>
<li>open/close index environment</li>
<li>select active order</li>
<li>seek and scoped traversal</li>
<li>expose logical CDX/CNX/INX concepts without leaking one physical backend</li>
<li>support LMDB-backed lifecycle where applicable</li>
<li>report integrity and rebuild diagnostics</li>
</ul>
<p>The Open Index API is the stable integration boundary for alternate index backends, teaching tools, validation tools, and GUI order selectors.</p>
<h2 id="open-gui-api"><a class="no-underline" href="#open-gui-api">Open GUI API</a></h2>
<p>The Open GUI API should keep user interface work separate from table mechanics. GUI implementations should call services instead of reimplementing the engine.</p>
<p>Expected service groups:</p>
<ul>
<li>browse table and current work area.</li>
<li>inspect field and structure metadata.</li>
<li>run DotTalk++ commands and DotScript files.</li>
<li>display validation and SelfDoc reports.</li>
<li>show index/order state.</li>
<li>show relation graphs.</li>
<li>edit records through engine validation.</li>
<li>synchronize views with workspace and area events.</li>
</ul>
<p>This boundary allows wxWidgets, Python/Tk, Turbo Vision/TUI, and future frontends to share the same runtime contract.</p>
<h2 id="custom-field-type-api"><a class="no-underline" href="#custom-field-type-api">Custom field type API</a></h2>
<p>In addition to student code hooks, x64base should support custom field type hooks.</p>
<p>Custom field types should be able to provide:</p>
<ul>
<li>storage and conversion rules.</li>
<li>validation rules.</li>
<li>display and editor hints.</li>
<li>import/export behavior.</li>
<li>documentation metadata for SelfDoc and the Master Documentation Organizer.</li>
</ul>
<h2 id="memo-api"><a class="no-underline" href="#memo-api">Memo API</a></h2>
<p>The memo API should reflect the object-oriented memo model:</p>
<ul>
<li>memo store.</li>
<li>memo manager.</li>
<li>memo object.</li>
<li>memo reference.</li>
<li>read/write/update operations.</li>
<li>validation and repair diagnostics.</li>
</ul>
<h2 id="errors"><a class="no-underline" href="#errors">Errors</a></h2>
<ul>
<li>typed errors</li>
<li>numeric codes</li>
<li>structured diagnostics for script, index, memo, workspace, and GUI boundaries</li>
</ul>
<h2 id="integration"><a class="no-underline" href="#integration">Integration</a></h2>
<ul>
<li>DotScript command files.</li>
<li>Python bindings.</li>
<li>C-compatible boundary candidates.</li>
<li>GUI-neutral service bridge.</li>
<li>generated documentation and SelfDoc extraction.</li>
