<h2 id="the-output-channel"><a class="no-underline" href="#the-output-channel">The output channel</a></h2>
<p>Command diagnostics flow through one channel, <code>cli::cmdout</code>
(<code>src/cli/command_output.hpp</code>), rather than raw <code>std::cout</code>. The channel offers:</p>
<ul>
<li><code>print_prefixed_message(cmd, id, args)</code> — a localized, <code>cmd:</code>-prefixed line;</li>
<li><code>emit_error(cmd, code, ...)</code> / <code>emit_warning(...)</code> — the same, plus a canonical
<a href="/docs/engine/error-codes">error code</a> recorded via <code>set_last_error</code>.</li>
</ul>
<p>Routing a command's diagnostics through this channel pays four dividends at once:
regional <strong>language</strong> (from the catalog), <strong>severity</strong> (<code>xbase::error</code> code),
<strong><code>SET ERRORSTOP</code></strong> awareness (via <code>set_last_error</code>), and consistent <code>cmd:</code>
formatting. Tabular payload — <code>LIST</code>, <code>DUMP</code>, <code>SMARTLIST</code> output — deliberately
stays on the data channel and is <strong>not</strong> localized.</p>
<h2 id="the-message-catalog"><a class="no-underline" href="#the-message-catalog">The message catalog</a></h2>
<p>Localized text is keyed by a stable <code>MessageId</code> (roughly 900+ message ids in
<code>src/help/helpdata_messages.hpp</code>) and rendered per locale by
<code>format_message_catalog(locale, ...)</code>. Messages live in a runtime table,
<code>SYSTEM_MESSAGES.dbf</code>, and are maintained through the <code>MSGMGR</code> command, which
seeds and checks message rows in reviewed priority tiers:</p>
<ul>
<li><strong>Priority A</strong> — <code>SET MESSAGE</code> / <code>MSGMGR</code> rows;</li>
<li><strong>Priority B</strong> — the demonstrated command-surface rows;</li>
<li><strong>Priority C</strong> — <code>USE</code> / <code>DISPLAY</code> / navigation runtime lines.</li>
</ul>
<p><code>MSGMGR STATUS</code> and <code>MSGMGR CHECK</code> are read-only report surfaces; the
<code>SEED ... CHECK</code> / <code>SEED ... APPLY</code> forms maintain the rows.</p>
<h2 id="locales-and-regions"><a class="no-underline" href="#locales-and-regions">Locales and regions</a></h2>
<p>The active locale set is data-driven through a DBF/CDX/LMDB <strong>locale spine</strong>
(<code>SYSTEM_LOCALES.dbf</code> plus <code>SYSTEM_LOCALE_FALLBACK.dbf</code>, with matching CDX and
LMDB index containers), inspected by <code>locale_spine_catalog</code>.</p>



































<table><thead><tr><th>Locale</th><th>Region</th><th>Role</th></tr></thead><tbody><tr><td><code>en-US</code></td><td>United States English</td><td>Base / fallback locale</td></tr><tr><td><code>de</code></td><td>German</td><td>Localized surface</td></tr><tr><td><code>es</code></td><td>Spanish</td><td>Localized surface</td></tr><tr><td><code>fr</code></td><td>French</td><td>Localized surface</td></tr><tr><td><code>it</code></td><td>Italian</td><td>Localized surface</td></tr></tbody></table>
<p>Resolution uses a simple <strong>fallback chain</strong>: the requested locale first, then
<code>en-US</code>. So a message missing in <code>fr</code> renders its <code>en-US</code> text rather than a raw
key, and the base locale always resolves.</p>
<h2 id="proof"><a class="no-underline" href="#proof">Proof</a></h2>
<p>The <code>REGRESSION LANGUAGE</code> battery is the runtime proof for this surface:
"Messaging-normalization locale proof: <strong>es/fr/de/it USAGE render</strong> across the
localized command surface." It exercises the localized command help across the
four non-base locales and confirms each renders through the catalog.</p>
<h2 id="status"><a class="no-underline" href="#status">Status</a></h2>
<p>The channel, catalog, and locale spine are <strong>source-defined and runtime-read-back</strong>
for the surfaces above. Two honest caveats:</p>
<ul>
<li><strong>Adoption is in progress.</strong> Not every command yet routes its diagnostics
through <code>cmdout</code> / canonical codes; converting the remaining command surface —
and making each converted command <code>SET ERRORSTOP</code>-aware — is an active lane.</li>
<li><strong>Compiled vs active provider.</strong> The compiled catalog and the active
<code>SYSTEM_MESSAGES.dbf</code> provider are maintained separately and are not assumed to
be byte-synchronized; message counts cited from source describe the compiled
catalog, and a live count comes from the active DBF via <code>MSGMGR STATUS</code>.</li>
