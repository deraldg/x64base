17:T220e,<p>DBF_64 is x64base's 64-bit-capacity evolution of the xBase DBF table. It keeps
the familiar self-describing header + fixed-length records, but lifts record
counts and file offsets past the classic 32-bit ceiling while staying
backward-compatible with tools that read the legacy header. This page is a
source-derived summary of the engine's own reader/writer and validator
(<code>src/xbase/dbf_file.cpp</code>, <code>include/xbase/dbf_create.hpp</code>,
<code>src/cli/dbf64_header_validate.cpp</code>).</p>

<h2 id="version-byte"><a class="no-underline" href="#version-byte">Version byte</a></h2>
<p>The header's version byte identifies the flavor. The engine recognizes the
classic families (<code>0x03</code>, <code>0x83</code>, <code>0xF5</code>) and Visual FoxPro (<code>0x30</code>, <code>0x31</code>,
<code>0x32</code>), and adds its own <strong><code>DBF_VERSION_64</code></strong> marker for 64-bit tables. When the
version byte is <code>DBF_VERSION_64</code>, the authoritative record count is the 64-bit
field (below) rather than the legacy 32-bit one.</p>
<h2 id="table-header"><a class="no-underline" href="#table-header">Table header</a></h2>
<p>The base header follows the VFP-style layout (version byte, last-update date,
record count, header length, and record length), so legacy readers still see a
valid DBF. DBF_64 then carries a <strong>large-header extension</strong> whose <code>record_count</code>
is a 64-bit value.</p>
<ul>
<li>The classic <strong>32-bit record count at offset 4</strong> is preserved and patched for
compatibility (clamped when the true count exceeds a 32-bit maximum).</li>
<li>The <strong>64-bit record count</strong> in the extension is the authoritative row count for
a <code>DBF_VERSION_64</code> table.</li>
<li>Header length and record length locate the first data byte and the fixed record
stride.</li>
</ul>
<h2 id="field-descriptors"><a class="no-underline" href="#field-descriptors">Field descriptors</a></h2>
<p>Each field descriptor defines a name, a type, a length, and decimal places (for
numeric types). Which type codes <code>CREATE</code> accepts depends on the <strong>DBF flavor</strong>.
The classic types are shared by every flavor; the binary Visual FoxPro types
(<code>I</code>/<code>B</code>/<code>Y</code>/<code>T</code>) are stored as <strong>real binary</strong> — not the fixed-width ASCII
placeholders older builds used — and are accepted by the VFP and x64 flavors.</p>

<p>The four binary types round-trip as their true VFP encodings — verified against an
independent, spec-based VFP reader and writer in both directions — rather than the
ASCII placeholders earlier builds stored. <strong>Custom field types</strong> register a type
code once (codec + on-disk width + flavor eligibility + display name) and then flow
through <code>CREATE</code>, validation, and storage with no further engine change; the
built-in binary types use that same registry.</p>
<p>Memo (<code>M</code>) fields store a reference, not the payload — see
<a href="/docs/engine/fpt64-memo-format">FPT64 Memo Format</a>.</p>
<p>The catalog also reserves <code>V</code> (Varchar), <code>Q</code> (Varbinary), <code>W</code> (Blob), <code>G</code>
(General), and the SQL-native extensions (<code>BigInt</code>, <code>Timestamp</code>, <code>Uuid</code>, …) as
<strong>planned</strong> types; these are not yet accepted by <code>CREATE</code>.</p>
<h2 id="records"><a class="no-underline" href="#records">Records</a></h2>
<ul>
<li><strong>Fixed-length</strong> records; the stride is the header's record length.</li>
<li>A leading <strong>deleted flag</strong> byte (space = live, <code>*</code> = deleted) precedes the
field-aligned payload.</li>
<li>Record positions are computed as <code>data_start + recno * record_length</code> using
<strong>64-bit arithmetic</strong> with overflow checks, so the positioning math itself is
not the 32-bit ceiling (subject to the path-by-path status in
<a href="/docs/engine/x64-capacity-math">64-Bit Capacity Math</a>).</li>
<li>A trailing <strong><code>0x1A</code></strong> byte marks end-of-file.</li>
</ul>
<h2 id="runtime-widening-status-public-build-2026-07"><a class="no-underline" href="#runtime-widening-status-public-build-2026-07">Runtime widening status (public build, 2026-07)</a></h2>
<p>The on-disk layout above is 64-bit, and the runtime that operates on it has been
widened to match. In the current public build, 64-bit record positioning
(<code>gotoRec64</code>), the navigation commands, the record-lock API, and the active
CDX/LMDB index record numbers are widened and regression-checked (CURSOR,
INDEX_X64); the classic <code>recno()</code>/<code>recLength()</code>/<code>recCount()</code>/<code>cpr()</code> accessors
now return <code>-1</code> past their 32-bit range instead of clamping to <code>INT_MAX</code>. A synthetic boundary
unit test <strong>and a real sparse-file end-to-end test</strong> resolve distinct record numbers
past <code>2^31</code> — the engine reads records at recno <code>2^31+1</code>/<code>+2</code> off a sparse x64 table
whose <code>record_count</code> is <code>2^31+2</code> — so <strong>64-bit record addressing is proven past
<code>2^31</code></strong>. Materializing a <em>fully populated</em> multi-billion-row table is a separate
data-volume question. The path-by-path status lives in
<a href="/docs/engine/x64-capacity-math">64-Bit Capacity Math</a>.</p>
