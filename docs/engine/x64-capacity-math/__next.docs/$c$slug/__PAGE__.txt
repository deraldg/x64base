<p>A table format can put a 64-bit record count in the header, but the system is
not truly x64 all the way if navigation, locks, memo pointers, index payloads,
or GUI APIs still pass record numbers through 16-bit or 32-bit slots.</p>
<h2 id="why-x64-all-the-way"><a class="no-underline" href="#why-x64-all-the-way">Why x64 all the way?</a></h2>
<p>Classic DBF-family files are beautifully simple: a header, field descriptors,
fixed-length records, optional memo sidecars, and indexes. That simplicity also
means the math is visible. If a header field is 16 bits wide, its maximum is
bounded by <code>2^16 - 1</code>. If a record counter is 32 bits wide, its natural ceiling
is <code>2^32 - 1</code>, and many products impose lower practical limits for file size,
locking, compatibility, or runtime memory.</p>
<p>Going x64 changes the shape of the possible database:</p>






























<table><thead><tr><th>Width</th><th align="right">Unsigned maximum</th><th>What it means in a DBF-style engine</th></tr></thead><tbody><tr><td>8-bit</td><td align="right">255</td><td>Classic one-byte field lengths and flags.</td></tr><tr><td>16-bit</td><td align="right">65,535</td><td>Header length, record length, field offsets, and compact index limits in many classic paths.</td></tr><tr><td>32-bit</td><td align="right">4,294,967,295</td><td>Record numbers, file offsets, memo blocks, lock offsets, and index payloads in many older engines.</td></tr><tr><td>64-bit</td><td align="right">18,446,744,073,709,551,615</td><td>Large record counts, file offsets, memo object ids, index page addresses, and future table metadata.</td></tr></tbody></table>
<p>The lesson is not that every table should contain trillions of records. The
lesson is that each old counter becomes a choke point unless it is widened
through the whole stack.</p>
<p>For x64base, it helps to distinguish three different ceilings:</p>





















<table><thead><tr><th>Ceiling type</th><th>Current meaning in x64base</th></tr></thead><tbody><tr><td>64-bit design field</td><td>The x64 extension can store wide counts, offsets, and row geometry.</td></tr><tr><td>16-bit compatible mirror</td><td>The VFP-style compatibility header still mirrors header size and record size through 16-bit fields, and those mirrors may saturate at <code>65535</code>.</td></tr><tr><td>Parser / construction limit</td><td>Some creation surfaces still impose narrower convenience limits, such as the current <code>CREATE X64</code> single-character-field cap of <code>4096</code>, even though the runtime can now honor wider total row geometry.</td></tr></tbody></table>
<p>These are not the same thing, and the matrix below should not pretend they are.</p>
<h2 id="the-choke-point-chain"><a class="no-underline" href="#the-choke-point-chain">The choke-point chain</a></h2>
<p>For a DBF-style runtime to be truly 64-bit, the following layers need to agree:</p>
<ol>
<li><strong>Table header</strong> - record count, record length, header length, and data-start offset.</li>
<li><strong>Record navigation</strong> - <code>GOTO</code>, <code>SKIP</code>, <code>RECNO</code>, EOF/BOF, append, delete, and scan loops.</li>
<li><strong>Record locking</strong> - lock-file names, lock offsets, owner records, and stale-lock cleanup.</li>
<li><strong>Memo references</strong> - in-row memo reference width and sidecar object/block addressing.</li>
<li><strong>Index payloads</strong> - key entries must carry wide record numbers and wide page or object offsets.</li>
<li><strong>Expression/runtime APIs</strong> - scalar functions, commands, GUI bindings, and scripting need wide-safe values.</li>
<li><strong>Diagnostics and manuals</strong> - reports must say whether a value is a design target, a header capability, or a runtime-proven limit.</li>
</ol>
<p>If any one of those layers still clamps a value to 16 or 32 bits, the system may
have an x64 file header but not an x64 operating envelope.</p>
<h3 id="record-number-widening-status-development-2026-07"><a class="no-underline" href="#record-number-widening-status-development-2026-07">Record-number widening status (development, 2026-07)</a></h3>
<p>The record-number widening lane (internal name RECNO64) is progressively removing
the 32-bit choke points listed above. The table records the current
development-only state; none of it is promoted, and prior states are preserved in
this site's version history.</p>





































<table><thead><tr><th>Choke-point layer</th><th>Current development state</th></tr></thead><tbody><tr><td>Table header</td><td>64-bit record count, length, and data-start via the <code>*64</code> accessors.</td></tr><tr><td>Record navigation</td><td>64-bit positioning (<code>gotoRec64</code>); <code>GO</code>/<code>GOTO</code>/<code>SKIP</code>/<code>RECNO</code>/<code>TOP</code>/<code>BOTTOM</code>/<code>FIRST</code>/<code>LAST</code>/<code>NEXT</code>/<code>PRIOR</code> regression-checked (CURSOR, INDEX_X64).</td></tr><tr><td>Record locking</td><td>64-bit — lock API and lock-file record keys widened from 32-bit.</td></tr><tr><td>Memo references</td><td>64-bit DTX object ids, offsets, and append offsets.</td></tr><tr><td>Index payloads</td><td>Active CDX/LMDB path is 64-bit (index hooks + keyed-seek decoders); classic CNX and legacy <code>.inx</code> kept 32-bit by design with a capability report.</td></tr><tr><td>Expression / runtime APIs</td><td>Mixed: the classic <code>recno()</code>/<code>recLength()</code>/<code>cpr()</code> accessors still saturate at <code>INT_MAX</code> — the last layer that decides the ceiling.</td></tr><tr><td>Diagnostics and manuals</td><td>This page and the DBF-64 specification separate design target from runtime-proven limit.</td></tr></tbody></table>
<p>The decisive measurement — resolving a distinct record beyond <code>2^31</code> — has not yet
been performed, because no table of that size exists in the current proof lanes.
Until that boundary test exists, the honest claim is <em>the record-number path is
being widened through the stack</em>, not that unlimited scale is proven.</p>

<p>It proves two separate x64 row-geometry cases:</p>
<ul>

active path.</p>

<h2 id="student-exercise-find-the-hidden-32-bit-ceiling"><a class="no-underline" href="#student-exercise-find-the-hidden-32-bit-ceiling">Student exercise: find the hidden 32-bit ceiling</a></h2>
<p>Take any proposed x64 feature and ask these questions:</p>





























<table><thead><tr><th>Question</th><th>Why it matters</th></tr></thead><tbody><tr><td>Is the value stored on disk as 64-bit?</td><td>If not, the file format itself is the ceiling.</td></tr><tr><td>Is the value kept in memory as 64-bit?</td><td>If not, large values are lost after open.</td></tr><tr><td>Do commands accept and print the value as 64-bit?</td><td>If not, scripts and diagnostics lie.</td></tr><tr><td>Do locks, indexes, and memos carry the same width?</td><td>If not, mutation and navigation break before storage does.</td></tr><tr><td>Do GUI/TUI/API bindings expose the wide value?</td><td>If not, the engine may be wider than the application surface.</td></tr></tbody></table>
<p>For x64base, that exercise produces real engineering lanes:</p>
<ul>
<li>keep wide x64 geometry authoritative even when compatibility mirrors saturate.</li>
<li>audit remaining create/export/utility surfaces that still emit or assume classic-sized geometry.</li>
<li>audit every <code>int32_t</code> and <code>uint32_t</code> record-number path.</li>
<li>prove x64 record navigation, append, delete, replace, lock, and scan behavior.</li>
<li>prove index backends with 64-bit record payloads.</li>
<li>keep fallback descriptor tokens for DBF interoperability while preserving x64 vector names.</li>
<li>make the manuals report current, proven, planned, and design-target limits separately.</li>
</ul>
