7:T5c1a,<h2 id="purpose"><a class="no-underline" href="#purpose">Purpose</a></h2>
<p>This comparison places x64base / DotTalk++ Workbench in the broader xBase,
DBF, Clipper, FoxPro, and DBF-library ecosystem.</p>
<p>It is not a replacement-claim table. Some systems here are mature commercial
products, some are open-source compilers, some are migration platforms, and
some are file-access libraries. x64base is different in emphasis: its primary
intent is educational and architectural, with practical database/runtime uses
developing around that core.</p>
<h2 id="reading-the-table"><a class="no-underline" href="#reading-the-table">Reading The Table</a></h2>
<ul>
<li><strong>Yes</strong> means the feature is present or clearly supported.</li>
<li><strong>Partial</strong> means the feature exists in a limited, developing, or scoped form.</li>
<li><strong>No</strong> means it is not a meaningful feature of that system.</li>
<li><strong>Unknown</strong> means we should not claim it without better source proof.</li>
<li><strong>Not same claim</strong> means the project has adjacent capability but not the same
architectural goal.</li>
</ul>
<p>A "Yes" for Harbour or Xbase++ may mean mature application-platform support. A
"Yes" for x64base means project/source/runtime evidence exists, but alpha status
still applies unless specifically marked production-ready.</p>

<h2 id="application-ui-dsl-gap"><a class="no-underline" href="#application-ui-dsl-gap">Application UI DSL Gap</a></h2>
<p>DotTalk++ can currently define repeatable database/runtime behavior through
commands and DotScript, but it does not yet offer a general language for creating
custom menus, windows, dialogs, popups, controls, and event handlers.</p>
<p>The proposed lane starts with this syntax seed:</p>
<pre><code class="language-text">CREATE MENU MainMenu
CREATE WINDOW CustomerWin
CREATE DIALOG EditCustomer
DEFINE BUTTON Save
ON CLICK DO save_customer
</code></pre>
<p>This is a planned lane, not current runtime syntax. It would let future
DotTalk++ scripts define small database applications that can target Arctic TUI
first, then Parallel GUI/TUI, wxWidgets GUI, and web surfaces later.</p>
<h2 id="summary-position"><a class="no-underline" href="#summary-position">Summary Position</a></h2>
<p>x64base should be positioned as:</p>
<p><strong>An educational-first, open, object-oriented hybrid xBase/DBF architecture
project exploring 64-bit DBF-style tables, memos, indexes, metadata, command
runtime behavior, GUI/TUI tooling, and repeatable self-documentation.</strong></p>
<p>It should not be positioned as:</p>
<ul>
<li>a drop-in Harbour replacement</li>
<li>a drop-in Xbase++ replacement</li>
<li>a finished commercial DBMS</li>
<li>a fully compatible Visual FoxPro clone</li>
<li>the only 64-bit xBase-related technology</li>
</ul>
<p>For the capacity side of this comparison, see
<a href="/docs/engine/x64-capacity-math">x64 Capacity Math</a>. That page separates
published ecosystem limits from x64base source-evidenced limits and planned
x64 widening work.</p>
