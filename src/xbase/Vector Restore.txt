## Project handoff

### Core correction

We went off track by modifying `xbase.hpp` for x64 vectored metadata. That was the wrong layer.

The correct architecture is:

* `xbase.hpp` = core engine/runtime contract only
* `xbase_vfp.hpp` = classic/VFP lineage loader
* `xbase_64.hpp` = x64 extension layer, including vectored metadata
* `xbase_cli.hpp` = engine/CLI separator, not part of the storage codex

Use lowercase `trinity` and `god` unless referring to the theological real thing.

## Decision now locked

### Do not extend `xbase.hpp` for x64 vector metadata

Do not add to `DbArea` in `xbase.hpp`:

* vector name cache
* metadata version/flags
* table/field name byte-length caches
* x64 metadata visibility fields

`DbArea` should only receive the **resolved final names** from the x64 loader.

### Put the vector system in `xbase_64.hpp`

That includes:

* `X64MetaLocator`
* `X64MetaHeader`
* `X64FieldNameEntry`
* metadata block reading/validation
* resolving vectored names into existing runtime fields:

  * `_logical_name`
  * `FieldDef.name`

No new x64 metadata state needs to live in `DbArea`.

## x64 vector metadata contract

### Capability

* table name max: 128
* field name max: 64
* practical default field-name policy: 16
* recommended target: 16–20
* UTF-8
* no truncation
* tooling may abbreviate for display only

### Fallback short-name policy

* default: relaxed
* toggleable: strict

Relaxed is best for conversion/import scenarios. Strict is for data-dictionary-grade schemas.

### Runtime rule

Loader reads x64 metadata block and resolves:

* table name -> `_logical_name`
* field names -> `FieldDef.name`

Then discard metadata details. `DbArea` is not the metadata mirror.

## Current technical problem that must be fixed next

### x64 create/write path and x64 loader are out of sync

The x64 loader now expects:

* 32-byte base header
* 64-byte x64 extension
* 16-byte `X64MetaLocator`
* field descriptors

So field descriptors start at byte **112**.

But `CREATE X64` was still writing the old layout, causing reopen failure with:

* truncated classic field descriptor
* broken reopen after create

### Required fix in `cmd_create.cpp`

For x64 create:

* write the 16-byte `X64MetaLocator` after `LargeHeaderExtension`
* even if metadata block is absent initially:

  * `meta_offset = 0`
  * `meta_length = 0`
  * `meta_version = 0`
  * `meta_flags = 0`
* recalculate header/data start using the new x64 fixed pre-descriptor size

Until that is fixed, newly created x64 tables will mismatch the loader.

## Tuple / buffer architecture state

This remains valid and should be preserved:

* `Cell` = edit unit
* `Row` = row object
* `Schema` = runtime column contract
* tuple table = ideal edit workspace
* row = commit unit
* cell = edit unit
* if any field changes, row is dirty
* if indexed field changes, index is stale
* commit writes full row from the buffer

That logic is separate from header/vector storage and should remain separate.
