AI Portal Collection Point
==========================

Status: source-side seed for AI-assisted work in the DotTalk++ / x64base /
Laboratory Campus ecosystem.

Purpose
-------

This folder is the intake point for AI-facing rules, handoffs, source truth
notes, documentation-lane policy, and preservation guidance. It should point AI
work back to source/runtime/HELP/metadata/SelfDoc/MDO evidence rather than
letting website or chat prose drift into technical truth.

Promotion / Staging Convention
------------------------------

Normal source-code flow:

D:/code/ccode -> C:/x64base -> GitHub repository

Normal website flow:

D:/dev/x64base-site -> build/public artifact -> GitHub Pages -> x64base.com

Normal documentation evidence flow:

D:/code/ccode source/runtime
  -> HELP / metadata / comments / contracts
  -> SelfDoc / MDO / manualgen reports
  -> reviewed manual sections
  -> reviewed website summaries

Website-only artifact flow:

D:/dev/x64base-site public asset
  -> reviewed website page
  -> manual only if the artifact is non-derivable elsewhere, separately
     maintained, provenance-labeled, and owned.

Do not reverse the authority chain by copying website prose into manuals or
source docs as technical truth. If the website text describes command behavior,
file formats, data mutation, indexes, memo storage, triggers, buffering, record
locking, locale, SQL/relational behavior, or runtime claims, return to the
implementation checkout and regenerate or re-review the evidence.

Core Gates
----------

Rule 1: most technical data and information on x64base.com must be derived,
harvested, calculated, or reviewed from the implementation checkout. The website
may frame and organize that information, but it is not the source of truth for
runtime behavior, command syntax, file formats, index behavior, memo behavior,
HELP, metadata, or manual truth.

Rule 2: manuals may contribute reviewed sections, diagrams, summaries, and
publication-ready explanations to the website. Manuals must not consume website
copy as technical truth unless the website artifact is not derivable from the
implementation checkout and is maintained separately as a public artifact.

Rule 3: README.* files are sacred orientation and provenance artifacts. They may
be improved, annotated, or superseded, but if replaced the prior version must be
preserved as a versioned artifact. Do not clobber README files during website,
manual, staging, or AI-assisted work.

Promotion Direction
-------------------

Normal simplex flow:

implementation checkout
  -> HELP / metadata / contracts / SelfDoc / MDO / manualgen
  -> reviewed manual
  -> reviewed website summary

Allowed duplex flow:

reviewed manual section -> reviewed website summary

Blocked by default:

website prose -> manual technical truth

Allowed exception:

website-owned public artifact -> manual

The exception is valid only when the artifact is non-derivable elsewhere,
separately maintained, provenance-labeled, and owned. Examples include curated
screenshots, public-only media, branding/navigation framing, contact details,
and hosted-download metadata.

AI Safety Notes
---------------

- Do not treat chat output or website prose as source truth.
- Do not silently promote planned or canary behavior into stable documentation.
- Do not let SelfDoc, MDO, metadata, HELP, comments, contracts, catalog pages,
  manualgen, and website pages drift apart.
- For commands, functions, file formats, mutation behavior, buffering, locking,
  indexing, triggers, or runtime claims, return to implementation evidence.
- Preserve README files and other orientation artifacts before replacement.
