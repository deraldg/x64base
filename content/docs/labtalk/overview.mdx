The **Laboratory Campus** is the learning-lab and collaboration framing for the
x64base / DotTalk++ ecosystem. LabTalk® remains the project mark and legacy name,
but the campus is the public frame to emphasize.

It is not just a single app. It is an alpha hybrid of planning material,
implemented pieces, in-development labs, datasets, runtime demonstrations, help
systems, GUI experiments, and proof artifacts that make computing systems easier
to study and easier to collaborate on.

## Laboratory Campus

Think of it as a campus:

- **Runtime Systems Lab** - DotTalk++ commands, scripts, work areas, tables, records, indexes, and relations.
- **Historical Data Systems Lab** - punch cards, COBOL, CODASYL, xBase, SQL, ERP, cloud computing, and AI-era data literacy.
- **Self-Documenting Systems Lab** - comments, contracts, HELP, CMDHELP, CMDHELPCHK, metadata, and proof readback.
- **Dataset Library** - small inspectable datasets for lessons and repeatable labs.
- **Case Library** - engineering and historical cases that connect live demonstrations to real system stories.
- **GUI, pycrud, and Portal Lab** - local launchers, CRUD/database literacy tools, dashboards, and front-end experiments that expose the same underlying evidence.

The first public history path is the [Database Evolution Path](/docs/labtalk/database-evolution),
which integrates the LabTalk / DotTalk++ storyboard deck as a teaching artifact
from punch cards and COBOL through xBase, enterprise data, cloud computing, and AI.

The [Education Features](/docs/labtalk/education-features) page now exposes
source-defined teaching surfaces such as `IDX` timed index/sort labs, student
command/function hooks, `SORT` lab behavior, and pre/post polling hooks.

The [Runtime Evidence Gallery](/docs/labtalk/runtime-evidence) records reviewed screenshots from Arctic, the local campus portal, workspace loading, table browsing, tuple output, and GUI/TUI readback for use by the manual and website.

The [Lesson Platform](/docs/labtalk/lessons) now separates student lesson seeds
from career lessons learned while building the system. The site is also open to
structured input through [Suggest a Lesson](/docs/labtalk/suggest-a-lesson).

The [LMS Communications Lane](/docs/labtalk/lms-integration-lane) documents a
provider-neutral local queue, CLI, and portal status command. Moodle is recorded
only as a future provider candidate: live delivery is disabled, and no endpoint,
token, student data, or grade data is configured.

## Repository Boundary

The Laboratory Campus is a **consumer layer**, not a replacement authority for
the engine.

- `x64base` remains the owner of engine/runtime truth, including command
  semantics, cursor state, relations, indexing, and the Arctic GUI/TUI
  workbench surfaces.
- `DotTalk++` remains the best runtime/manual identity for the command shell and
  teaching shell surfaces.
- `LabTalk` owns the portal, registries, labs, proofs, assignments, and campus
  overlays that consume runtime truth.

That means the campus may launch, explain, and sequence runtime behavior, but
it should not silently redefine the behavior that belongs to the runtime.

The [curated ecosystem relationship map](/docs/engine/architecture#ecosystem-relationship-map)
shows this boundary across x64base engine services, DotTalk++, DotScript,
SelfDoc/MDO publication, proof artifacts, curriculum, and campus delivery lanes.
It explicitly labels active, evolving, alpha, experimental, and reserved work.

## Core Pattern

Campus material should follow a proof-aware path:

[![Laboratory Campus proof-aware learning path](/images/labtalk/laboratory-campus-proof-path-v2.svg)](/docs/labtalk/cases-storyboard)

```text
concept -> app -> dataset -> command -> proof -> case -> lesson
```

This keeps lessons connected to live or reviewed evidence instead of becoming
detached prose.

## Current Status

The Laboratory Campus is an active alpha direction. Public x64base pages should
describe it as a campus model, collaboration area, and learning-lab program, not
as a finished packaged distribution unless that packaging has been explicitly
released.

## Academic and Research Direction

The [Academic Positioning and Research Agenda](/docs/labtalk/academic-positioning)
is supporting Laboratory Campus material, not the campus hero or primary entry
point. It frames candidate work across database education, learning sciences,
HCI, end-user programming, literate computing, explainable systems, and
computing education research while separating implemented system evidence from
educational hypotheses that still require literature review and empirical
study.
