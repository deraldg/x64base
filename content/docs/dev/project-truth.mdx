## Current status

DotTalk++ / x64base should be described as a working educational and research runtime in active beta, not as a finished commercial database product.

Current proven or active areas include:

- DBF-style table runtime, cursor state, records, fields, work areas, and table inspection.
- x64-family table and metadata work.
- INX, CNX, CDX, and LMDB-backed index work.
- Memo subsystem with object-oriented architecture and canary coverage.
- DotTalk++ command shell and DotScript automation.
- HELP, command reflection, `CMDHELPCHK`, and metadata congruence work.
- Relation-aware browsing through `ERSATZ` / `RBROWSE`.
- Table listing plus record and tuple browsing through `SMARTLIST`, `SMARTBROWSE`, `SM`, `SMART`, `SIMPLEBROWSE`, `SB`, `BROWSE`, `BROWSER`, and `BROWSETUI`.
- TableTalk is the table-buffering lane: `SET TABLE BUFFER`, `TABLE`, dirty/stale state, `COMMIT`, `ROLLBACK`, and buffered mutator behavior.
- Data dictionary readers, catalog paths, and object resolver code.
- Python binding experiments through `pydottalk`.
- Indexing architecture across INX, CNX, CDX, LMDB, Open Index API boundaries, and backend diagnostics.
- GUI work across wxWidgets, Python/Tk preview tools, pycrud, and Turbo Vision-style TUI surfaces.
- GUI/front-end branding uses `DotTalk++ Workbench` as the family name.
- `Parallel GUI/TUI` is the public product lane for aligned graphical and terminal interfaces.
- `Arctic` remains the TUI code name.
- `Foxtalk` lineage names remain only where they still explain the subsystem story.

See [Engine Feature Crosswalk](/docs/engine/feature-crosswalk) for the current evidence-backed hierarchy and [Current Work Lanes](/docs/dev/current-lanes) for the non-front-page lane inventory.

## Build configuration truth

The top-level CMake options define the real build surface:

```text
DOTTALK_WITH_TV        ON by default
DOTTALK_WITH_GUI       OFF by default
DOTTALK_WITH_WX        OFF by default
DOTTALK_TV_TESTS       ON by default
DOTTALK_WITH_INDEX     ON by default
DOTTALK_WITH_EDUCATION ON by default
BUILD_PYDOTTALK        OFF by default
DOTTALK_WITH_RELATIONS ON by default
DOTTALK_PROFILE        DEV or PROD
```

The build also wires `metacollect` as a developer executable for low-level metadata and reflection collection. Its CMake comment explicitly says it must not rebuild HELP data, mutate `CMDHELPCHK`, write DBFs, or mutate publication artifacts.

## Command surface truth

The command reference shows active command surfaces including:

- `BUILDLMDB` for materializing LMDB-backed tag databases from current table data.
- `CMDHELPCHK` for reflection and HELP/catalog validation.
- `ERSATZ` and `RBROWSE` for relation-aware browsing.
- `SMARTLIST`, `SIMPLEBROWSE` / `SB`, and `SMARTBROWSE` / `SM` / `SMART` for table, tuple, and record browsing.
- `WORKSPACE` commands for opening, saving, loading, and closing workspaces.
- `DDL` commands for schema-oriented DBF creation and validation work.
- `DRAWIO` helpers for opening diagram tooling, with metadata-generated diagrams still a future direction.

## Website wording rule

Use these words carefully:

- Use "active beta", "runtime-proven", "active", "partial", "experimental", and "canary" when that is the observed state.
- Avoid presenting x64base as complete, commercial, or production-finished.
- Avoid claiming DBF_64 and FPT64 are the only storage truth. The local tree contains DBF-style runtime work, x32 and x64-family work, memo systems, xindex backends, data dictionary code, and metadata catalogs.
- Treat DotTalk++ as the canonical runtime and teaching shell, not merely a small scripting add-on.
- Treat the website as a reviewed derivative of SelfDoc/contracts/manual work,
  not as an independent source of technical truth.
